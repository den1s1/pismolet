## Что реализовано

- Добавлен .NET web-проект `src/Pismolet.Web`.
- Реализован вертикальный auth-flow:
  - регистрация;
  - вход;
  - выход;
  - ограничение `/dashboard` только для авторизованных пользователей;
  - базовые ошибки регистрации и входа.
- Реализовано подтверждение email через dev/fake mailer:
  - письмо сохраняется во внутренний outbox;
  - dev-ссылка доступна сразу после регистрации;
  - есть сценарий повторной отправки подтверждения.
- При регистрации создаётся `ClientProfile`.
- Добавлены дефолтные лимиты нового клиента:
  - дневной лимит — `1000`;
  - общий лимит — `10000`;
  - обязательная премодерация — `true`.
- Поддержаны статусы клиента:
  - `active`;
  - `blocked` предусмотрен в правилах спринта и должен быть вынесен в постоянную БД при переходе на EF/PostgreSQL.
- Добавлен первый экран личного кабинета:
  - список рассылок;
  - статусы рассылок на русском языке;
  - карточка «Создать рассылку»;
  - внутренние технические сущности не выводятся пользователю.
- Добавлен audit log для событий:
  - регистрация;
  - вход;
  - выход;
  - подтверждение email.

## Структура после доработки

`Program.cs` оставлен только как composition root:

- подключение cookie-auth;
- регистрация зависимостей;
- подключение endpoint-групп;
- health-check.

Код спринта разнесён по слоям:

- `Domain` — доменные модели пользователя, профиля, рассылки, fake mail и audit record;
- `Application` — сценарии регистрации, входа, подтверждения email, интерфейсы репозиториев, mailer и audit logger;
- `Infrastructure` — in-memory реализации хранилища, fake mailer и audit logger;
- `Endpoints` — HTTP-маршруты;
- `Rendering` — временный HTML-рендеринг до появления полноценного UI/Blazor-слоя.

## Технические допущения

- Реализация первого среза использует in-memory инфраструктуру, чтобы быстро получить вертикальный пользовательский сценарий.
- Для production/MVP persistence нужно перенести инфраструктурные реализации на EF Core + PostgreSQL согласно `docs/architecture_dotnet.md`.
- Пароль пока хранится через dev-хешер внутри `UserAccountService`; перед реальной авторизацией нужен нормальный password hasher/Identity-подход.
- Unit/integration тесты из спринта пока не добавлены как исполняемые тестовые проекты; это нужно закрыть следующим техническим коммитом после создания solution/test-каркаса.

## Ручная проверка

```bash
dotnet run --project src/Pismolet.Web/Pismolet.Web.csproj
```

Сценарий:

1. Открыть `/account/register`.
2. Зарегистрировать пользователя.
3. Подтвердить email по dev-ссылке или через `/dev/fake-mailer`.
4. Войти через `/account/login`.
5. Проверить `/dashboard`.
6. Выйти через кнопку в ЛК.
7. Открыть `/dashboard` без авторизации и убедиться, что доступ закрыт.
