# Sprint 2 — создание рассылки и импорт адресов с учётом текущего .NET-кода

Статус: план задач  
Ветка: `Development`  
Источник: фактический код ветки `Development`, `README.md`, `docs/sprints.md`, `docs/architecture_dotnet.md`, `source/tasks/sprint_0.md`, `source/tasks/sprint_1.md`, отчёты Sprint 0 и Sprint 1

## Контекст текущего состояния ветки

- dev-only in-memory инфраструктура:
  - `InMemoryUserRepository`;

## Цель спринта

Дать пользователю первый рабочий сценарий после регистрации и входа:

> открыть ЛК → нажать «Создать рассылку» → создать черновик рассылки → загрузить CSV с email → увидеть понятный отчёт проверки адресов.

Главная цель Sprint 2 — не подключить всю production-инфраструктуру, а расширить текущий in-memory вертикальный срез до сценария создания рассылки и импорта адресов.

## Видимый результат

Авторизованный пользователь может:

- открыть `/dashboard`;
- нажать «Создать рассылку»;
- создать новую рассылку с названием/темой;
- загрузить файл адресов;
- увидеть отчёт:
  - всего строк;
  - валидных email;
  - дублей;
  - невалидных email;
  - исключённых адресов из глобальной отписки;
- вернуться к рассылке или в ЛК;
- увидеть созданную рассылку в списке ЛК.

Пользовательский интерфейс должен использовать простые названия: «Рассылка», «Адресаты», «Загрузка адресов», «Результат проверки». Внешний UI не должен показывать технические сущности `ImportBatch`, `Recipient`, `GlobalSuppression`, `Campaign`, `SendEvent`.

## Границы спринта

### Входит в Sprint 2

- Создание черновика рассылки.
- Привязка рассылки к текущему пользователю.
- Подключение кнопки «Создать рассылку» в ЛК к реальному маршруту.
- Загрузка CSV-файла с email.
- Минимальный импорт XLSX только после явного выбора библиотеки и проверки лицензии; если библиотека не выбрана, XLSX фиксируется как перенос на следующий спринт или отдельную задачу.
- Нормализация email.
- Проверка дублей внутри загруженного файла.
- Синтаксическая проверка email.
- Проверка глобальной отписки через in-memory suppression list.
- Сохранение результата импорта в памяти в рамках dev-среза.
- Отчёт импорта в UI.
- Unit-тесты и web/integration smoke tests на новый flow.

### Не входит в Sprint 2

- Реальная отправка писем.
- Редактор текста письма.
- Декларация законности базы.
- Оплата.

- EF Core / PostgreSQL migration как обязательное условие Sprint 2.

- Хранение ФИО адресатов.
- LLM-модерация.

EF Core/PostgreSQL, Hangfire и production-хранилище остаются целевой архитектурой, но в Sprint 2 не должны смешиваться с задачей пользовательского импорта, если для этого нет отдельного архитектурного решения.

## Основные задачи

### 1. Связать ЛК с созданием рассылки

- Заменить `href="#"` у карточки «Создать рассылку» на рабочий маршрут, например `/mailings/new`.
- Добавить endpoint-группу, например `MailingEndpoints`.
- Подключить `app.MapMailingEndpoints()` в `Program.cs`.
- Ограничить все маршруты создания и импорта рассылки авторизацией.
- Если пользователь не авторизован, перенаправлять на `/account/login`.

### 2. Расширить доменную модель рассылки

Текущий `Mailing` содержит только `Subject` и `StatusRu`. Для Sprint 2 нужно добавить минимальные данные, достаточные для черновика и импорта:

- публичный идентификатор рассылки, например `Guid Id`;
- тема или название рассылки;
- статус на русском языке или enum + отображение;
- дата создания;
- статистика последнего импорта;
- список разрешённых адресатов после проверки.

Важно: на уровне UI пользователь видит «Рассылку», а не внутреннюю модель.

### 3. Добавить модели импорта адресов

Добавить минимальные доменные/application-модели:

- `Recipient`:
  - email;
  - нормализованный email;
  - статус проверки;
  - причина исключения, если адрес не принят;
- `ImportBatch` или внутренняя модель результата импорта:
  - id;
  - mailing id;
  - дата импорта;
  - статистика;
  - список ошибок/предупреждений;
- `ImportStats`:
  - всего строк;
  - валидных;
  - дублей;
  - невалидных;
  - исключённых глобальной отпиской.

Если `ImportBatch` создаётся как внутренняя сущность, не показывать это название в UI.

### 4. Добавить in-memory хранилище рассылок и импорта

С учётом текущего кода есть два возможных пути:

1. на коротком Sprint 2 хранить рассылки внутри `UserAccount.Mailings` и расширить `IUserRepository` методами обновления рассылок пользователя;
2. добавить отдельный интерфейс `IMailingRepository` и dev-only `InMemoryMailingRepository`.

Предпочтительный вариант для дальнейшего роста — отдельный `IMailingRepository`, потому что далее появятся импорт, декларация, письмо, оплата, отправка и события.

Задачи:

- добавить `IMailingRepository` в `Application/Persistence`;
- добавить `InMemoryMailingRepository` в `Infrastructure/Persistence`;
- хранить рассылки с привязкой к email/ID пользователя;
- обеспечить, что пользователь видит и изменяет только свои рассылки;
- не ломать текущий список рассылок в `/dashboard`.

### 5. Реализовать создание черновика рассылки

- Добавить GET-страницу формы создания рассылки.
- Поля MVP:
  - название или тема рассылки;
  - краткое описание опционально, если не усложняет UI.
- Добавить POST-обработчик создания рассылки.
- После создания переводить пользователя на страницу загрузки адресов.
- Добавить audit log событие `mailing_created`.
- Проверять:
  - пустое название;
  - слишком длинное название;
  - доступ только владельца.

### 6. Реализовать загрузку CSV

- Добавить страницу загрузки адресов для конкретной рассылки.
- Поддержать `multipart/form-data`.
- Для MVP принимать CSV с колонкой `email`.
- Допустить простой CSV без сложной обработки Excel-формул и макросов.
- Ограничить размер файла на dev-уровне.
- Ограничить количество строк на импорт в рамках текущих лимитов нового клиента.
- Вернуть понятную ошибку, если:
  - файл не выбран;
  - формат не CSV;
  - нет колонки `email`;
  - файл пустой;
  - превышен лимит строк.

### 7. XLSX как управляемая задача, а не скрытое усложнение

`docs/sprints.md` указывает CSV/XLSX, но текущий `.csproj` не содержит библиотек для XLSX.

Задачи:

- выбрать библиотеку для XLSX, если XLSX нужен именно в Sprint 2:
  - ClosedXML;
  - EPPlus только после проверки лицензии;
  - другой вариант с приемлемой лицензией;
- добавить dependency через `Directory.Packages.props`;
- покрыть XLSX импорт тестами;
- если выбор библиотеки не подтверждён, оставить XLSX как явно перенесённую задачу, а Sprint 2 закрыть CSV-импортом.

### 8. Нормализация и валидация email

- Вынести нормализацию email в отдельный application/domain-сервис, чтобы не дублировать логику из `UserAccountService.NormalizeEmail`.
- Минимальная нормализация:
  - trim;
  - lowercase invariant;
  - удаление пустых строк;
- Синтаксическая проверка:
  - базовая проверка формата email;
  - без DNS/MX-проверки в Sprint 2;
  - без попытки подтвердить существование ящика.
- Сохранять исходную строку только если это нужно для отчёта ошибок; не показывать лишние персональные данные.

### 9. Дубли и глобальная отписка

- Определять дубли внутри одного файла после нормализации email.
- Для дублей учитывать первый валидный адрес, остальные отмечать как дубли.
- Добавить application-интерфейс `IGlobalSuppressionRepository`.
- Добавить dev-only `InMemoryGlobalSuppressionRepository`.
- Проверять адреса из suppression list при импорте.
- Исключённые адреса не попадaют в список разрешённых получателей.
- UI должен говорить: «исключены по глобальной отписке», а не раскрывать внутренние детали suppression-хранилища.

### 10. Отчёт импорта

После загрузки показать страницу результата:

- название рассылки;
- сколько строк обработано;
- сколько адресов принято;
- сколько дублей;
- сколько невалидных;
- сколько исключено по глобальной отписке;
- список первых ошибок/предупреждений, если их немного;
- кнопка «Вернуться к рассылке»;
- кнопка «Загрузить другой файл».

В отчёте не показывать технические названия `ImportBatch`, `Recipient`, `GlobalSuppression`.

### 11. Обновить ЛК

- После создания рассылки показывать её в списке `/dashboard`.
- Для рассылки со свежим импортом показывать понятный статус, например:
  - «Черновик»;
  - «Адреса загружены»;
  - «Требуется текст письма».
- Добавить рядом с рассылкой ссылку на следующий шаг, если это не усложняет текущий HTML-рендеринг.
- Не показывать пользователю внутренние ID, если они не нужны.

### 12. Аудит действий

Добавить audit log для событий:

- `mailing_created`;
- `recipients_import_started`;
- `recipients_import_completed`;
- `recipients_import_failed`.

Минимальный контекст:

- пользователь;
- mailing id;
- дата/время;
- IP;
- user-agent;
- статистика импорта без раскрытия лишних данных.

### 13. Безопасность и ограничения

- Все POST-операции должны быть доступны только авторизованному пользователю.
- Пользователь не может открыть или изменить чужую рассылку.
- Проверить, нужен ли antiforgery для текущего Minimal API / HTML form flow. Если не внедряется в Sprint 2, явно зафиксировать риск в отчёте.
- Не хранить ФИО адресатов.
- Для MVP принимать только email как обязательное поле.
- Не подключать реальный email provider.
- Не отправлять письма по импортированным адресам в этом спринте.

- Обновить `README.md`, если появляются новые dev-маршруты.
- Обновить `AGENTS.md`, если добавляются правила по импорту.
- При необходимости добавить короткий раздел в `docs/architecture_dotnet.md` о том, что импорт в текущем срезе in-memory, а production-замена будет через EF Core/PostgreSQL и фоновые задачи.
- В отчёте Sprint 2 явно указать, выполнен ли XLSX или перенесён.

## Definition of Done

Sprint 2 считается завершённым, если:

- карточка «Создать рассылку» в ЛК ведёт на рабочий flow;
- авторизованный пользователь может создать рассылку;
- пользователь может загрузить CSV с колонкой `email`;
- система показывает понятный отчёт проверки адресов;
- валидные адреса, дубли, невалидные адреса и глобальная отписка считаются корректно;
- созданная рассылка отображается в ЛК;
- пользователь не может работать с чужими рассылками;
- добавлены unit-тесты для сервисов создания рассылки и импорта;
- добавлены web/integration smoke tests для основных маршрутов;
- `dotnet restore`, `dotnet format --verify-no-changes`, `dotnet build`, `dotnet test` проходят локально или в CI;
- в отчёте спринта указано, реализован ли XLSX или перенесён;
- UI остаётся русскоязычным и не раскрывает внутренние технические сущности.
