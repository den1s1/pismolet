# Sprint 0 — стабилизация существующего .NET-каркаса и подготовка базы разработки

Статус: план задач  
Ветка: `Development`  
Источник: фактический код ветки `Development`, `README.md`, `docs/platform_tz.md`, `source/tasks/sprint_1.md`

## Контекст текущего состояния ветки

Sprint 0 составлен не для пустого репозитория, а с учётом уже существующего кода.

В ветке `Development` уже есть первый вертикальный .NET-срез приложения:

- web-проект `src/Pismolet.Web/Pismolet.Web.csproj` на `net9.0`;
- запуск через `dotnet run --project src/Pismolet.Web/Pismolet.Web.csproj`;
- ASP.NET Core Minimal API в `Program.cs`;
- cookie-аутентификация;
- endpoints:
  - `/`;
  - `/account/register`;
  - `/account/login`;
  - `/account/logout`;
  - `/account/confirm-email`;
  - `/account/resend-confirmation`;
  - `/dashboard`;
  - `/dev/fake-mailer`;
  - `/health`;
- структура кода:
  - `Domain`;
  - `Application`;
  - `Infrastructure`;
  - `Endpoints`;
  - `Rendering`;
- in-memory реализации:
  - `InMemoryUserRepository`;
  - `InMemoryFakeMailer`;
  - `InMemoryAuditLogger`;
- базовые доменные сущности:
  - `UserAccount`;
  - `ClientProfile`;
  - `Mailing`;
  - `AuditRecord`;
- частично реализованы сценарии, которые были запланированы для Sprint 1:
  - регистрация;
  - подтверждение email через dev/fake mailer;
  - вход;
  - выход;
  - личный кабинет;
  - audit log ключевых действий.

При этом есть важное расхождение: `docs/platform_tz.md` пока описывает MVP-стек как Python/Django/PostgreSQL/Redis/Celery, а фактический код уже выполнен на .NET/ASP.NET Core. В Sprint 0 нельзя игнорировать это расхождение.

## Цель спринта

Стабилизировать уже созданный .NET-каркас, привести его к понятному состоянию для дальнейшей разработки LLM-агентом и подготовить решение к осознанному продолжению Sprint 1.

Главная цель — не переписывать код “с нуля”, а:

1. зафиксировать фактический .NET-стек как текущее состояние ветки;
2. оформить структуру solution;
3. добавить тестовый и dev-контур;
4. отделить уже реализованное от того, что ещё нужно делать в Sprint 1;
5. явно вынести архитектурное расхождение с `docs/platform_tz.md` в отдельное решение/синхронизацию.

## Видимый результат

После Sprint 0 разработчик или LLM-агент может:

- открыть репозиторий и понять, что текущая реализация — .NET/ASP.NET Core;
- собрать проект одной командой;
- запустить приложение локально;
- пройти ручной smoke-flow регистрации и входа;
- запустить тесты;
- увидеть, какие части Sprint 1 уже реализованы, а какие требуют доработки;
- не начать случайный перепис проекта на Django только потому, что старое ТЗ ещё не синхронизировано с кодом.

## Основные задачи

### 1. Зафиксировать фактическое состояние кода

- Описать в Sprint 0, что проект уже содержит .NET web-приложение в `src/Pismolet.Web`.
- Проверить, что README соответствует фактическому запуску.
- Проверить, что перечисленные endpoints действительно доступны после запуска.
- Зафиксировать, что Sprint 0 не создаёт новый Django-проект поверх существующего кода.
- Зафиксировать, что изменение стека требует отдельного архитектурного решения и синхронизации `docs/platform_tz.md`.

### 2. Оформить .NET solution

- Добавить solution-файл, например `Pismolet.sln`, если его ещё нет.
- Подключить в solution существующий проект `src/Pismolet.Web/Pismolet.Web.csproj`.
- Подготовить место для будущих тестовых проектов:
  - `tests/Pismolet.Web.Tests`;
  - при необходимости позже — `tests/Pismolet.IntegrationTests`.
- Проверить команды:
  - `dotnet restore`;
  - `dotnet build`;
  - `dotnet run --project src/Pismolet.Web/Pismolet.Web.csproj`.

### 3. Стабилизировать текущий запуск

- Проверить локальный запуск приложения без дополнительных скрытых действий.
- Убедиться, что `/health` возвращает успешный ответ.
- Убедиться, что главная страница открывается.
- Убедиться, что статические файлы подключаются корректно.
- Проверить, что dev/fake mailer доступен по `/dev/fake-mailer`.
- Обновить README, если какие-то команды запуска отличаются от фактических.

### 4. Разобрать границу Sprint 0 / Sprint 1

- Отметить, что часть задач Sprint 1 уже реализована в коде:
  - регистрация;
  - вход;
  - выход;
  - подтверждение email;
  - fake mailer;
  - личный кабинет;
  - создание `ClientProfile`;
  - базовые mailing-черновики;
  - audit log.
- В Sprint 0 не удалять эту реализацию.
- В Sprint 1 после Sprint 0 нужно будет не начинать заново, а провести ревизию существующего вертикального среза:
  - качество валидации;
  - безопасность пароля;
  - persistence;
  - тесты;
  - UX;
  - соответствие ТЗ.

### 5. Добавить тестовый проект

- Создать `tests/Pismolet.Web.Tests`.
- Выбрать тестовый стек для .NET:
  - xUnit или NUnit;
  - FluentAssertions при необходимости;
  - `Microsoft.AspNetCore.Mvc.Testing` для web/integration smoke tests.
- Добавить тестовый проект в solution.
- Добавить первые smoke/unit tests для уже существующего кода, а не для абстрактного будущего каркаса.

Минимальный набор тестов:

- приложение собирается;
- `/health` возвращает OK;
- главная страница открывается;
- регистрация создаёт пользователя в in-memory repository;
- регистрация создаёт `ClientProfile` с дефолтными лимитами;
- регистрация отправляет письмо в `InMemoryFakeMailer`;
- подтверждение email по dev-token меняет статус пользователя;
- вход невозможен до подтверждения email;
- вход возможен после подтверждения email;
- audit log создаётся для регистрации, подтверждения email, входа и выхода.

### 6. Разделить dev-only реализации и будущие production-замены

- Явно пометить in-memory реализации как временные:
  - `InMemoryUserRepository`;
  - `InMemoryFakeMailer`;
  - `InMemoryAuditLogger`.
- Проверить, что они находятся в `Infrastructure`, а доменная и application-логика не завязаны на конкретную in-memory реализацию.
- Убедиться, что интерфейсы уже позволяют позже заменить:
  - repository на БД;
  - fake mailer на email provider adapter;
  - audit logger на persistent audit storage.
- Не подключать реальный email-провайдер в Sprint 0.
- Не добавлять реальные секреты в репозиторий.

### 7. Безопасность текущего dev-среза

- Зафиксировать, что текущий `HashPassword` с префиксом `dev:` — только временная dev-заглушка.
- Добавить задачу на замену dev-хеша на штатный password hasher ASP.NET Core Identity или отдельный безопасный механизм до production/MVP-релиза.
- Проверить cookie settings для dev-режима.
- Убедиться, что `/dashboard` требует авторизации.
- Проверить, что logout доступен только авторизованному пользователю.
- Проверить, что dev endpoints явно отделены от production-режима или помечены как dev-only.

### 8. Конфигурация и окружения

- Добавить `appsettings.Development.json`, если требуется для явных dev-настроек.
- Добавить `.env.example` только если проект действительно начинает читать переменные окружения.
- Не имитировать Docker/PostgreSQL/Redis, если они ещё не подключаются фактическим кодом.
- Зафиксировать будущие инфраструктурные решения отдельными задачами, а не смешивать их с текущим in-memory-срезом.
- Настроить русский язык пользовательского интерфейса как обязательное требование для текущих страниц.

### 9. Линтинг и форматирование

- Добавить или проверить `.editorconfig` для C#.
- Включить nullable и implicit usings уже включены в `.csproj`; проверить, что предупреждения не игнорируются без причины.
- Зафиксировать команды:
  - `dotnet format`;
  - `dotnet build`;
  - `dotnet test`.
- При необходимости добавить базовые analyzer-пакеты позже, но не блокировать Sprint 0 избыточной настройкой.

### 10. CI

- Добавить базовый GitHub Actions workflow, если его ещё нет.
- Проверять минимум:
  - `dotnet restore`;
  - `dotnet build`;
  - `dotnet test`.
- CI должен запускаться на push/PR в `Development` и main-ветку, если это соответствует процессу репозитория.
- Если CI не добавляется в Sprint 0, явно зафиксировать причину в отчёте спринта.

### 11. Документация для LLM-агента

- Добавить или обновить краткую инструкцию для LLM-агентов в репозитории.
- Указать:
  - фактический код сейчас находится в `src/Pismolet.Web`;
  - задачи спринтов находятся в `source/tasks`;
  - перед изменениями нужно смотреть существующий код, а не только ТЗ;
  - нельзя переписывать стек без отдельного решения;
  - dev/fake mailer нельзя заменять реальным провайдером без отдельной задачи;
  - LLM-модерация не входит в MVP;
  - пользовательский UI не должен раскрывать внутренние технические сущности.

### 12. Синхронизация с `docs/platform_tz.md`

- Создать отдельную задачу/ADR: выбрать один из вариантов:
  - подтвердить переход MVP на .NET/ASP.NET Core и обновить `docs/platform_tz.md`;
  - признать текущий .NET-код прототипом и запланировать переписывание под Django;
  - временно продолжать .NET-срез как эксперимент, но не считать его финальной архитектурой.
- До принятия решения не добавлять задачи, которые предполагают одновременное развитие двух разных стеков.
- В отчёте Sprint 0 явно указать выбранный вариант или статус открытого архитектурного вопроса.

## Unit-тесты

- `ClientProfile.NewClientDefault()` возвращает:
  - `Status = active`;
  - `DailySendLimit = 1000`;
  - `PremoderationRequired = true`.
- `InMemoryUserRepository` добавляет пользователя и ищет его по email.
- `InMemoryUserRepository` не допускает дубли по email.
- `InMemoryUserRepository` ищет пользователя по confirmation token.
- `InMemoryFakeMailer` сохраняет письмо подтверждения в outbox.
- `InMemoryAuditLogger` сохраняет audit record.
- `UserAccountService.Register` создаёт пользователя, профиль, confirmation token и письмо подтверждения.
- `UserAccountService.Authenticate` не пускает пользователя до подтверждения email.
- `UserAccountService.ConfirmEmail` подтверждает email и пишет audit record.

## Интеграционные / smoke-тесты

- `GET /health` возвращает успешный ответ.
- `GET /` возвращает HTML-страницу.
- `GET /account/register` возвращает HTML-форму регистрации.
- `POST /account/register` создаёт пользователя и показывает dev-ссылку подтверждения.
- `GET /dev/fake-mailer` показывает подготовленное письмо.
- `GET /account/confirm-email?token=...` подтверждает email.
- `POST /account/login` после подтверждения email создаёт auth-cookie и перенаправляет в `/dashboard`.
- `GET /dashboard` без авторизации закрыт или перенаправляет на login.
- `GET /dashboard` после входа показывает личный кабинет.
- `POST /account/logout` завершает сессию.

## Ручные тесты

- Перейти в ветку `Development`.
- Выполнить `dotnet restore`.
- Выполнить `dotnet build`.
- Запустить приложение командой `dotnet run --project src/Pismolet.Web/Pismolet.Web.csproj`.
- Открыть `/health`.
- Открыть `/`.
- Зарегистрировать пользователя.
- Перейти в `/dev/fake-mailer` и найти письмо подтверждения.
- Подтвердить email по dev-ссылке.
- Войти под пользователем.
- Открыть `/dashboard`.
- Выйти из аккаунта.
- Проверить, что без авторизации `/dashboard` недоступен.
- Выполнить `dotnet test`, когда тестовый проект будет добавлен.
- Выполнить `dotnet format --verify-no-changes`, когда форматирование будет настроено.

## Не входит в Sprint 0

- Переписывание проекта на Django.
- Подключение PostgreSQL/Redis/Celery без отдельного решения по стеку.
- Реальная отправка писем через ESP.
- Production password storage, если задача явно переносится в Sprint 1/2, но dev-риск должен быть зафиксирован.
- Импорт CSV/XLSX.
- Создание полноценной рассылки.
- Расчёт стоимости.
- Оплата.
- Глобальная отписка.
- Risk-проверка.
- Ручная модерация.
- Обработка ответов и webhooks.

## Definition of Done

- `sprint_0.md` отражает фактический .NET-код ветки `Development`.
- Есть понятное решение, как собирать и запускать текущий проект.
- Есть solution-файл или явно зафиксирована причина его отсутствия.
- Есть тестовый проект или явно зафиксирована причина переноса.
- Минимальные smoke/unit tests для существующего кода добавлены или перенесены с причиной.
- README не противоречит фактическому запуску.
- Dev-only реализации явно отделены от будущих production-замен.
- Зафиксирован архитектурный конфликт между `docs/platform_tz.md` и текущим .NET-кодом.
- Есть решение или открытая задача по синхронизации `docs/platform_tz.md`.
- Sprint 1 можно продолжать как доработку существующего вертикального среза, а не как повторную реализацию с нуля.
