# Sprint 0 — подготовка проекта и технический фундамент

Статус: план задач  
Ветка: `Development`  
Источник: `docs/platform_tz.md`, текущее состояние ветки `Development`

## Контекст текущего состояния ветки

В ветке `Development` уже есть файл `source/tasks/sprint_1.md` с задачами первого продуктового спринта: регистрация, вход, подтверждение email, `ClientProfile`, первый экран личного кабинета и аудит базовых действий.

Sprint 0 нужен как подготовительный спринт перед Sprint 1. Его цель — создать минимальную техническую основу, чтобы задачи Sprint 1 можно было выполнять не в пустой папке, а в понятной структуре проекта с локальным запуском, тестами, базовыми настройками и правилами для LLM-агентов.

## Цель спринта

Подготовить рабочий каркас приложения SimpleMail / «Письмолёт» на принятом MVP-стеке:

- Python 3.12+;
- Django;
- Django Templates;
- HTMX;
- PostgreSQL;
- Redis;
- Celery;
- Docker Compose;
- pytest / pytest-django / factory_boy;
- ruff;
- fake email provider для разработки и тестов.

## Видимый результат

Разработчик или LLM-агент может открыть репозиторий, запустить проект локально через Docker Compose, выполнить миграции, открыть пустое Django-приложение, запустить тесты и понять, куда добавлять код Sprint 1.

## Основные задачи

### 1. Структура проекта

- Создать базовую структуру Django-проекта в папке `source`.
- Зафиксировать единый entrypoint приложения.
- Подготовить структуру backend-приложений согласно ТЗ:
  - `accounts`;
  - `clients`;
  - `mailings`;
  - `imports`;
  - `recipients`;
  - `campaigns`;
  - `sending`;
  - `inbound`;
  - `suppression`;
  - `moderation`;
  - `billing`;
  - `audit`;
  - `admin_tools`.
- Для приложений, которые не реализуются в Sprint 0, оставить минимальные заготовки или явно отложить создание до соответствующего спринта.
- Не добавлять сложную бизнес-логику в Sprint 0.

### 2. Локальная инфраструктура

- Подготовить `docker-compose.yml` для локального запуска.
- Добавить сервисы:
  - Django web app;
  - PostgreSQL;
  - Redis;
  - Celery worker, если он уже может запускаться без бизнес-задач.
- Подготовить `.env.example` с обязательными переменными окружения.
- Настроить подключение Django к PostgreSQL через переменные окружения.
- Проверить, что проект запускается локально одной командой.

### 3. Django settings и окружения

- Разделить настройки на базовые и dev-настройки, если это не усложняет проект.
- Вынести секреты и параметры окружения из кода.
- Включить русский язык интерфейса по умолчанию.
- Настроить часовой пояс проекта.
- Подготовить базовые настройки static files и templates.
- Подготовить настройки логирования для dev-режима.

### 4. База данных и миграции

- Подключить PostgreSQL как основную БД.
- Проверить создание и применение начальных миграций Django.
- Не проектировать все бизнес-сущности заранее сверх минимального каркаса.
- Подготовить место для будущих моделей Sprint 1:
  - пользователь;
  - профиль клиента;
  - базовый audit log.

### 5. Тестовый каркас

- Настроить pytest.
- Настроить pytest-django.
- Подготовить `factory_boy` для будущих фабрик.
- Добавить минимальный smoke-test, который проверяет, что Django-проект загружается.
- Добавить тест проверки доступности базовой страницы или healthcheck endpoint.
- Настроить отдельную тестовую БД или тестовые настройки.

### 6. Линтинг и форматирование

- Подключить `ruff`.
- Добавить базовую конфигурацию `ruff`.
- Зафиксировать команды для:
  - запуска тестов;
  - запуска линтера;
  - запуска форматирования, если используется.
- Убедиться, что пустой каркас проекта проходит линтинг.

### 7. Celery и фоновые задачи

- Добавить базовую конфигурацию Celery.
- Подключить Redis как broker.
- Подготовить структуру очередей из ТЗ:
  - `import_queue`;
  - `risk_queue`;
  - `send_queue`;
  - `webhook_queue`;
  - `reply_queue`;
  - `billing_queue`;
  - `cleanup_queue`.
- В Sprint 0 не реализовывать бизнес-задачи очередей.
- Добавить простую dev/smoke задачу только при необходимости для проверки запуска worker.

### 8. Email и fake provider

- Подготовить интерфейс или место для будущего `EmailProviderAdapter`.
- Добавить fake email provider для разработки и тестов.
- Не подключать реальный ESP в Sprint 0.
- Не хранить реальные ключи провайдеров в репозитории.

### 9. Healthcheck и базовая стартовая страница

- Добавить простой healthcheck endpoint.
- Добавить минимальную стартовую страницу для dev-проверки.
- Не начинать полноценный личный кабинет — это задача Sprint 1.
- Интерфейс и тексты делать на русском языке.

### 10. Django Admin

- Включить Django Admin.
- Подготовить возможность создания superuser в локальном окружении.
- Не делать кастомные admin views в Sprint 0.
- Зафиксировать, что расширение админки будет выполняться в будущих спринтах.

### 11. Правила для LLM-агентов

- Добавить или актуализировать краткий файл с инструкциями для LLM-агента в репозитории, если его ещё нет.
- Указать:
  - основной технический документ: `docs/platform_tz.md`;
  - текущую папку задач: `source/tasks`;
  - запрет на самостоятельное изменение архитектурных решений без ADR;
  - обязательность тестов для изменений;
  - необходимость использовать fake/mock provider для внешних интеграций;
  - запрет на LLM-модерацию в MVP.
- Указать, что клиентский UI не должен раскрывать внутренние сущности вроде `Campaign`, `ImportBatch`, `SendEvent`, `PaymentAttempt`.

### 12. Документация запуска

- Добавить или обновить README для разработки.
- Описать:
  - как скопировать `.env.example`;
  - как запустить Docker Compose;
  - как выполнить миграции;
  - как создать superuser;
  - как запустить тесты;
  - как запустить линтер;
  - где смотреть задачи спринтов.

### 13. CI, если применимо на текущем этапе

- Добавить базовый GitHub Actions workflow, если CI ещё не настроен.
- Проверять минимум:
  - установку зависимостей;
  - линтинг;
  - unit/smoke tests.
- Если CI пока не настраивается, явно зафиксировать это в отчёте Sprint 0 и не считать блокером для локальной разработки.

## Unit-тесты

- Smoke-test загрузки Django settings.
- Проверка healthcheck endpoint.
- Проверка подключения URL routing.
- Проверка fake email provider без реальной отправки.
- Проверка, что Celery app импортируется без ошибки.

## Интеграционные тесты

- Локальный запуск web app через Docker Compose.
- Подключение Django к PostgreSQL.
- Применение миграций на чистой БД.
- Подключение к Redis.
- Запуск Celery worker в dev-режиме, если worker добавлен в compose.
- Запуск тестового набора внутри контейнера или локального venv.

## Ручные тесты

- Склонировать репозиторий и перейти в ветку `Development`.
- Скопировать `.env.example` в `.env`.
- Запустить проект по инструкции из README.
- Открыть стартовую страницу.
- Проверить healthcheck endpoint.
- Применить миграции.
- Создать superuser.
- Открыть Django Admin.
- Запустить тесты.
- Запустить линтер.
- Остановить и повторно поднять окружение без ручной чистки данных.

## Не входит в Sprint 0

- Регистрация и вход пользователя.
- Подтверждение email.
- Создание `ClientProfile`.
- Первый экран личного кабинета.
- Импорт CSV/XLSX.
- Создание рассылки.
- Расчёт стоимости.
- Оплата.
- Реальная отправка писем через ESP.
- Глобальная отписка.
- Risk-проверка.
- Ручная модерация.
- Обработка ответов и webhooks.

Эти задачи начинаются со Sprint 1 и следующих спринтов.

## Definition of Done

- Ветка `Development` содержит понятный каркас проекта.
- Проект локально запускается через Docker Compose.
- Django подключается к PostgreSQL.
- Redis доступен для Celery.
- Есть базовая конфигурация Celery.
- Есть fake email provider или явное место под него.
- Есть healthcheck endpoint.
- Django Admin доступен локально.
- Есть `.env.example`.
- Есть README или dev-инструкция запуска.
- pytest настроен и минимальные тесты проходят.
- ruff настроен и каркас проекта проходит проверку.
- В `source/tasks` есть `sprint_0.md` и `sprint_1.md`.
- Sprint 1 можно начинать без дополнительных архитектурных уточнений по базовому запуску проекта.
