# Sprint 5 — формальная risk-проверка и ручная модерация

Ветка: `Development`  
Основание: `docs/sprints.md`, раздел «Спринт 5 — формальная risk-проверка и ручная модерация»  
Дата актуализации: 2026-06-18  
Статус: актуальный task-list для закрытия Sprint 5

## 1. Проверенный текущий срез кода

По состоянию ветки `Development` Sprint 5 уже частично реализован поверх flow Sprint 4:

- solution содержит проекты `Pismolet.Domain`, `Pismolet.Application`, `Pismolet.Infrastructure`, `Pismolet.Web`, `Pismolet.Web.Tests`, `Pismolet.IntegrationTests`;
- приложение запускается через ASP.NET Core minimal endpoints;
- в `Program.cs` подключены:
  - `MapPaymentEndpoints()`;
  - `MapCheckEndpoints()`;
  - `MapAdminEndpoints()`;
- после оплаты на странице `/mailings/{id}/payment` доступен переход к проверке перед отправкой;
- есть пользовательские endpoints проверки:
  - `GET /mailings/{id:guid}/checks`;
  - `POST /mailings/{id:guid}/checks/start`;
- есть базовый admin UI:
  - `GET /admin`;
  - `GET /admin/moderation`;
  - `GET /admin/moderation/{reviewId:guid}`;
  - `POST /admin/moderation/{reviewId:guid}/approve`;
  - `POST /admin/moderation/{reviewId:guid}/reject`;
- admin UI показывает очередь модерации, карточку review, текст письма, служебные блоки, risk reasons и лог действий;
- доступ к admin endpoints пока ограничен только `.RequireAuthorization()`; полноценной RBAC-модели ещё нет;
- блокировка клиента/рассылки через admin UI не реализована и остаётся вне обязательного Sprint 5, если не появится готовая доменная модель блокировки;
- отправка писем не запускается: после `approved` показывается заглушка «Отправка — Sprint 6».

## 2. Цель Sprint 5

Закрыть и стабилизировать сценарий:

```text
создать рассылку → загрузить CSV → подтвердить базу → написать письмо → оплатить fake-оплатой → пройти risk-проверку → получить «Одобрено» или «На ручной проверке» → администратор одобряет или отклоняет
```

Sprint 5 считается закрытым, когда формальная проверка и ручная модерация работают не только в UI, но и как устойчивый application/domain flow с тестами, идемпотентностью, логированием и понятными статусами.

## 3. Рамки Sprint 5

- Risk-проверка только формальная и детерминированная, без LLM.
- Никакие тексты писем и базы получателей не отправляются во внешние сервисы.
- Проверка стартует только после успешной fake-оплаты.
- UI остаётся на текущем подходе minimal endpoints + `HtmlRenderer`.
- Пользовательский язык интерфейса — русский.
- В пользовательском UI нельзя показывать внутренние термины `RiskCheckResult`, `ModerationReview`, `review_required`, `pending_checks`, rule codes и raw score.
- Admin UI может показывать технические детали правил, но без лишних персональных данных получателей.
- Реальная отправка писем, Hangfire и provider integration относятся к Sprint 6.
- Полноценная RBAC-модель админки не входит в обязательный Sprint 5, но все TODO по ролям должны быть явно оставлены в коде и task-list.

## 4. Задачи Sprint 5

### T5-01. Зафиксировать доменную модель risk-проверки и модерации

**Цель:** убедиться, что `RiskCheckResult` и `ModerationReview` являются бизнес-состоянием, а не только HTML/UI-состоянием.

Задачи:

- Проверить текущие модели `RiskCheckResult`, `ModerationReview`, `RiskDecision`, `ModerationReviewStatus`.
- Убедиться, что модели не зависят от `Pismolet.Web.Endpoints` и HTML-рендеринга.
- Зафиксировать минимальные поля `RiskCheckResult`:
  - `Id`;
  - `MailingId`;
  - `Score`;
  - `Decision`;
  - список сработавших правил;
  - `CheckedAt`;
  - безопасное публичное пояснение.
- Зафиксировать минимальные поля `ModerationReview`:
  - `Id`;
  - `MailingId`;
  - `Status`;
  - `Reason`;
  - `CreatedAt`;
  - `ResolvedAt`;
  - `ResolvedBy`;
  - `ModeratorComment`.
- Если часть полей сейчас отсутствует, добавить их или явно зафиксировать технический долг.

Acceptance criteria:

- При повторном открытии страницы проверки виден тот же результат, а не новый пересчёт без необходимости.
- Для рассылки можно объяснить, почему она прошла автоматически или попала на ручную модерацию.
- Модераторское решение не хранится только в HTML/UI.

### T5-02. Проверить и стабилизировать статусы рассылки после оплаты

**Цель:** сделать переходы после `paid` пригодными для Sprint 6.

Задачи:

- Проверить, что после fake-оплаты рассылка имеет состояние, эквивалентное `paid` / «Оплачено».
- Проверить и при необходимости доработать переходы:
  - `paid` → `pending_checks`;
  - `pending_checks` → `approved`;
  - `pending_checks` → `review_required`;
  - `review_required` → `approved`;
  - `review_required` → `rejected`.
- Убедиться, что `CheckEndpoints` и admin flow не принимают бизнес-решения только по русским строкам статуса.
- Добавить или укрепить технический enum/value object статусов, если сейчас часть логики держится на `StatusRu`.
- Запретить переход к будущей отправке из статусов:
  - `paid`;
  - `pending_checks`;
  - `review_required`;
  - `rejected`.

Acceptance criteria:

- Рассылка без оплаты не может попасть в проверку.
- Рассылка в `rejected` не может быть отправлена в Sprint 6.
- Повторное открытие/refresh страницы не меняет состояние случайным образом.

### T5-03. Довести формальный risk engine без LLM

**Цель:** стабильно отделять нормальные письма от писем, требующих ручной проверки.

Задачи:

- Проверить текущий сервис risk-проверки и вынести всю бизнес-логику из endpoints, если она там есть.
- Реализовать/проверить правила из `docs/sprints.md`:
  - подозрительные ссылки;
  - запрещённые тематики;
  - агрессивная реклама;
  - отсутствие понятного отправителя;
  - отсутствие причины получения письма.
- Для каждого правила хранить:
  - стабильный код правила;
  - вес/score;
  - внутреннее пояснение для admin UI;
  - безопасное публичное пояснение для пользователя.
- Начать с детерминированных MVP-эвристик:
  - слишком много ссылок;
  - ссылка без `https`;
  - подозрительный домен из локального списка;
  - стоп-слова/фразы из локального MVP-списка;
  - пустой или слишком короткий отправитель;
  - пустая/непонятная причина получения письма.
- Пороги risk-score держать в коде/config, без отдельного UI в Sprint 5.

Acceptance criteria:

- Нормальное письмо без срабатываний получает «Одобрено».
- Письмо с подозрительной ссылкой попадает на ручную проверку.
- Письмо без понятного отправителя попадает на ручную проверку или получает достаточный risk-score.
- Unit-тесты покрывают каждое MVP-правило.

### T5-04. Реализовать принудительную премодерацию клиента

**Цель:** учесть продуктовые правила о клиентах, которым нужна обязательная проверка.

Задачи:

- Проверить, есть ли в текущем `ClientProfile` флаг обязательной премодерации.
- Добавить/использовать настройки:
  - премодерация для конкретного клиента;
  - премодерация для новых клиентов.
- Risk-check должен учитывать эти настройки до финального решения.
- Если клиент под обязательной премодерацией, даже нормальное письмо должно попасть в `review_required`.
- В admin UI показывать понятную причину:
  - «Новый клиент»;
  - «Для клиента включена премодерация».

Acceptance criteria:

- Клиент с обязательной премодерацией всегда попадает на ручную проверку после оплаты.
- Причина обязательной премодерации видна администратору.
- Клиенты без такого флага продолжают проходить auto-approve при нормальном письме.

### T5-05. Стабилизировать `IMailingReviewService` как orchestration layer

**Цель:** отделить бизнес-flow проверки от HTTP endpoints.

Задачи:

- Проверить текущий `IMailingReviewService` / `MailingReviewService`.
- Метод запуска проверки должен:
  - проверить владельца рассылки;
  - проверить успешную оплату;
  - перевести рассылку в `pending_checks`;
  - запустить risk engine;
  - сохранить `RiskCheckResult`;
  - перевести рассылку в `approved` или `review_required`;
  - при необходимости создать `ModerationReview`;
  - залогировать ключевые действия.
- Повторный запуск проверки должен возвращать существующий результат, если проверка уже выполнена.
- Ошибки обычных бизнес-сценариев возвращать как application result, а не как необработанные исключения.

Acceptance criteria:

- `CheckEndpoints` не содержит risk-score бизнес-логики.
- Проверку можно покрыть unit/integration тестами без HTML.
- Повторный запуск не создаёт дублирующий `ModerationReview`.

### T5-06. Доработать пользовательский UI проверки

**Цель:** сделать послеоплатный шаг понятным для пользователя.

Задачи:

- Проверить страницу `/mailings/{id}/payment`: после `Paid` пользователь должен видеть не заглушку, а понятную кнопку «Проверить перед отправкой».
- Проверить страницу `/mailings/{id}/checks`:
  - до запуска проверки;
  - после auto-approve;
  - после попадания на ручную проверку;
  - после отклонения.
- Для `review_required` показывать нейтральный текст: «Рассылка отправлена на ручную проверку. Мы покажем результат здесь.»
- Для `rejected` показывать безопасную причину без rule codes и raw score.
- Для `approved` явно показывать, что отправка будет доступна в Sprint 6.
- Не показывать пользователю внутренние score, rule codes и технические названия статусов.

Acceptance criteria:

- После оплаты пользователь может запустить проверку из текущего flow.
- Нормальное письмо приводит к экрану «Одобрено».
- Подозрительное письмо приводит к экрану «На ручной проверке».
- Refresh страницы не создаёт новую проверку.

### T5-07. Завершить admin UI очереди модерации

**Цель:** дать администратору минимальный рабочий инструмент ручной проверки.

Задачи:

- Проверить `/admin/moderation`: список должен показывать только открытые review.
- В списке показывать минимум:
  - дата попадания на review;
  - тема письма;
  - клиент/owner email;
  - причины проверки;
  - ссылка на карточку.
- Проверить `/admin/moderation/{reviewId}`:
  - статус review;
  - тема и текст письма;
  - sender name;
  - причина получения письма;
  - служебный блок отписки;
  - service identifier;
  - risk rules;
  - лог действий;
  - формы approve/reject.
- Проверить idempotency approve/reject для закрытого review.
- Оставить TODO на RBAC рядом с admin endpoints до появления модели ролей.
- Не добавлять блокировку клиента в обязательный scope Sprint 5 без отдельного решения.

Acceptance criteria:

- Администратор видит очередь рассылок на ручной проверке.
- Approve переводит рассылку в «Одобрено».
- Reject переводит рассылку в «Отклонено».
- Повторный approve/reject закрытой проверки не ломает состояние.

### T5-08. Довести логирование модерации

**Цель:** обеспечить минимальную прослеживаемость решений.

Задачи:

- При создании `ModerationReview` логировать причину попадания на ручную проверку.
- При approve/reject логировать:
  - `ReviewId`;
  - `MailingId`;
  - email/идентификатор администратора;
  - действие;
  - комментарий модератора;
  - дату/время;
  - предыдущее и новое состояние.
- Использовать существующий audit/dev log, если он подходит.
- Если существующий audit log не подходит, добавить минимальный `ModerationActionLog`.
- Не писать в лог CSV получателей, полные базы адресов и лишние персональные данные.

Acceptance criteria:

- После approve/reject видно, кто и когда принял решение.
- Лог не содержит CSV получателей и лишние персональные данные.
- Повторное действие по закрытой проверке не создаёт ложный успешный лог.

### T5-09. Покрыть Sprint 5 тестами

**Цель:** зафиксировать бизнес-правила risk-проверки и модерации тестами.

Unit-тесты:

- risk-score по набору правил;
- правило подозрительной ссылки;
- правило отсутствия отправителя;
- правило отсутствия причины получения письма;
- клиент с обязательной премодерацией всегда попадает на review;
- отклонённая рассылка не может быть отправлена;
- повторный запуск проверки не создаёт дубликаты;
- approve/reject закрытого review не ломает состояние.

Интеграционные/web-тесты:

- `paid` → `pending_checks` → `approved`;
- `paid` → `pending_checks` → `review_required`;
- admin approve: `review_required` → `approved`;
- admin reject: `review_required` → `rejected`;
- пользователь видит корректный статус после refresh;
- пользователь не видит чужую проверку;
- неавторизованный пользователь не попадает в `/mailings/{id}/checks`;
- неавторизованный пользователь не попадает в `/admin/moderation`.

Acceptance criteria:

- Тесты запускаются в существующих `Pismolet.Web.Tests` / `Pismolet.IntegrationTests`.
- Тесты бизнес-правил не завязаны на порядок HTML-атрибутов.
- Основные переходы Sprint 5 покрыты без ручной проверки.

### T5-10. Подготовить ручные сценарии приёмки Sprint 5

**Цель:** дать короткий чеклист для разработчика/тестировщика.

Сценарии:

1. Нормальное письмо:
   - создать рассылку;
   - загрузить валидный CSV;
   - подтвердить базу;
   - написать нейтральное письмо с понятным отправителем и причиной получения;
   - пройти fake-оплату;
   - запустить проверку;
   - убедиться, что статус стал «Одобрено».
2. Подозрительная ссылка:
   - добавить в письмо подозрительную ссылку или несколько ссылок;
   - пройти оплату;
   - запустить проверку;
   - убедиться, что статус стал «На ручной проверке».
3. Admin approve:
   - открыть `/admin/moderation`;
   - открыть карточку review;
   - нажать approve;
   - убедиться, что пользователь видит «Одобрено».
4. Admin reject:
   - создать новую рассылку, попавшую на review;
   - отклонить её;
   - убедиться, что пользователь видит «Отклонено» и не может перейти к отправке.
5. Refresh/idempotency:
   - обновить страницу после проверки;
   - повторно нажать запуск проверки или approve/reject;
   - убедиться, что дубли review/result не создаются.
6. Access control MVP:
   - неавторизованным пользователем открыть `/mailings/{id}/checks`;
   - неавторизованным пользователем открыть `/admin/moderation`;
   - убедиться, что доступ закрыт.

## 5. Что не входит в Sprint 5

- Реальная отправка писем.
- Очереди Hangfire и batch sending.
- LLM-модерация текста.
- Интеграция с внешними антиспам/blacklist API.
- Полноценная RBAC-модель админки.
- Реальная блокировка аккаунта клиента, если для неё нет готовой доменной модели.
- Юридически значимая экспертиза текста письма.
- Автоматическая проверка экстремизма как гарантированный legal/compliance verdict.

## 6. Риски и технический долг перед Sprint 6

- Если статусная логика всё ещё частично опирается на русские строки, это нужно исправить до очереди отправки.
- Если moderation state хранится только in-memory, нужно явно решить, переносится ли он в EF/PostgreSQL до Sprint 6.
- Если admin endpoints остаются только под `.RequireAuthorization()`, перед production/MVP-beta нужен RBAC или другой механизм ограничения доступа.
- Если risk rules слишком легко обходятся, их нужно рассматривать как MVP-фильтр, а не как юридически значимую проверку.
