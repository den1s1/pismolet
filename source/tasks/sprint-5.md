# Sprint 5 — формальная risk-проверка и ручная модерация

Ветка: `Development`  
Основание: `docs/sprints.md`, раздел «Спринт 5 — формальная risk-проверка и ручная модерация»  
Дата составления: 2026-06-17  
Статус: backlog для реализации

## 1. Текущий срез кода

По состоянию ветки `Development` перед Sprint 5:

- solution уже содержит проекты `Pismolet.Domain`, `Pismolet.Application`, `Pismolet.Infrastructure`, `Pismolet.Web`, `Pismolet.Web.Tests`, `Pismolet.IntegrationTests`;
- приложение запускается через ASP.NET Core minimal endpoints;
- в `Program.cs` уже подключены `MapPaymentEndpoints()` и `MapAdminEndpoints()`;
- Sprint 4 добавил экран `/mailings/{id}/payment`, fake-start и fake-success endpoints;
- после успешной fake-оплаты UI показывает статус «Оплачено» и заглушку «Проверка перед отправкой — Sprint 5»;
- `/admin` пока является общей заглушкой внутренних инструментов MVP;
- по доступной проверке кода пока нет полноценных `ModerationReview`, `RiskCheckResult`, очереди ручной модерации и admin approve/reject flow.

Sprint 5 должен продолжать текущий flow, а не начинать новый сценарий с нуля.

## 2. Цель Sprint 5

После оплаты рассылка проходит формальную risk-проверку без LLM и либо автоматически получает статус «Одобрено», либо попадает на ручную модерацию.

Sprint 5 должен завершиться работающим сценарием:

```text
создать рассылку → загрузить CSV → подтвердить базу → написать письмо → оплатить fake-оплатой → пройти risk-проверку → получить «Одобрено» или «На проверке» → администратор одобряет или отклоняет
```

## 3. Принятые рамки для Sprint 5

- Risk-проверка в Sprint 5 только формальная и детерминированная, без LLM.
- Правила должны быть вынесены из UI/endpoints в application/domain слой.
- Модерация должна работать после оплаты, начиная со статуса `paid`.
- UI остаётся на текущем подходе minimal endpoints + `HtmlRenderer`.
- Админка в Sprint 5 остаётся простой, без полноценной RBAC-модели, но код должен быть готов к добавлению ролей позже.
- Действия модератора логируются в dev/audit-механизм или отдельный moderation log, если он будет добавлен в рамках задачи.
- Sprint 5 не запускает реальную отправку писем. Отправка относится к Sprint 6.
- Sprint 5 не подключает LLM-анализ, внешние антиспам API или юридически значимую экспертизу текста.
- Пользовательский язык интерфейса — русский.
- В UI не показывать внутренние термины `RiskCheckResult`, `ModerationReview`, `review_required`, `pending_checks`.

## 4. Задачи реализации

### T5-01. Добавить доменные модели risk-проверки и модерации

**Цель:** зафиксировать результат автоматической проверки и состояние ручной модерации как часть бизнес-состояния рассылки.

Задачи:

- Добавить модель `RiskCheckResult` или аналог в домене/application-модели рассылок.
- Добавить модель `ModerationReview` или аналог для ручной проверки.
- Минимальные поля `RiskCheckResult`:
  - `Id`;
  - `MailingId`;
  - `Score`;
  - `Decision`;
  - `TriggeredRules`;
  - `CheckedAt`;
  - `CheckedBy` = `System`;
  - краткое безопасное пояснение для UI/админки без лишних персональных данных.
- Минимальные поля `ModerationReview`:
  - `Id`;
  - `MailingId`;
  - `Status`;
  - `Reason`;
  - `CreatedAt`;
  - `ResolvedAt`;
  - `ResolvedBy`;
  - `ModeratorComment`.
- Добавить enum/value object для решений проверки:
  - `Approved`;
  - `ReviewRequired`;
  - `Rejected`, если отклонение может быть автоматическим;
  - `Blocked`, если блокировка клиента/рассылки будет реализована в Sprint 5.
- Модели должны нормально мапиться в EF Core и не зависеть от `Pismolet.Web`.

Acceptance criteria:

- Результат risk-проверки сохраняется и доступен при повторном открытии страницы рассылки.
- Для одной рассылки можно понять, почему она прошла автоматически или попала на review.
- Модераторское решение не хранится только в HTML/UI.

### T5-02. Расширить статусную модель рассылки под проверки и модерацию

**Цель:** добавить понятные технические переходы после `paid`.

Задачи:

- Добавить или расширить технические статусы рассылки:
  - `paid` / «Оплачено»;
  - `pending_checks` / «Проверяем перед отправкой»;
  - `approved` / «Одобрено»;
  - `review_required` / «На ручной проверке»;
  - `rejected` / «Отклонено».
- Описать допустимые переходы:
  - `paid` → `pending_checks`;
  - `pending_checks` → `approved`;
  - `pending_checks` → `review_required`;
  - `review_required` → `approved`;
  - `review_required` → `rejected`.
- Запретить переход к Sprint 6/отправке из статусов `paid`, `pending_checks`, `review_required`, `rejected`.
- Сделать повторный запуск проверки идемпотентным: повторный запрос не должен создавать конфликтующие результаты.
- Добавить русские UI-подписи статусов.

Acceptance criteria:

- Рассылка без оплаты не может попасть в `pending_checks`.
- Рассылка в `rejected` не может быть отправлена.
- Повторное обновление страницы не меняет состояние случайным образом.

### T5-03. Реализовать формальный risk engine без LLM

**Цель:** автоматически отделять очевидно нормальные письма от писем, требующих ручной проверки.

Задачи:

- Добавить сервис risk-проверки, например `IRiskCheckService` / `RiskCheckService`.
- Реализовать набор формальных правил из `docs/sprints.md`:
  - подозрительные ссылки;
  - запрещённые тематики;
  - агрессивная реклама;
  - отсутствие понятного отправителя;
  - отсутствие причины получения письма.
- Для каждого правила возвращать:
  - код правила;
  - вес/баллы;
  - краткое пояснение;
  - решение: informational / review_required / reject, если нужно.
- Начать с простых детерминированных эвристик:
  - слишком много ссылок;
  - ссылка без `https`;
  - короткие или подозрительные домены из настраиваемого списка;
  - стоп-слова/фразы из локального MVP-списка;
  - пустой или слишком короткий отправитель;
  - пустая причина получения письма.
- Сделать пороги score настраиваемыми через code/config, но без отдельного сложного UI в Sprint 5.
- Не отправлять текст письма во внешние сервисы.

Acceptance criteria:

- Нормальное письмо без срабатываний получает `approved`.
- Письмо с подозрительной ссылкой попадает в `review_required`.
- Письмо без понятного отправителя попадает в `review_required` или получает достаточный risk-score.
- Результаты правил покрыты unit-тестами.

### T5-04. Добавить настройки принудительной премодерации

**Цель:** дать системе возможность отправлять часть клиентов/рассылок на ручную проверку независимо от текста.

Задачи:

- Добавить настройки:
  - премодерация для конкретного клиента;
  - премодерация для новых клиентов.
- Решить, где хранить настройки MVP:
  - в `ClientProfile`;
  - в отдельной модели moderation/risk settings;
  - во временной конфигурации с обязательным TODO на перенос в persistence.
- Проверка должна учитывать настройки до финального решения.
- Если клиент под обязательной премодерацией, нормальное письмо тоже должно получить `review_required`.
- В admin/dev UI показать причину попадания на review: «Новый клиент» или «Для клиента включена премодерация».

Acceptance criteria:

- Клиент с флагом обязательной премодерации всегда попадает в `review_required` после оплаты.
- Причина обязательной премодерации видна администратору.
- Настройка не ломает обычный автоматический approve для клиентов без флага.

### T5-05. Реализовать application service запуска проверки

**Цель:** отделить orchestration проверки от HTTP endpoints.

Задачи:

- Добавить сервис, например `IMailingReviewService` / `MailingReviewService`.
- Метод запуска проверки должен:
  - проверить владельца рассылки;
  - проверить, что рассылка оплачена;
  - перевести рассылку в `pending_checks`;
  - запустить risk engine;
  - сохранить `RiskCheckResult`;
  - перевести рассылку в `approved` или `review_required`;
  - при необходимости создать `ModerationReview`.
- Повторный запуск должен возвращать текущий результат, если проверка уже выполнена.
- Ошибки должны возвращаться как application result, а не как исключения для обычных бизнес-сценариев.

Acceptance criteria:

- Endpoint не содержит бизнес-логики risk-score.
- Проверку можно покрыть unit/integration тестами без HTML.
- Повторный запуск не создаёт дублирующий `ModerationReview`.

### T5-06. Добавить пользовательский UI проверки после оплаты

**Цель:** заменить заглушку Sprint 5 на понятный пользовательский сценарий.

Задачи:

- На странице `/mailings/{id}/payment` после `Paid` заменить текст-заглушку на кнопку/переход «Проверить перед отправкой».
- Добавить endpoint, например:
  - `POST /mailings/{id}/checks/start`;
  - `GET /mailings/{id}/checks` или секцию статуса на странице рассылки.
- Пользователь должен видеть статусы:
  - «Проверяем перед отправкой»;
  - «Одобрено»;
  - «На ручной проверке»;
  - «Отклонено».
- Для `review_required` показать нейтральный текст: «Рассылка отправлена на ручную проверку. Мы покажем результат здесь.»
- Для `rejected` показать причину в безопасной формулировке без внутренних rule codes.
- Не показывать пользователю внутренний score, если это может провоцировать обход правил.

Acceptance criteria:

- После оплаты пользователь может запустить проверку из текущего flow.
- Нормальное письмо приводит к экрану «Одобрено».
- Подозрительное письмо приводит к экрану «На ручной проверке».
- Refresh страницы не создаёт новую проверку.

### T5-07. Реализовать admin UI очереди модерации

**Цель:** дать администратору минимальный рабочий инструмент для ручной проверки.

Задачи:

- Расширить текущую `/admin` заглушку ссылкой на очередь модерации.
- Добавить страницы/endpoints:
  - `GET /admin/moderation` — список рассылок на проверке;
  - `GET /admin/moderation/{reviewId}` — карточка проверки;
  - `POST /admin/moderation/{reviewId}/approve`;
  - `POST /admin/moderation/{reviewId}/reject`;
  - опционально `POST /admin/moderation/{reviewId}/block`, если блокировка входит в Sprint 5.
- В списке показать минимум:
  - дата попадания на review;
  - тема письма;
  - отправитель/owner email;
  - risk reasons;
  - текущий статус.
- В карточке показать:
  - текст письма/preview;
  - причину получения письма;
  - служебный блок отписки;
  - результаты risk rules;
  - кнопки approve/reject.
- Пока нет полноценной RBAC-модели, доступ оставить под `.RequireAuthorization()`, но явно отметить TODO на роли администратора.

Acceptance criteria:

- Администратор видит очередь рассылок в `review_required`.
- Approve переводит рассылку в `approved`.
- Reject переводит рассылку в `rejected`.
- Повторный approve/reject уже закрытой проверки не ломает состояние.

### T5-08. Логировать действия модератора

**Цель:** обеспечить минимальную прослеживаемость решений модерации.

Задачи:

- При создании `ModerationReview` логировать причину попадания на ручную проверку.
- При approve/reject логировать:
  - `ReviewId`;
  - `MailingId`;
  - email/идентификатор администратора;
  - действие;
  - комментарий модератора;
  - дату/время;
  - предыдущее и новое состояние.
- Использовать существующий audit/dev log, если он уже подходит.
- Если существующий audit log не подходит, добавить отдельную минимальную модель `ModerationActionLog`.
- Не писать в лог избыточные персональные данные получателей.

Acceptance criteria:

- После approve/reject можно увидеть, кто и когда принял решение.
- Лог не содержит CSV получателей и лишние персональные данные.
- Повторное действие по закрытой проверке логируется как rejected/conflict или не выполняется с понятным сообщением.

### T5-09. Добавить тесты risk-проверки и модерации

**Цель:** зафиксировать бизнес-правила Sprint 5 тестами.

Unit-тесты:

- risk-score по набору правил;
- правило подозрительной ссылки;
- правило отсутствия отправителя;
- правило отсутствия причины получения письма;
- клиент с обязательной премодерацией всегда попадает на review;
- отклонённая рассылка не может быть отправлена;
- повторный запуск проверки не создаёт дубликаты.

Интеграционные/web-тесты:

- `paid` → `pending_checks` → `approved`;
- `paid` → `pending_checks` → `review_required`;
- admin approve: `review_required` → `approved`;
- admin reject: `review_required` → `rejected`;
- пользователь видит корректный статус после refresh;
- неавторизованный пользователь не видит чужую проверку;
- неавторизованный пользователь не может попасть в admin moderation endpoints.

Acceptance criteria:

- Тесты запускаются в существующих `Pismolet.Web.Tests` / `Pismolet.IntegrationTests`.
- Новые правила не завязаны на порядок HTML-атрибутов, если это не критично.
- Ключевые бизнес-переходы покрыты без ручной проверки.

### T5-10. Подготовить ручные сценарии проверки Sprint 5

**Цель:** дать LLM/разработчику и тестировщику короткий чеклист приёмки.

Сценарии:

1. Нормальное письмо:
   - создать рассылку;
   - загрузить валидный CSV;
   - подтвердить базу;
   - написать нейтральное письмо с понятным отправителем и причиной получения;
   - пройти fake-оплату;
   - запустить проверку;
   - убедиться, что статус стал «Одобрено».
2. Подозрительная ссылка:
   - добавить в письмо подозрительную ссылку/несколько ссылок;
   - пройти оплату;
   - запустить проверку;
   - убедиться, что статус стал «На ручной проверке».
3. Admin approve:
   - открыть `/admin/moderation`;
   - открыть карточку review;
   - нажать approve;
   - убедиться, что пользователь видит «Одобрено».
4. Admin reject:
   - создать новую рассылку, попавшую на review;
   - отклонить её;
   - убедиться, что пользователь видит «Отклонено» и не может перейти к отправке.
5. Refresh/idempotency:
   - обновить страницу после проверки;
   - повторно нажать запуск проверки или approve/reject;
   - убедиться, что дубли review/result не создаются.

## 5. Что не входит в Sprint 5

- Реальная отправка писем.
- Очереди Hangfire и batch sending.
- LLM-модерация текста.
- Интеграция с внешними антиспам/blacklist API.
- Полноценная RBAC-модель админки.
- Реальная блокировка аккаунта клиента, если для неё нет готовой доменной модели.
- Юридически значимая экспертиза текста письма.
- Автоматическая проверка экстремизма как гарантированный legal/compliance verdict.
