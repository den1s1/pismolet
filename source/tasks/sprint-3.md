# Sprint 3 — декларация базы и редактор письма

Ветка: `Development`  
Основание: `docs/sprints.md`, раздел «Спринт 3 — декларация базы и редактор письма»  
Дата составления: 2026-06-16  
Статус: backlog для реализации

## 1. Текущий срез кода

По состоянию ветки `Development` реализован vertical slice до конца Sprint 2:

- регистрация, вход, выход и dev/fake подтверждение email;
- создание `ClientProfile` с базовыми лимитами;
- защищённый `/dashboard`;
- создание черновика рассылки через `/mailings/new` и `POST /mailings`;
- загрузка CSV-файла с колонкой `email` через `/mailings/{id}/recipients`;
- проверка адресов: валидные, дубли, невалидные, глобально отписанные;
- сохранение импортированных адресатов в in-memory `Mailing`;
- dev-only in-memory audit log для регистрации, входа и импорта.

В коде пока отсутствуют:

- доменная модель декларации базы;
- версия текста декларации;
- UI шага декларации после импорта;
- проверка рекламного согласия для рекламного письма;
- редактор письма;
- модель содержимого письма;
- генерация служебных блоков письма;
- plain text версия письма;
- unsubscribe token / ссылка глобальной отписки;
- запрет перехода к оплате без декларации и заполненного письма.

## 2. Цель Sprint 3

Пользователь после импорта адресов подтверждает законность базы, при необходимости подтверждает рекламное согласие, затем заполняет имя отправителя, тему, текст и тип письма.

Sprint 3 должен завершиться работающим сценарием:

```text
создать рассылку → загрузить CSV → увидеть результат импорта → заполнить декларацию базы → написать письмо → увидеть preview служебных блоков
```

## 3. Принятые рамки для Sprint 3

- Хранилище остаётся in-memory, как в текущем dev-срезе.
- UI остаётся на текущем подходе minimal endpoints + `HtmlRenderer`, без миграции на Razor Pages внутри этого спринта.
- Реальная отправка писем, расчёт стоимости и fake-оплата относятся к Sprint 4+ и не входят в Sprint 3.
- XLSX-импорт не блокирует Sprint 3 и остаётся отдельной задачей Sprint 2 debt, так как текущий flow уже позволяет пройти CSV-сценарий.
- Пользовательский язык интерфейса — русский.
- Внешние технические сущности не показывать пользователю: в UI использовать простые понятия «рассылка», «адреса», «подтверждение базы», «письмо».

## 4. Задачи реализации

### T3-01. Добавить доменную модель декларации базы

**Цель:** зафиксировать юридически значимое подтверждение пользователя для конкретной рассылки/импорта.

Задачи:

- Добавить модель `MailingDeclaration` или аналог внутри домена рассылок.
- Поля модели:
  - `MailingId`;
  - `UserEmail`;
  - `BaseSource`;
  - `IsBaseLegalityConfirmed`;
  - `IsAdvertisingConsentConfirmed`;
  - `DeclarationVersion`;
  - `CreatedAt`;
  - `Ip`;
  - `UserAgent`.
- Добавить enum/value object для источника базы:
  - клиенты/покупатели;
  - участники мероприятия;
  - подписчики формы;
  - члены организации/сообщества;
  - другое.
- Добавить человекочитаемые русские названия источников для UI.

Acceptance criteria:

- Декларация относится к конкретной рассылке владельца.
- Без `IsBaseLegalityConfirmed = true` декларация не считается действительной.
- Для рекламного типа письма требуется `IsAdvertisingConsentConfirmed = true`.

### T3-02. Добавить версионирование текста декларации

**Цель:** сохранять, с каким текстом согласился пользователь.

Задачи:

- Добавить константу текущей версии декларации, например `BaseDeclarationText.CurrentVersion = "2026-06-16-v1"`.
- Добавить текст декларации на русском языке для MVP.
- Показывать текст декларации в форме подтверждения.
- Сохранять версию в `MailingDeclaration`.
- Не перезаписывать старую версию при будущих изменениях текста.

Acceptance criteria:

- При сохранении декларации в объекте есть версия текста.
- Unit-тест проверяет сохранение версии.

### T3-03. Расширить модель `Mailing` под шаги Sprint 3

**Цель:** хранить состояние декларации и письма в текущей in-memory модели.

Задачи:

- Добавить в `Mailing` поля/свойства:
  - `Declaration`;
  - `MessageDraft`;
  - технический `ServiceId` или `PublicId` для служебного идентификатора рассылки.
- Добавить методы состояния:
  - `WithDeclaration(...)`;
  - `WithMessageDraft(...)`.
- Обновить `StatusRu` после декларации и после сохранения письма:
  - `База подтверждена`;
  - `Письмо подготовлено`.
- Не ломать существующие тесты импорта.

Acceptance criteria:

- После импорта можно сохранить декларацию.
- После декларации можно сохранить письмо.
- Существующий сценарий Sprint 2 продолжает работать.

### T3-04. Добавить application service для декларации

**Цель:** вынести бизнес-правила декларации из endpoint.

Задачи:

- Добавить интерфейс `IMailingDeclarationService`.
- Добавить команду `ConfirmMailingDeclarationCommand`.
- Проверять:
  - пользователь авторизован;
  - рассылка принадлежит пользователю;
  - у рассылки есть принятые адресаты после импорта;
  - выбран источник базы;
  - подтверждена законность базы;
  - для рекламного письма подтверждено рекламное согласие.
- Сохранять декларацию через текущий `IMailingRepository.Update`.
- Писать audit log событие `mailing_declaration_confirmed`.
- Для ошибок возвращать русские сообщения без технических терминов.

Acceptance criteria:

- Нельзя подтвердить декларацию для чужой рассылки.
- Нельзя подтвердить декларацию до импорта адресов.
- Audit log содержит дату, email пользователя, IP, user-agent, тип события и контекст с id рассылки/версией декларации.

### T3-05. Добавить UI шага декларации

**Цель:** после результата импорта пользователь переходит к подтверждению базы.

Задачи:

- Добавить route `GET /mailings/{id}/declaration`.
- Добавить route `POST /mailings/{id}/declaration`.
- На странице показать:
  - название рассылки;
  - статистику импорта;
  - select/radio «Источник базы»;
  - чекбокс «Подтверждаю законность базы»;
  - чекбокс рекламного согласия, если выбран рекламный тип письма или если тип письма выбирается на этом шаге;
  - текст декларации и версию.
- После успешного подтверждения вести на редактор письма.
- В результат импорта добавить кнопку «Подтвердить базу и написать письмо».
- На странице рассылки добавить ссылку на следующий незавершённый шаг.

Acceptance criteria:

- После CSV-импорта пользователь видит понятный следующий шаг.
- При пропуске обязательной галочки показывается ошибка на русском языке.
- Пользователь не видит внутренних терминов `ImportBatch`, `Campaign`, `MailingDeclaration`.

### T3-06. Добавить модель черновика письма

**Цель:** хранить пользовательский контент письма до оплаты/отправки.

Задачи:

- Добавить модель `MailingMessageDraft`.
- Поля:
  - `SenderName`;
  - `Subject`;
  - `Body`;
  - `MessageType`;
  - `UpdatedAt`.
- Добавить enum `MessageType`:
  - `Transactional` / информационное;
  - `Advertising` / рекламное.
- Ограничения:
  - имя отправителя обязательно;
  - тема обязательна;
  - текст обязателен;
  - максимальная длина темы — 160 символов;
  - максимальная длина имени отправителя — 80 символов;
  - тело письма для dev-среза — plain text / simple text, HTML-редактор не подключать.

Acceptance criteria:

- Модель не принимает пустую тему, отправителя и тело письма.
- Тип письма сохраняется и влияет на требование рекламного согласия.

### T3-07. Добавить application service редактора письма

**Цель:** вынести правила сохранения письма из endpoint.

Задачи:

- Добавить интерфейс `IMailingMessageService`.
- Добавить команду `SaveMailingMessageCommand`.
- Проверять:
  - рассылка принадлежит пользователю;
  - адреса импортированы;
  - декларация подтверждена;
  - если тип письма рекламный, в декларации есть рекламное согласие;
  - обязательные поля письма заполнены.
- Сохранять `MailingMessageDraft` в `Mailing`.
- Писать audit log событие `mailing_message_saved`.

Acceptance criteria:

- Без декларации нельзя сохранить письмо как готовое к следующему шагу.
- Рекламное письмо без рекламного согласия не сохраняется.
- Ошибки возвращаются на русском языке.

### T3-08. Добавить UI редактора письма

**Цель:** пользователь пишет письмо после декларации.

Задачи:

- Добавить route `GET /mailings/{id}/message`.
- Добавить route `POST /mailings/{id}/message`.
- Поля формы:
  - имя отправителя;
  - тема письма;
  - тип письма;
  - текст письма.
- Добавить preview служебных блоков:
  - ссылка отписки;
  - «Почему вы получили это письмо»;
  - служебный идентификатор рассылки.
- После сохранения показывать страницу предпросмотра или карточку рассылки со статусом «Письмо подготовлено».
- Добавить ссылку «Назад к подтверждению базы».

Acceptance criteria:

- Пользователь проходит `импорт → декларация → редактор` без ручного ввода URL.
- Preview показывает пользовательский текст и автоматически добавленные блоки.

### T3-09. Добавить генерацию служебных блоков письма

**Цель:** подготовить обязательные элементы письма до будущей отправки.

Задачи:

- Добавить сервис `IMessageRenderingService` или аналог.
- Генерировать:
  - служебный идентификатор рассылки;
  - блок «Почему вы получили это письмо» на основе источника базы;
  - ссылку глобальной отписки с token placeholder;
  - plain text версию письма.
- На Sprint 3 допускается dev-placeholder для unsubscribe URL, например `/unsubscribe/{token}`.
- Token должен быть стабильным для пары `mailingId + recipientEmail` либо иметь явно описанный временный dev-подход.

Acceptance criteria:

- Unit-тест проверяет наличие unsubscribe URL в preview.
- Unit-тест проверяет наличие блока «Почему вы получили это письмо».
- Unit-тест проверяет наличие служебного идентификатора.

### T3-10. Добавить unsubscribe token dev-генератор

**Цель:** подготовить основу для глобальной отписки, не реализуя полный публичный flow отписки.

Задачи:

- Добавить интерфейс `IUnsubscribeTokenService`.
- Реализовать dev-вариант генерации token.
- Не хранить секреты в коде; если нужен секрет, использовать configuration placeholder.
- Зафиксировать, что публичная обработка `/unsubscribe/{token}` будет реализована в отдельном спринте.

Acceptance criteria:

- Token генерируется детерминированно или сохраняется в in-memory структуре.
- В preview письма есть ссылка глобальной отписки.

### T3-11. Обновить навигацию и статусы flow

**Цель:** сделать сценарий понятным пользователю.

Задачи:

- На странице рассылки показывать текущий статус и следующий шаг:
  - нет адресов → «Загрузить адреса»;
  - адреса загружены, декларации нет → «Подтвердить базу»;
  - декларация есть, письма нет → «Написать письмо»;
  - письмо есть → «Перейти к проверке и оплате» как disabled/заглушка Sprint 4.
- На dashboard показывать обновлённые статусы рассылок.
- Не показывать пользователю технические названия статусов.

Acceptance criteria:

- Пользователь всегда видит, что делать дальше.
- Кнопка оплаты в Sprint 3 не запускает оплату, а явно помечена как следующий спринт/заглушка.

## 8. Технический долг, который выявлен перед Sprint 3

Эти пункты не должны блокировать Sprint 3, но их нужно учитывать:

- Текущий UI генерируется строками в `HtmlRenderer` и endpoint-классах; при росте форм декларации/редактора может потребоваться перенос на Razor Pages/partial views.
- Хранилище in-memory подходит для dev-среза, но декларации и письма являются юридически/операционно значимыми данными, поэтому при переходе к EF Core их нужно мигрировать первыми.
- XLSX-импорт из Sprint 2 не реализован; он не нужен для Sprint 3 CSV-flow, но остаётся открытым долгом.
- Модель `Mailing` сейчас хранит русскоязычный статус строкой; для дальнейших переходов к оплате/отправке лучше добавить внутренний enum статуса и отдельно UI-label.

## 10. Definition of Done

Sprint 3 считается завершённым, если:

- пользователь проходит полный сценарий `создать рассылку → импорт CSV → декларация → редактор письма`;
- без декларации нельзя перейти к сохранению письма/следующему шагу;
- рекламный тип письма требует отдельного рекламного согласия;
- декларация сохраняет версию текста, дату, пользователя, IP и user-agent;
- письмо сохраняет имя отправителя, тему, текст и тип письма;
- preview содержит unsubscribe link, причину получения письма и служебный идентификатор;
- добавлены unit-тесты и, по возможности, интеграционные тесты HTTP-flow;
- `dotnet build Pismolet.sln` и `dotnet test Pismolet.sln` проходят;
- README обновлён после реализации, но не в рамках этого backlog-файла.
