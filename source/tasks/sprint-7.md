# Sprint 7 — глобальная отписка и публичная unsubscribe-страница

Ветка: `Development`  
Основание: `docs/sprints.md`, раздел «Спринт 7 — глобальная отписка и публичная unsubscribe-страница»  
Дата актуализации: 2026-06-20  
Статус: backlog для реализации

## 1. Текущий срез кода

По состоянию ветки `Development` перед Sprint 7:

- приложение собрано как .NET / ASP.NET Core vertical slice с проектами `Pismolet.Domain`, `Pismolet.Application`, `Pismolet.Infrastructure`, `Pismolet.Web`, `Pismolet.Web.Tests`, `Pismolet.IntegrationTests`;
- в `Program.cs` уже подключены `MapPaymentEndpoints()`, `MapCheckEndpoints()`, `MapSendEndpoints()`, `MapAdminEndpoints()`;
- Sprint 6 должен дать страницу отправки, запуск/продолжение отправки, fake email provider flow, summary отправки и dev-сводку событий;
- в пользовательском UI уже предусмотрены агрегированные счётчики для исключённых по глобальной отписке адресов;
- в summary отправки должен быть показатель «Отписано / исключено»;
- публичного unsubscribe endpoint, защищённого unsubscribe token, устойчивого `GlobalSuppression` persistence и полноценного сценария отписки пока нет;
- текущая модель `Recipient` не содержит собственного `Id`, поэтому реализация Sprint 7 не должна бездумно ссылаться на несуществующий `recipientId`;
- текущий preview письма не должен создавать рабочие unsubscribe tokens: реальные tokens допустимы только при построении фактически отправляемого письма для конкретного адресата.

Sprint 7 продолжает существующий MVP-flow, а не создаёт отдельный маркетинговый модуль.

## 2. Цель Sprint 7

Получатель может перейти по ссылке из письма и глобально отписаться от писем через сервис. После этого email больше не должен получать письма через SimpleMail / «Письмолёт» ни от одного клиента.

Рабочий сценарий Sprint 7:

```text
создать рассылку → загрузить CSV → подтвердить базу → написать письмо → оплатить fake-оплатой → пройти проверку → отправить через fake provider → открыть unsubscribe link из fake-письма → подтвердить отписку → email попадает в GlobalSuppression → повторный импорт/отправка исключают этот email
```

## 3. Обязательные рамки Sprint 7

- Отписка глобальная для всего сервиса, а не только для одного клиента.
- Unsubscribe-страница публичная: получатель не должен входить в аккаунт.
- Ссылка должна быть защищена от перебора и подделки.
- Один и тот же email можно отписать повторно без ошибки: операция идемпотентна.
- Клиент не должен видеть лишние сведения: кто, когда, из какой рассылки и от какого клиента отписался.
- При импорте и перед отправкой отписанные адреса исключаются без раскрытия деталей.
- Перед фактической отправкой нужно повторно проверять `GlobalSuppression`, даже если адрес был принят при импорте раньше.
- Публичные ответы unsubscribe flow должны быть безопасными и одинаково спокойными для валидных/повторных сценариев.
- Жалобы `complaint` из Sprint 8 должны в будущем использовать тот же механизм `GlobalSuppression`, поэтому Sprint 7 не должен замыкать модель только на клике по ссылке.
- Tracking pixel, delivery/bounce/complaint webhooks и обработка ответов получателей не входят в Sprint 7: это Sprint 8 и Sprint 9.
- Пользовательский язык интерфейса — русский.

## 4. Задачи реализации

### T7-01. Добавить доменную модель глобальной отписки

**Цель:** хранить факт глобальной отписки как устойчивое доменное состояние.

Задачи:

- Добавить сущность `GlobalSuppression` или аналог.
- Минимальные поля:
  - `Id`;
  - `EmailNormalized`;
  - `EmailHash`, если нужен безопасный поиск/логирование без лишнего раскрытия;
  - `Source` (`UnsubscribeLink`, позже `Complaint`, опционально `Admin`);
  - `SourceMailingId` nullable;
  - `SourceRecipientKey` nullable, если используется stable recipient key вместо `Recipient.Id`;
  - `CreatedAt`;
  - `CreatedIpHash` или безопасный вариант хранения IP, если нужно для аудита;
  - `UserAgentHash` или обрезанный/нормализованный user-agent, если нужно для аудита.
- Добавить уникальность по `EmailNormalized`.
- Не хранить в `GlobalSuppression` лишние сведения о клиенте в пользовательски доступном виде.
- Добавить value object / helper для нормализации email:
  - trim;
  - lowercase domain/local для MVP;
  - единая нормализация во всех местах: импорт, отправка, unsubscribe.

Acceptance criteria:

- Один email может существовать в `GlobalSuppression` только один раз.
- Повторное добавление того же email не создаёт дубль.
- Нормализация email совпадает для импорта, отправки и unsubscribe.
- Доменная модель не раскрывает клиенту данные о чужих рассылках.

### T7-02. Добавить persistence и repository для suppression list

**Цель:** сделать глобальную отписку устойчивой после рестарта приложения.

Задачи:

- Добавить интерфейс, например `IGlobalSuppressionRepository`.
- Минимальные методы:
  - `IsSuppressedAsync(normalizedEmail)`;
  - `AddOrGetAsync(...)` / idempotent add;
  - `GetByEmailAsync(...)` только для внутренних/admin-сценариев;
  - batch-check для импорта и отправки.
- Реализовать EF Core/PostgreSQL persistence.
- Добавить `DbSet` и entity configuration.
- Добавить миграцию для таблицы `global_suppressions`.
- Добавить индекс/unique constraint по `EmailNormalized`.
- Для массовых проверок не делать N запросов на N email: предусмотреть batch lookup.
- In-memory реализация допустима только для unit-тестов/fallback, но не как основное состояние Sprint 7.

Acceptance criteria:

- После рестарта dev-приложения отписанный email остаётся отписанным.
- Batch-проверка адресов работает для импорта и отправки.
- Повторная отписка не падает на unique constraint, а возвращает успешный идемпотентный результат.

### T7-03. Реализовать защищённый unsubscribe token

**Цель:** сделать ссылку отписки неперебираемой и неподлежающей простой подделке.

Задачи:

- Добавить сервис `IUnsubscribeTokenService` или аналог.
- Токен должен включать минимум:
  - normalized email или безопасный email id;
  - `mailingId`;
  - `recipientId` или stable recipient key;
  - purpose/version;
  - срок действия, если выбран expiring token;
  - подпись/HMAC/Data Protection payload.
- С учётом текущей модели `Recipient`:
  - нельзя обращаться к несуществующему `Recipient.Id`;
  - нужно выбрать один из вариантов:
    - добавить стабильный `Recipient.Id` в доменную модель и persistence;
    - либо использовать stable recipient key, например hash от `mailingId + normalizedEmail + importBatchId`;
  - выбранный вариант должен быть описан в комментарии/ADR или в коде рядом с token service.
- Использовать ASP.NET Core Data Protection или HMAC с секретом из конфигурации.
- Добавить версионирование токена, чтобы в будущем можно было менять формат.
- Не класть в URL сырой email в открытом виде, если выбран вариант payload + подпись.
- Обработать ошибки:
  - невалидный токен;
  - истёкший токен;
  - token purpose mismatch;
  - рассылка/получатель не найдены.

Acceptance criteria:

- Валидный токен позволяет определить email для отписки.
- Изменение одного символа токена делает его невалидным.
- Невалидный токен не раскрывает, существует ли email или рассылка.
- Token service не зависит от несуществующих полей модели.
- Unit-тесты покрывают генерацию, проверку и tamper-сценарий.

### T7-04. Встроить реальную unsubscribe-ссылку в письмо

**Цель:** заменить заглушку/служебный блок на рабочую ссылку для каждого получателя.

Задачи:

- При построении письма для фактической отправки формировать recipient-specific unsubscribe URL.
- Добавлять ссылку в HTML и plain text части письма.
- Сохранять общий текст служебного блока:
  - почему получатель получил письмо;
  - как отписаться;
  - что отписка действует для всего сервиса.
- Не использовать один общий токен на всю рассылку, если по нему нельзя безопасно определить получателя.
- Реальные tokens генерируются только при построении отправляемого письма для конкретного recipient.
- Preview в редакторе, preview в модерации и любые admin/dev previews должны показывать только пример ссылки или нейтральную заглушку, не создающую реальную отписку.
- Для fake provider в dev-сценарии сделать ссылку доступной для ручного теста: либо в dev-сводке события, либо в fake provider outbox, либо в тексте сохранённого fake message.

Acceptance criteria:

- В письме каждому recipient соответствует собственная unsubscribe-ссылка.
- Plain text версия тоже содержит unsubscribe URL.
- Preview/editor/admin preview не создаёт рабочий token.
- Ручной тест может открыть ссылку из fake-письма.

### T7-05. Добавить публичные unsubscribe endpoints и страницу подтверждения

**Цель:** получатель может отписаться без аккаунта через безопасный публичный flow.

Задачи:

- Добавить `MapUnsubscribeEndpoints()` в `Program.cs`.
- Реализовать endpoint просмотра:
  - `GET /unsubscribe/{token}` или `GET /u/{token}`;
  - показывает понятную страницу подтверждения.
- Реализовать endpoint подтверждения:
  - `POST /unsubscribe/{token}`;
  - добавляет email в `GlobalSuppression` идемпотентно.
- На странице не показывать название клиента, внутренний ID рассылки, полный email, если это не нужно.
- Допустимо показать маскированный email, например `a***@example.com`, если токен валиден.
- Для невалидного токена показывать нейтральную ошибку без раскрытия деталей.
- После успешной отписки показывать страницу «Вы отписаны от писем через сервис».
- Повторный переход по ссылке должен показывать успешный/нейтральный результат, а не ошибку.

Acceptance criteria:

- Получатель может пройти unsubscribe flow без логина.
- `POST` по валидному токену создаёт или подтверждает `GlobalSuppression`.
- Повторный `POST` не создаёт дубль и не падает.
- Невалидный токен не раскрывает существование email/рассылки.

### T7-06. Интегрировать suppression list в импорт адресов

**Цель:** отписанные email не должны попадать в принятые адреса новых рассылок.

Задачи:

- В `IRecipientImportService` / import pipeline добавить batch-проверку `GlobalSuppression`.
- Импорт должен сначала собрать normalized emails, затем одним запросом получить suppressed set, а не делать N запросов на N строк.
- При совпадении с suppression list:
  - не добавлять адрес в accepted recipients;
  - увеличивать счётчик `GloballySuppressed`;
  - логировать безопасное событие попытки добавления отписанного адреса.
- Не показывать клиенту список конкретных отписавшихся адресов.
- В UI оставить только агрегированный счётчик «Исключены по глобальной отписке».
- Проверить, что повторная загрузка CSV с отписанным email не меняет suppression state.

Acceptance criteria:

- Отписанный email исключается при новом импорте.
- Клиент видит только количество исключённых адресов, без деталей.
- Дубли, invalid и suppressed считаются независимо и предсказуемо.
- Интеграционный тест покрывает импорт CSV после отписки.
- Batch lookup используется в реализации, а не только описан в интерфейсе.

### T7-07. Интегрировать suppression list в отправку

**Цель:** отписанный email не должен получить письмо, даже если был принят до отписки.

Задачи:

- Перед вызовом fake email provider проверять recipient email в `GlobalSuppression`.
- Проверка должна выполняться внутри batch-send job/application service, а не только в UI.
- Если email suppressed:
  - не вызывать provider;
  - создать/обновить `SendEvent` со статусом `Skipped` и причиной `GlobalSuppression`;
  - увеличить summary `Suppressed` / «Отписано / исключено».
- Сохранить идемпотентность повторного запуска/refresh/resume.
- Убедиться, что повторная отправка не создаёт дубли skipped events.

Acceptance criteria:

- Email, отписавшийся после импорта, но до отправки, не получает письмо.
- Для suppressed recipient нет provider message id.
- Summary отправки показывает исключение.
- Повторный запуск отправки не создаёт дубли skipped events.

### T7-08. Добавить audit log для unsubscribe-сценариев

**Цель:** фиксировать важные действия без лишнего раскрытия персональных данных.

Задачи:

- Логировать:
  - успешную отписку;
  - повторную отписку;
  - невалидный/поддельный токен;
  - истёкший токен, если срок действия используется;
  - попытку импорта suppressed email;
  - пропуск suppressed email перед отправкой.
- В audit event хранить безопасный минимум:
  - тип события;
  - normalized/hash email;
  - source mailing id, если есть;
  - timestamp;
  - IP/user-agent в безопасном виде;
  - correlation id/request id, если уже используется.
- Не писать в пользовательский UI подробности audit log.

Acceptance criteria:

- По внутреннему журналу можно понять, что отписка сработала.
- Клиент не получает доступ к деталям отписки конкретного адресата.
- Невалидные токены логируются без утечки payload.

### T7-09. Зафиксировать границы admin-функций по suppression list

**Цель:** не расширять Sprint 7 небезопасным admin UI, пока в проекте нет полноценной admin/RBAC-модели.

Задачи:

- Полноценный admin UI для просмотра/ручного добавления global suppression не является обязательной частью Sprint 7 по `docs/sprints.md`.
- Основной результат Sprint 7 — публичная отписка получателя и применение `GlobalSuppression` при импорте/отправке.
- Если для разработки нужен внутренний инструмент ручного добавления email в suppression list, он допускается только при одном из условий:
  - route доступен только в Development environment;
  - или включена временная admin policy / allowlist из конфигурации;
  - или уже реализована отдельная admin authorization policy.
- Нельзя добавлять ручное управление suppression list под обычное `.RequireAuthorization()` без admin-ограничения.
- Удаление из suppression list в MVP не делать без отдельного решения/ADR.
- Если admin UI не реализован в Sprint 7, добавить TODO/ссылку на Sprint 10, где планируется полноценная админка.

Acceptance criteria:

- В Sprint 7 не появляется новый admin action, доступный любому авторизованному пользователю.
- Public unsubscribe flow работает независимо от admin UI.
- Если dev/admin ручное добавление реализовано, оно ограничено environment/policy/allowlist и логируется.
- Клиентский UI не получает новых деталей об отписках.

## 5. Unit-тесты

Обязательные unit-тесты Sprint 7:

- генерация unsubscribe token;
- проверка unsubscribe token;
- tampered token отклоняется;
- истёкший token отклоняется, если используется срок действия;
- token service работает с выбранным `recipientId` или stable recipient key и не зависит от несуществующего поля;
- preview/editor/admin preview не создаёт рабочий unsubscribe token;
- повторная отписка идемпотентна;
- нормализация email едина для unsubscribe/import/send;
- suppressed email исключается из импорта;
- import pipeline использует batch suppression lookup;
- suppressed email исключается перед отправкой;
- skipped send event создаётся с причиной `GlobalSuppression`;
- клиенту не возвращаются детали конкретного suppressed email;
- admin/dev ручное добавление, если реализовано, требует admin policy/allowlist/environment guard.

## 6. Интеграционные тесты

Обязательные интеграционные тесты Sprint 7:

- `GET /unsubscribe/{token}` показывает страницу подтверждения без логина;
- `POST /unsubscribe/{token}` создаёт `GlobalSuppression`;
- повторный `POST /unsubscribe/{token}` не создаёт дубль;
- невалидный token возвращает безопасную страницу/ошибку;
- после рестарта dev-приложения / пересоздания service scope `GlobalSuppression` сохраняется в PostgreSQL;
- новая загрузка CSV с этим email увеличивает `GloballySuppressed` и не добавляет адрес в accepted recipients;
- отправка рассылки, где recipient успел отписаться после импорта, создаёт skipped event и не вызывает fake provider;
- real unsubscribe token появляется в fake-письме/outbox, но не появляется в preview;
- если ручное admin/dev добавление реализовано, оно недоступно обычному авторизованному пользователю.

## 7. Ручные тесты

Проверить вручную:

1. Создать тестовую рассылку и отправить через fake provider.
2. Найти unsubscribe link в fake-письме/dev outbox.
3. Открыть ссылку в браузере без авторизации.
4. Подтвердить отписку.
5. Повторно открыть ту же ссылку и убедиться, что нет ошибки.
6. Загрузить новый CSV с этим email.
7. Проверить, что адрес исключён и клиент видит только агрегированный счётчик.
8. Создать рассылку, импортировать email, затем отписать его до запуска отправки.
9. Запустить отправку и проверить, что fake provider не получает этот email.
10. Проверить, что клиент не видит дату/источник/клиента отписки.
11. Проверить, что preview письма показывает только пример/заглушку unsubscribe-ссылки, а не рабочий token.
12. Перезапустить dev-приложение и проверить, что email остаётся в global suppression.
13. Если реализован dev/admin инструмент ручного добавления, проверить, что обычный пользователь не имеет к нему доступа.

## 8. Definition of Done

Sprint 7 считается закрытым, если:

- публичная unsubscribe-ссылка реально создаётся для отправляемых писем;
- real unsubscribe token создаётся только для конкретного recipient при фактической отправке, не в preview;
- token защищён от подделки и не зависит от несуществующего `Recipient.Id`;
- получатель может отписаться без аккаунта;
- email попадает в устойчивый `GlobalSuppression` в PostgreSQL;
- после рестарта dev-приложения отписка сохраняется;
- повторная отписка идемпотентна;
- отписанный email исключается при импорте через batch lookup;
- отписанный email повторно проверяется перед отправкой;
- fake provider не вызывается для suppressed recipient;
- клиент видит только безопасные агрегированные счётчики;
- audit log фиксирует unsubscribe-события;
- нет нового admin action, доступного любому авторизованному пользователю;
- есть unit, integration и manual coverage;
- реализация не мешает будущему Sprint 8 добавлять complaint в тот же `GlobalSuppression`.
