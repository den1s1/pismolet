# Sprint 10 — админка, лимиты, блокировки, аудит

Ветка: `Development`  
Основание: `docs/sprints.md`, раздел «Спринт 10 — админка, лимиты, блокировки, аудит»  
Дата актуализации: 2026-06-21  
Статус: backlog для реализации

## 1. Текущий срез кода

По состоянию ветки `Development` перед Sprint 10:

- Sprint 4 должен дать fake-оплату, `Payment`, `PaymentAttempt`, `PriceSettings`, расчёт стоимости и статус `paid`;
- Sprint 5 должен дать risk-проверку, `RiskCheckResult`, `ModerationReview`, очередь ручной модерации и действия администратора `approve` / `reject` / `block`;
- Sprint 6 должен дать отправку через очередь, fake email provider, `SendEvent`, provider message id, дневные лимиты и базовую статистику отправки;
- Sprint 7 должен дать глобальную отписку, `GlobalSuppression`, публичный unsubscribe flow и исключение suppressed email из будущих отправок;
- Sprint 8 должен дать delivery/bounce/complaint webhooks, обработку жалоб, hard bounce и provider event log;
- Sprint 9 должен дать `ReplyEvent`, inbound replies, пересылку ответа клиенту и cleanup временно хранимых тел ответов;
- Sprint 10 объединяет эти механики в управляемую админскую поверхность и не должен переписывать бизнес-flow рассылок;
- пользовательский результат Sprint 10 — администратор видит ключевые риски MVP и может управлять клиентами, лимитами, блокировками, премодерацией и ценой письма;
- все административные действия, влияющие на отправку, цену, лимиты, блокировки, премодерацию и доступ к данным, должны попадать в audit log.

Sprint 10 — это слой операционного контроля MVP. Он должен быть простым, но достаточным для ручного управления рисками, жалобами, лимитами и спорными клиентами.

## 2. Цель Sprint 10

Администратор может управлять рисками MVP: смотреть состояние клиентов и рассылок, ограничивать отправку, блокировать опасные объекты, менять лимиты и цену, включать премодерацию и видеть историю административных действий.

Рабочий сценарий Sprint 10:

```text
админ входит в админку → открывает список клиентов → видит риски/лимиты/статусы → открывает карточку клиента → меняет дневной лимит или включает премодерацию → действие фиксируется в audit log → клиентские сценарии отправки начинают учитывать новые ограничения → админ может найти рассылку/жалобу/ошибку доставки и при необходимости заблокировать клиента или рассылку
```

## 3. Обязательные рамки Sprint 10

- Админка должна быть отделена от клиентского UI и явно защищена от доступа обычного пользователя.
- Для MVP допускается простая admin-auth модель, но в коде должно быть видно, где находится граница доступа администратора.
- Нельзя показывать админские страницы без проверки роли/признака администратора даже в dev-сценарии.
- Админка не должна раскрывать raw provider payload, raw email body, токены отписки, reply tokens и секретные технические ключи без отдельной необходимости.
- Все изменения лимитов, блокировок, цены, премодерации и статусов должны фиксироваться в audit log.
- Audit log должен хранить факт действия, администратора, объект, время, старое/новое значение там, где это безопасно и полезно.
- Изменение цены письма должно применяться к новым расчётам, но не должно задним числом менять уже созданные `PaymentAttempt` / оплаченные рассылки.
- Изменение лимитов должно применяться к будущим постановкам в очередь и будущим batch-отправкам.
- Заблокированный клиент не может запускать новые рассылки и не должен обходить блокировку через refresh, повторный callback или повторную background job.
- Заблокированная рассылка не может быть отправлена, даже если ранее была оплачена или одобрена.
- Включение премодерации клиенту должно влиять на последующие рассылки и risk-flow.
- Настройка премодерации для новых клиентов должна применяться при создании новых клиентских профилей.
- Админские списки должны быть пагинированы или хотя бы иметь server-side limit для MVP.
- Пользовательский язык интерфейса — русский.

## 4. Задачи реализации

### T10-01. Защитить административную зону

**Цель:** отделить admin UI/API от обычного пользовательского доступа.

Задачи:

- Ввести единый guard/middleware/filter для admin endpoints/pages.
- Определить MVP-признак администратора:
  - роль `Admin`, claim, dev-admin flag или отдельная таблица пользователей;
  - способ назначения администратора для локальной разработки.
- Запретить доступ к `/admin/*` обычному пользователю.
- Для unauthenticated пользователя возвращать понятный redirect/401/403 по принятой модели приложения.
- Не полагаться только на скрытие ссылок в UI: backend endpoints тоже должны проверять доступ.
- Добавить явный warning, если dev-admin режим включён в production-like конфигурации.
- Проверить, что admin partials/components не подключаются в клиентский layout без проверки доступа.

Acceptance criteria:

- Обычный пользователь не может открыть admin pages и выполнить admin actions.
- Admin endpoints защищены на backend уровне.
- В коде есть единое место, где видна политика доступа администратора.
- Тесты покрывают запрет доступа обычному пользователю.

### T10-02. Сформировать навигацию и layout админки

**Цель:** сделать единую точку входа для операционного управления MVP.

Задачи:

- Добавить/расширить `/admin` dashboard.
- Добавить навигацию по разделам:
  - клиенты;
  - рассылки;
  - импорты;
  - платежи;
  - лимиты;
  - отписки;
  - жалобы;
  - ошибки доставки;
  - ответы;
  - audit log;
  - настройки.
- На dashboard вывести краткие счётчики:
  - клиенты всего / заблокированы / на премодерации;
  - рассылки на проверке;
  - рассылки с ошибками отправки;
  - жалобы за период;
  - hard bounce за период;
  - глобальные отписки за период;
  - последние административные действия.
- Не делать сложную аналитику: dashboard должен быть операционным.
- Добавить пустые состояния для разделов, где данных ещё нет.
- Использовать русский язык интерфейса.

Acceptance criteria:

- Администратор видит единый dashboard.
- Из dashboard можно перейти в основные admin-разделы.
- Счётчики берутся из persistence/сервисов, а не захардкожены.
- Пустые состояния не выглядят как ошибка.

### T10-03. Реализовать список и карточку клиента

**Цель:** дать администратору обзор клиента и быстрый доступ к ограничениям.

Задачи:

- Добавить admin list клиентов с фильтрами:
  - активные;
  - заблокированные;
  - с включённой премодерацией;
  - новые клиенты;
  - клиенты с жалобами / bounce / ошибками.
- В карточке клиента показать:
  - идентификатор клиента;
  - email/название, если есть;
  - статус блокировки;
  - дневной лимит;
  - общий лимит, если используется;
  - признак обязательной премодерации;
  - дата создания;
  - краткие счётчики рассылок, оплат, отписок, жалоб, ошибок;
  - последние рассылки;
  - последние audit actions по клиенту.
- Не показывать лишние персональные данные адресатов в карточке клиента.
- Добавить быстрые действия:
  - заблокировать клиента;
  - разблокировать клиента;
  - изменить лимит;
  - включить/выключить премодерацию.

Acceptance criteria:

- Администратор может найти клиента и открыть карточку.
- Карточка показывает состояние, достаточное для решения о рисках.
- Все быстрые действия защищены и логируются.
- Обычный пользователь не может открыть карточку клиента в admin UI.

### T10-04. Реализовать блокировку клиента

**Цель:** дать возможность остановить клиента, создающего риск для сервиса.

Задачи:

- Добавить/использовать поля блокировки клиента:
  - `IsBlocked`;
  - `BlockedAt`;
  - `BlockedByAdminId`;
  - `BlockReason`;
  - `UnblockedAt` nullable;
  - `UnblockedByAdminId` nullable.
- Реализовать admin action `BlockClient`.
- Реализовать admin action `UnblockClient`.
- Проверять блокировку клиента в точках:
  - создание новой рассылки;
  - переход к оплате;
  - запуск проверки;
  - постановка в очередь отправки;
  - batch-отправка;
  - повторный запуск background job.
- Для уже идущей отправки определить MVP-поведение:
  - либо `paused`/`blocked` при следующем batch;
  - либо отказ от постановки новых batch.
- Показать клиенту понятный русский статус без раскрытия внутренних причин, если блокировка не должна быть публичной.
- Записывать audit log при блокировке и разблокировке.

Acceptance criteria:

- Заблокированный клиент не может запустить новую отправку.
- Блокировка применяется не только в UI, но и в backend/background jobs.
- Разблокировка восстанавливает возможность работы, если нет других ограничений.
- Действие попадает в audit log.

### T10-05. Реализовать блокировку рассылки

**Цель:** остановить конкретную опасную или спорную рассылку без полной блокировки клиента.

Задачи:

- Добавить/использовать статус или флаг блокировки рассылки:
  - `Blocked` / `IsBlocked`;
  - `BlockedAt`;
  - `BlockedByAdminId`;
  - `BlockReason`.
- Реализовать admin action `BlockMailing`.
- Реализовать admin action `UnblockMailing`, если разблокировка разрешена в MVP.
- Проверять блокировку рассылки в точках:
  - отправка теста, если применимо;
  - запуск risk-check;
  - approve после moderation;
  - постановка в очередь;
  - batch job;
  - retry failed sends.
- Если рассылка уже оплачена, блокировка не должна автоматически делать refund в MVP; статус должен быть понятным для ручного решения.
- Связать блокировку рассылки с moderation review, если блокировка сделана из очереди модерации.
- Записывать audit log.

Acceptance criteria:

- Заблокированная рассылка не отправляется.
- Повторные background jobs не обходят блокировку.
- Администратор видит причину и историю блокировки.
- Действие попадает в audit log.

### T10-06. Реализовать управление лимитами

**Цель:** дать администратору управляемый способ ограничивать объём отправки.

Задачи:

- Добавить admin UI для изменения лимитов клиента:
  - дневной лимит отправки;
  - общий лимит, если используется;
  - возможно, лимит получателей в одной рассылке.
- Валидация:
  - лимит не может быть отрицательным;
  - нулевой лимит трактовать явно: полный запрет отправки или недопустимое значение;
  - большие значения требуют подтверждения, если есть UI-confirm.
- Применять лимиты к будущим расчётам, queueing и batch-send.
- В sender job учитывать, что лимит мог измениться после постановки задачи в очередь.
- В карточке клиента показывать текущий лимит и дату последнего изменения.
- Логировать старое и новое значение в audit log.

Acceptance criteria:

- Admin может изменить лимит клиента.
- Изменённый лимит применяется к отправке.
- Batch job не превышает актуальный лимит.
- Audit log показывает кто, когда и какое значение изменил.

### T10-07. Реализовать управление премодерацией

**Цель:** дать администратору возможность усилить ручной контроль для отдельных клиентов и новых клиентов.

Задачи:

- В карточке клиента добавить действие:
  - включить обязательную премодерацию;
  - выключить обязательную премодерацию.
- Добавить настройку «премодерация для новых клиентов».
- При создании нового клиента применять текущую global setting.
- В risk-flow Sprint 5 учитывать client-level forced moderation.
- В admin UI показывать, почему рассылка попала на moderation:
  - risk-score;
  - forced moderation для клиента;
  - forced moderation для новых клиентов;
  - конкретные risk flags.
- Логировать изменения client/global setting.

Acceptance criteria:

- Admin может включить премодерацию конкретному клиенту.
- Новые клиенты получают настройку согласно global setting.
- Risk-flow учитывает forced moderation.
- Audit log фиксирует изменения.

### T10-08. Реализовать управление ценой письма

**Цель:** дать администратору MVP-настройку стоимости письма без изменения кода.

Задачи:

- Добавить admin settings page для `PriceSettings`.
- Показать текущую цену письма и дату изменения.
- Изменение цены должно требовать явного подтверждения.
- Применять новую цену только к новым расчётам.
- Не пересчитывать уже созданные `PaymentAttempt`.
- Не менять уже оплаченные рассылки.
- В audit log записывать старую и новую цену.
- В UI расчёта стоимости показывать цену, которая фактически применена к конкретной рассылке/payment attempt.

Acceptance criteria:

- Admin меняет цену письма через admin UI.
- Новые расчёты используют новую цену.
- Старые payment attempts сохраняют старую цену.
- Изменение цены видно в audit log.

### T10-10. Реализовать раздел отписок, жалоб и ошибок доставки

**Цель:** дать администратору обзор deliverability-рисков и антиспам-сигналов.

Задачи:

- Раздел глобальных отписок:
  - email masked/normalized preview;
  - дата отписки;
  - source: public unsubscribe / complaint / admin / import;
  - связанная рассылка nullable.
- Раздел жалоб:
  - provider;
  - provider event id;
  - client/mailing;
  - received at;
  - action applied: global suppression, block, manual review.
- Раздел ошибок доставки:
  - soft bounce;
  - hard bounce;
  - rejected;
  - provider error;
  - retry status.
- Фильтры по клиенту, рассылке, типу события и периоду.
- Быстрые действия:
  - открыть клиента;
  - открыть рассылку;
  - заблокировать клиента;
  - заблокировать рассылку.
- Не показывать raw provider payload по умолчанию.
- Добавить безопасный technical details view только для admin/debug, если нужен.

Acceptance criteria:

- Admin видит жалобы и ошибки доставки.
- Complaint можно проследить до клиента/рассылки, если matching был успешным.
- Hard bounce и complaint влияют на будущие отправки согласно Sprint 8.
- Разделы не раскрывают лишние персональные данные.

### T10-11. Реализовать единый audit log

**Цель:** фиксировать административные действия, влияющие на риски и деньги.

Задачи:

- Добавить/уточнить сущность `AuditLogEntry`.
- Минимальные поля:
  - `Id`;
  - `ActorUserId` / `AdminId`;
  - `ActorEmail` или safe identifier;
  - `ActionType`;
  - `TargetType`;
  - `TargetId`;
  - `ClientId` nullable;
  - `MailingId` nullable;
  - `OccurredAt`;
  - `IpAddress`;
  - `UserAgent`;
  - `OldValueJson` safe nullable;
  - `NewValueJson` safe nullable;
  - `Reason` nullable;
  - `CorrelationId` / request id.
- Покрыть audit log действиями:
  - block/unblock client;
  - block/unblock mailing;
  - change daily limit;
  - change total limit;
  - enable/disable client moderation;
  - change new-client moderation setting;
  - change price settings;
  - approve/reject moderation;
  - manual suppression/blocklist action, если реализуется;
  - просмотр raw/debug details, если такой просмотр добавлен.
- Не писать в audit log секретные токены, полные raw payload, тела писем и лишние персональные данные.
- Добавить admin page `/admin/audit` с фильтрами.

Acceptance criteria:

- Все критичные admin actions создают audit entry.
- Audit log можно отфильтровать по клиенту, рассылке, действию и периоду.
- Audit log сохраняет старое/новое значение там, где это безопасно.
- Тесты подтверждают запись audit log.

- Некорректные значения отклоняются.

### T10-13. Обновить сервисные проверки блокировок и лимитов

**Цель:** убедиться, что admin-настройки реально влияют на backend-flow.

Задачи:

- Добавить application-level service/policy для проверки возможности отправки:
  - client not blocked;
  - mailing not blocked;
  - payment ok;
  - moderation ok;
  - limits ok;
  - not globally suppressed;
  - not locally blocked by bounce/blocklist.
- Использовать policy в:
  - старт отправки;
  - постановка в очередь;
  - batch job;
  - retry job;
  - fake/dev sender actions.
- Не размазывать правила блокировок по UI handlers.
- Возвращать понятные failure codes для UI/admin diagnostics.
- Добавить тесты на обходные сценарии:
  - refresh после блокировки;
  - повторный fake payment callback;
  - уже queued job после блокировки;
  - retry после блокировки.

Acceptance criteria:

- Backend-flow не обходит admin-блокировки.
- Лимиты применяются в актуальном состоянии, а не только в момент создания рассылки.
- UI получает понятные статусы и не показывает «успешно», если backend отказал.
- Тесты закрывают обходные сценарии.

### T10-14. Добавить тесты Sprint 10

**Цель:** закрепить поведение админки, блокировок, лимитов и audit log.

Unit-тесты:

- Заблокированный клиент не может отправлять.
- Заблокированная рассылка не отправляется.
- Изменение дневного лимита применяется к отправке.
- Изменение цены применяется только к новым расчётам.
- Forced moderation для клиента переводит рассылку на review.
- Global forced moderation применяется к новым клиентам.
- Audit log создаётся для block/unblock client.
- Audit log создаётся для изменения лимита.
- Audit log создаётся для изменения цены.
- Sensitive данные не попадают в audit payload.

Интеграционные тесты:

- Admin меняет лимит → новая отправка учитывает лимит.
- Admin блокирует клиента → клиент не может запустить рассылку.
- Admin блокирует рассылку → queued/retry job не отправляет её.
- Admin включает премодерацию → следующая рассылка попадает на moderation.
- Admin меняет цену → новый payment attempt использует новую цену.
- Audit log фиксирует последовательность действий.
- Обычный пользователь получает 403/redirect при доступе к `/admin`.

Ручные тесты:

- Зайти админом.
- Открыть dashboard.
- Найти клиента.
- Изменить дневной лимит.
- Заблокировать клиента.
- Проверить, что клиент больше не может запустить рассылку.
- Разблокировать клиента.
- Включить клиенту премодерацию.
- Создать новую рассылку и проверить попадание на review.
- Изменить цену письма и проверить новый расчёт.
- Открыть audit log и найти все действия.
- Открыть раздел жалоб/ошибок доставки и перейти в карточку рассылки.

Acceptance criteria:

- Основные правила Sprint 10 покрыты unit/integration tests.
- Ручной smoke-flow описан и воспроизводим локально.
- Тесты не зависят от реальных email/payment provider.

## 5. Definition of Done

Sprint 10 считается завершённым, когда:

- `/admin` защищён от обычных пользователей;
- администратор видит dashboard и основные разделы MVP;
- есть карточка клиента с лимитами, блокировкой, премодерацией и краткой историей;
- admin может заблокировать/разблокировать клиента;
- admin может заблокировать рассылку;
- admin может изменить лимиты клиента;
- admin может включить/выключить премодерацию клиента;
- admin может изменить настройку премодерации для новых клиентов;
- admin может изменить цену письма для новых расчётов;
- есть разделы рассылок, импортов, платежей, отписок, жалоб, ошибок доставки и audit log;
- критичные действия фиксируются в audit log;
- backend-flow отправки учитывает блокировки и лимиты, а не только UI;
- unit/integration/manual tests из этого файла добавлены или явно отражены в тест-плане;
- реальные платёжные и email-интеграции не подключены: Sprint 10 работает поверх fake provider/fake payment MVP-flow.
