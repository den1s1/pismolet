# Sprint 1 — регистрация, вход и первый экран ЛК

Статус: план задач  
Ветка: `Development`  
Источник: общий план спринтов MVP из репозитория

## Цель спринта

Пользователь может зарегистрироваться, войти в сервис и увидеть простой стартовый экран личного кабинета со сценарием создания рассылки.

## Видимый результат

Пользователь входит в сервис и видит список рассылок, их статусы и кнопку или карточку «Создать рассылку».

## Основные задачи

### 1. Регистрация и аутентификация

- Реализовать регистрацию пользователя.
- Реализовать вход пользователя.
- Реализовать выход пользователя.
- Ограничить доступ к личному кабинету только для авторизованных пользователей.
- Подготовить базовую обработку ошибок регистрации и входа.

### 2. Подтверждение email

- Реализовать подтверждение email через dev/fake mailer.
- Сохранять факт подтверждения email.
- Предусмотреть сценарий повторной отправки письма подтверждения для разработки и тестирования.

### 3. Профиль клиента

- Создавать `ClientProfile` при регистрации пользователя.
- Добавить дефолтные лимиты нового клиента:
  - дневной лимит отправки;
  - общий лимит отправки;
  - флаг обязательной премодерации.
- Поддержать базовые статусы клиента:
  - `active`;
  - `blocked`.

### 4. Первый экран личного кабинета

- Сделать страницу личного кабинета.
- Показать список рассылок пользователя.
- Показать статусы рассылок.
- Добавить кнопку или карточку «Создать рассылку».
- Не показывать пользователю внутренние технические сущности вроде `Campaign`, `ImportBatch`, `SendEvent`, `PaymentAttempt`.
- Использовать русский язык интерфейса.

### 5. Аудит действий

- Добавить записи в audit log для событий:
  - регистрация;
  - вход;
  - выход;
  - подтверждение email.
- Для событий аудита сохранять как минимум:
  - дату и время;
  - пользователя;
  - тип события;
  - технический контекст, достаточный для разбирательства.

## Unit-тесты

- Создание клиента после регистрации.
- Создание `ClientProfile` вместе с пользователем.
- Проверка дефолтных лимитов нового клиента.
- Проверка статуса клиента: `active` / `blocked`.
- Запрет операций для заблокированного клиента, если операция требует активного статуса.
- Создание audit log события регистрации.
- Создание audit log события подтверждения email.

## Интеграционные тесты

- Регистрация пользователя через web-flow.
- Вход пользователя через web-flow.
- Выход пользователя.
- Доступ к личному кабинету только авторизованному пользователю.
- Создание `ClientProfile` после регистрации.
- Подтверждение email через dev/fake mailer.
- Проверка сохранения audit log в базе.

## Ручные тесты

- Зарегистрироваться новым пользователем.
- Подтвердить email в dev/fake сценарии.
- Войти под зарегистрированным пользователем.
- Выйти из аккаунта.
- Открыть личный кабинет неавторизованным пользователем и убедиться, что доступ закрыт.
- Проверить, что в личном кабинете видны:
  - список рассылок;
  - статусы рассылок;
  - кнопка или карточка «Создать рассылку».
- Проверить, что UI соответствует текущим макетам и HTML-шаблонам из `ui`, если они есть в репозитории.

## Definition of Done

- Все задачи спринта реализованы или явно перенесены в следующий спринт с причиной.
- Приложение локально запускается.
- Регистрация, вход, выход и личный кабинет работают в одном вертикальном сценарии.
- Unit-тесты и интеграционные тесты проходят локально и в CI, если CI уже настроен.
- Ручные тесты из этого файла пройдены и отмечены в отчёте спринта.
- Пользовательский интерфейс не раскрывает внутренние технические сущности.
