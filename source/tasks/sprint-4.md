# Sprint 4 — расчёт стоимости и fake-оплата

Ветка: `Development`  
Основание: `docs/sprints.md`, раздел «Спринт 4 — расчёт стоимости и fake-оплата»  
Дата составления: 2026-06-17  
Статус: backlog для реализации

## 1. Текущий срез кода

По состоянию ветки `Development` реализован vertical slice до конца Sprint 3:

- solution `Pismolet.sln` с проектами `Pismolet.Domain`, `Pismolet.Application`, `Pismolet.Infrastructure`, `Pismolet.Web`, `Pismolet.Web.Tests`, `Pismolet.IntegrationTests`;
- UI/endpoints текущего vertical slice находятся преимущественно в `src/Pismolet.Web`, но доменные модели, application services и persistence уже разнесены по `Pismolet.Domain`, `Pismolet.Application`, `Pismolet.Infrastructure`;
- используется ASP.NET Core minimal endpoints и временный `HtmlRenderer`, а не Razor Pages;
- реализованы регистрация, вход, выход и dev/fake подтверждение email;
- создаётся `ClientProfile` с базовыми лимитами;
- есть защищённый `/dashboard`;
- пользователь создаёт черновик рассылки через `/mailings/new` и `POST /mailings`;
- пользователь загружает CSV с колонкой `email` через `/mailings/{id}/recipients`;
- импорт проверяет валидные адреса, дубли, невалидные адреса и глобально отписанные адреса;
- после импорта пользователь проходит `/mailings/{id}/declaration`;
- после декларации пользователь заполняет plain text письмо через `/mailings/{id}/message`;
- preview письма показывает служебные блоки;
- после сохранения письма в UI сейчас показана заглушка «Проверка и оплата — Sprint 4»;
- persistence текущей ветки уже ориентирован на EF Core/PostgreSQL по умолчанию; in-memory допускается только как тестовый/dev fallback;
- dev-only audit log уже используется в текущем сценарии.

В коде пока отсутствуют:

- доменные модели `Payment`, `PaymentAttempt`, `PriceSettings`;
- value object для денег/цены письма, если он не будет вынесен из существующих наработок Sprint 0;
- расчёт стоимости рассылки по принятым адресам;
- экран «Проверка и оплата»;
- fake payment provider;
- стабильные технические статусы рассылки `priced`, `payment_pending`, `paid` и русские UI-подписи к ним;
- идемпотентная обработка повторного fake callback/webhook;
- запрет перехода к отправке без оплаты;
- persistent-хранилище платежей, попыток оплаты и настроек цены.

## 2. Цель Sprint 4

Пользователь после подготовки письма видит понятный экран проверки и оплаты: сколько адресов принято, сколько исключено, сколько стоит отправка, и может пройти fake-оплату.

Sprint 4 должен завершиться работающим сценарием:

```text
создать рассылку → загрузить CSV → подтвердить базу → написать письмо → увидеть стоимость → пройти fake-оплату → увидеть статус «Оплачено»
```

## 3. Принятые рамки для Sprint 4

- Sprint 4 должен встраиваться в текущую модель persistence: EF Core/PostgreSQL по умолчанию, in-memory только как тестовый/dev fallback.
- Нельзя реализовывать оплату как изолированное in-memory-only состояние, если приложение запущено на Postgres provider.
- Если временное отклонение от EF/PostgreSQL неизбежно, оно должно быть явно зафиксировано в `docs/sprints.md` и в этом task-файле перед продолжением следующих спринтов.
- UI остаётся на текущем подходе minimal endpoints + `HtmlRenderer`.
- Реальная платёжная интеграция не подключается.
- Денежный баланс клиента не реализуется и не хранится.
- Оплата считается только для конкретной рассылки.
- Fake-оплата должна остаться пригодной для следующих MVP-спринтов до подключения реального платёжного агента.
- Пользовательский язык интерфейса — русский.
- В UI не показывать внутренние термины `PaymentAttempt`, `PriceSettings`, `Campaign`, `ImportBatch`.
- Расчёт стоимости не должен менять статистику импорта: он только использует уже посчитанные принятые/исключённые адреса.

## 4. Задачи реализации

### T4-01. Добавить доменные модели оплаты и цены

**Цель:** описать оплату рассылки как часть бизнес-состояния, а не как UI-заглушку.

Задачи:

- Добавить модель `Payment` или аналог внутри домена рассылок/оплат.
- Добавить модель `PaymentAttempt`.
- Добавить модель/настройку `PriceSettings`.
- Добавить enum/value object статусов оплаты и попытки оплаты, не полагаясь на произвольные строки в UI.
- Минимальные поля `Payment`:
  - `Id`;
  - `MailingId`;
  - `OwnerEmail`;
  - `AcceptedRecipientsCount`;
  - `ExcludedRecipientsCount`;
  - `PricePerRecipient`;
  - `TotalAmount`;
  - `Currency`, для MVP `RUB`;
  - `Status`;
  - `CreatedAt`;
  - `PaidAt`.
- Минимальные поля `PaymentAttempt`:
  - `Id`;
  - `PaymentId`;
  - `Provider` = `Fake`;
  - `ProviderOperationId`;
  - `Status`;
  - `CreatedAt`;
  - `CompletedAt`;
  - `RawCallback` или безопасный dev-контекст callback без персональных данных.
- Минимальные поля `PriceSettings`:
  - `PricePerRecipient`;
  - `Currency`;
  - `EffectiveFrom`;
  - `IsActive`.
- Проектировать модели так, чтобы они нормально мапились в EF Core без специальных UI-зависимостей.

Acceptance criteria:

- Цена письма не зашита в endpoint.
- `TotalAmount = AcceptedRecipientsCount * PricePerRecipient`.
- В модели нельзя создать оплату с отрицательной ценой или отрицательным количеством адресов.
- Валюта явно задана как `RUB`.
- Модели оплаты не зависят от `Pismolet.Web`.

### T4-02. Добавить статусы и переходы оплаты в модели рассылки

**Цель:** встроить оплату в текущий flow рассылки после сохранения письма.

Задачи:

- Текущий `Mailing` использует русскую строку статуса для отображения. В рамках Sprint 4 нужно добавить стабильную техническую статусную модель: enum/value object или аналогичный тип, пригодный для доменной логики, EF Core и тестов.
- Минимальный набор технических статусов должен покрывать текущий flow и Sprint 4:
  - `draft` / «Черновик»;
  - `recipients_imported` / «Адреса загружены»;
  - `declaration_confirmed` / «База подтверждена»;
  - `message_prepared` / «Письмо подготовлено»;
  - `priced` / «Стоимость рассчитана»;
  - `payment_pending` / «Ожидает оплаты»;
  - `paid` / «Оплачено».
- Русские подписи оставить как UI mapping/вычисляемое представление, а не как единственный источник бизнес-состояния.
- Добавить методы доменной модели, например:
  - `MarkPriced(...)`;
  - `MarkPaymentPending(...)`;
  - `MarkPaid(...)`.
- Запретить расчёт стоимости, если:
  - нет принятых адресов;
  - не подтверждена декларация базы;
  - не сохранён черновик письма.
- Запретить переход в `paid` без успешного `PaymentAttempt`.
- Обновить русские подписи статусов в UI.

Acceptance criteria:

- После сохранения письма следующий шаг ведёт не в заглушку, а на расчёт и оплату.
- Нельзя оплатить рассылку без декларации и письма.
- Статус `paid` выставляется только через сервис оплаты.
- Новые статусы не реализованы только через произвольные русские строки.

### T4-03. Реализовать сервис расчёта стоимости

**Цель:** считать стоимость только по адресам, принятым к отправке.

Задачи:

- Добавить интерфейс `IMailingPricingService` в application/domain-слое, а не в endpoint.
- Добавить команду/метод, например `CalculateMailingPriceCommand`.
- Брать количество принятых адресов из текущей статистики импорта или из списка сохранённых получателей.
- Исключать из оплаты:
  - дубли;
  - невалидные адреса;
  - глобально отписанные адреса;
  - заблокированные адреса, если в текущей модели уже есть или будет добавлен такой признак.
- Возвращать DTO для UI:
  - принято к оплате;
  - исключено всего;
  - разбивка исключений;
  - цена за адрес;
  - итоговая стоимость;
  - валюта.
- Использовать `PriceSettings` через отдельный интерфейс настроек, а не константу в endpoint.

Acceptance criteria:

- Рассылка на 10 принятых адресов при цене 1 RUB показывает итог 10 RUB.
- Дубли, невалидные и глобально отписанные адреса не входят в стоимость.
- Повторный пересчёт без изменения импорта и настроек возвращает тот же результат.

### T4-04. Добавить persistence-репозитории и настройки Sprint 4

**Цель:** встроить оплату в текущую инфраструктуру хранения и не создать параллельное in-memory-only состояние.

Задачи:

- Добавить интерфейсы:
  - `IPaymentRepository`;
  - `IPriceSettingsProvider` или `IPriceSettingsRepository`.
- Разместить контракты в подходящем слое `Pismolet.Application`/`Pismolet.Domain`, чтобы endpoints не зависели от конкретного хранилища.
- Добавить EF Core поддержку в `Pismolet.Infrastructure`:
  - `DbSet`/конфигурации для `Payment`, `PaymentAttempt`, `PriceSettings`;
  - EF-репозитории для платежей и настроек цены;
  - миграцию БД для новых таблиц/полей;
  - seed/dev-настройку цены по умолчанию `1 RUB` за принятый адрес.
- Добавить in-memory/fake реализации только для тестов и dev fallback, если такой provider уже используется в проекте.
- Зарегистрировать сервисы в текущем DI-модуле `AddPismoletWebServices` или инфраструктурном DI-модуле, сохранив текущий выбор provider.
- Не хранить баланс клиента.
- Не создавать реальные платежные документы.

Acceptance criteria:

- В режиме Postgres provider платежи, попытки оплаты и настройки цены сохраняются в БД и не теряются после рестарта приложения.
- In-memory/fake реализации доступны для unit/web-тестов и могут сбрасывать состояние между тестами.
- Все зависимости оплаты подключаются через DI.
- Тесты могут подменять цену через fake/in-memory настройки.

### T4-05. Реализовать fake payment provider

**Цель:** имитировать успешную оплату без внешней платёжной интеграции.

Задачи:

- Добавить интерфейс `IPaymentProvider`.
- Добавить реализацию `FakePaymentProvider`.
- При запуске fake-оплаты создавать `PaymentAttempt` со статусом `Pending` или сразу `Succeeded`, в зависимости от выбранной простоты dev-flow.
- Добавить стабильный `ProviderOperationId`, пригодный для идемпотентности.
- Добавить обработку успешного fake callback/action.
- Повторный callback/action с тем же `ProviderOperationId` не должен создавать вторую оплату и не должен менять сумму.
- Для dev-среза допустима кнопка «Оплатить тестово», ведущая на локальный POST endpoint.

Acceptance criteria:

- Первый успешный fake callback переводит оплату в `paid`.
- Повторный fake callback возвращает успешный результат без дублей.
- У рассылки остаётся один актуальный успешный платёж.

### T4-06. Добавить экран «Проверка и оплата»

**Цель:** заменить заглушку после редактора письма на следующий рабочий шаг.

Задачи:

- Добавить route `GET /mailings/{id}/payment`.
- Добавить route `POST /mailings/{id}/payment/fake-start` или аналог.
- Добавить route `POST /mailings/{id}/payment/fake-success` или совместить старт и успех для простого MVP-flow.
- На странице показать:
  - название рассылки;
  - статус рассылки;
  - принято адресов;
  - исключено всего;
  - разбивку исключений: дубли, невалидные, глобально отписанные, заблокированные;
  - цену за адрес;
  - итоговую стоимость;
  - предупреждение, что это fake-оплата для MVP;
  - кнопку «Оплатить тестово».
- После успешной оплаты показать экран/блок «Оплачено» и следующий шаг «Проверка перед отправкой» как заглушку Sprint 5.
- В карточке рассылки обновить `NextStep`: после письма вести на `/mailings/{id}/payment`.

Acceptance criteria:

- Пользователь после сохранения письма видит кнопку перехода к оплате.
- На странице нет внутренних терминов `PaymentAttempt` и `PriceSettings`.
- После оплаты refresh страницы не запускает новую оплату.

### T4-07. Добавить application service оплаты

**Цель:** не размещать бизнес-правила оплаты в endpoint.

Задачи:

- Добавить интерфейс `IMailingPaymentService`.
- Добавить команды/методы:
  - `GetPaymentReview`;
  - `CreateOrGetPayment`;
  - `StartFakePayment`;
  - `ConfirmFakePaymentSuccess`.
- Проверять:
  - пользователь авторизован;

  - импорт содержит принятые адреса;
  - декларация подтверждена;
  - черновик письма сохранён;

- Писать audit log события:
  - `mailing_price_calculated`;
  - `payment_attempt_created`;
  - `fake_payment_succeeded`.
- Возвращать ошибки на русском языке.

Acceptance criteria:

- Endpoint только читает форму/параметры и вызывает application service.
- Ошибки flow отображаются пользователю понятным русским текстом.
- Audit log содержит id рассылки, сумму, валюту и id попытки без лишних персональных данных.

**Цель:** закрепить бизнес-правила Sprint 4.

Задачи:

- Тест: дубли не оплачиваются.
- Тест: невалидные адреса не оплачиваются.
- Тест: глобально отписанные адреса не оплачиваются.

### T4-09. Добавить интеграционные/web-тесты Sprint 4

**Цель:** проверить полный пользовательский сценарий оплаты в текущем Web-проекте.

Задачи:

- Тест: пользователь проходит flow до `/mailings/{id}/payment`.
- Тест: экран оплаты показывает принято/исключено/стоимость.

- Тест: повторный POST fake-success идемпотентен.

### T4-10. Описать ручные тесты Sprint 4

**Цель:** дать проверочный сценарий для человека и LLM-агента.

Ручные проверки:

1. Зарегистрироваться/войти в dev-среде.
2. Создать рассылку.
3. Загрузить CSV на 10 валидных адресов.
4. Подтвердить базу.
5. Написать plain text письмо.
6. Открыть экран «Проверка и оплата».
7. Проверить, что итог равен `10 * цена за адрес`.
8. Загрузить CSV с дублями и невалидными адресами, проверить, что они не оплачиваются.
9. Пройти fake-оплату.
10. Обновить страницу оплаты и убедиться, что повторное списание не происходит.
11. Вернуться в карточку рассылки и проверить статус «Оплачено».
12. Перезапустить dev-приложение с Postgres provider и убедиться, что оплаченная рассылка и платёжное состояние не потерялись.

Acceptance criteria:

- Ручной сценарий проходит без изменения кода и без внешних интеграций.
- Пользовательские тексты на русском языке.
- В Postgres/dev-сценарии оплата сохраняется после рестарта приложения.

## 5. Что не входит в Sprint 4

- Реальная платёжная интеграция.
- Счета, чеки, интеграция с «Мой налог» и платёжными агентами.
- Денежный баланс клиента.
- Промокоды, тарифные планы, скидки от объёма.
- Ручная модерация и risk-проверка: это Sprint 5.
- Очередь отправки и fake email provider: это Sprint 6.
- Полная переработка всей persistence-модели проекта вне задач, необходимых для оплаты Sprint 4.

## 6. Зависимости и синхронизация

Перед реализацией Sprint 4 желательно проверить закрытие Sprint 3:

- декларация базы сохраняется;
- редактор письма сохраняет `MailingMessageDraft`;
- после письма есть стабильный следующий шаг;
- тесты Sprint 3 проходят.

Документы/разделы, которые могут потребовать синхронизации после реализации:

- `README.md` — обновить текущий срез и добавить route оплаты;
- `docs/sprints.md` — отметить фактические отклонения, если fake-flow или persistence оплаты будут отличаться от плана;
