# Ревью UI личного кабинета

Дата: 2026-06-26

Статус: ревью без правок кода. Админка сознательно не входит в задачу по выравниванию рабочей ширины: она может оставаться широкой.

## Короткий вывод

Личный кабинет уже функционально собран, но визуально ощущается как набор технических экранов, а не простой сервис рассылок для неподготовленного пользователя. Главные причины:

- глобальные стили слишком агрессивно влияют на все `label`, `input`, `button`;
- checkbox-сценарии не имеют отдельного компонента, поэтому текст отрывается от чекбоксов;
- кнопки и формы наследуют одинаковую «большую синюю таблетку» независимо от важности действия;
- подтверждения дублируются между шагами;
- экран отправки показывает слишком много технической статистики;
- ЛК не имеет единой рабочей ширины и устойчивой мобильной сетки;
- часть UI собирается regex-middleware и inline styles, что делает поведение хрупким.

## P0. Чекбоксы и согласия

### Симптомы

- На регистрации чекбокс визуально отделён от текста, текст выглядит как отдельный жирный абзац.
- Появляются непонятные «висящие» маркеры/точки.
- На шаге базы чекбоксы и подписи выглядят как случайные строки, а не как связанные элементы управления.
- На оплате длинные подтверждения превращаются в тяжёлые карточки с жирным текстом.

### Источники

- `HtmlRenderer.AccountForm(...)` формирует согласия регистрации как обычные `label` с вложенным `input type='checkbox'`: `src/Pismolet.Web/Rendering/HtmlRenderer.cs`.
- Глобальный `label { display: grid; font-weight: 700; }` и `input { width: 100%; ... }` применяются и к чекбоксам: `src/Pismolet.Web/wwwroot/app.css`.
- `.confirmation-list label { font-weight: 800; padding: 14px; }` делает каждое подтверждение тяжёлым: `src/Pismolet.Web/wwwroot/payment.css`.

### Что сделать

- Ввести отдельный компонент для чекбоксов: `.checkbox-list`, `.checkbox-row`, `.checkbox-control`, `.checkbox-text`.
- Для checkbox input всегда задавать `width: 18px; height: 18px; flex: 0 0 18px; margin-top: 2px;`.
- Не использовать общий `label` как универсальный layout для всех форм. Поля ввода должны быть `.field`, чекбоксы - `.checkbox-row`.
- Снизить вес текста подтверждений до `400/500`, жирным оставлять только короткий смысловой акцент.
- Для регистрации сделать две компактные строки согласий, без карточек и без больших отступов.

## P0. Дублирующиеся подтверждения

### Симптомы

- Пользователь подтверждает законность базы на шаге адресов, а затем снова видит похожие подтверждения перед оплатой.
- Подтверждения оказываются на страницах, где они не являются основным действием.
- Для новичка это выглядит как юридическая ловушка, а не как понятный wizard.

### Источники

- Шаг адресов содержит `baseLegality` и рекламное согласие: `src/Pismolet.Web/Endpoints/SimplifiedRecipientStepEndpoints.cs`.
- Шаг оплаты снова требует `paymentBaseLegality`, `paymentBaseOwnership`, а для рекламы ещё `advertisingConsent`: `src/Pismolet.Web/Endpoints/PaymentEndpoints.cs`.

### Что сделать

- На шаге адресов оставить юридически значимые подтверждения базы и рекламного согласия.
- На оплате показывать read-only summary: «База подтверждена», «Тип письма: информационное/рекламное», «Согласие на рекламу подтверждено/не требуется».
- На оплате оставить одно короткое финальное подтверждение: «Проверил рассылку и понимаю условия оплаты и запуска».
- Если тип письма рекламный, не дублировать checkbox, а показывать статус ранее сохранённого подтверждения. Если подтверждения нет - отправлять пользователя назад на шаг адресов/типа письма.

## P1. Кнопки непропорционально большие

### Симптомы

- `Найти`, `Добавить`, `Создать аккаунт`, `Оплатить` выглядят одинаково важными.
- На страницах управления адресами кнопки занимают слишком много горизонтального пространства.
- Вторичные действия конкурируют с главным CTA.

### Источники

- Глобальное правило `.btn, .button, button` задаёт всем кнопкам большой padding, жирный шрифт, pill-radius и тень: `src/Pismolet.Web/wwwroot/app.css`.
- На управлении адресами формы используют `.row`, которая рассчитана на две равные колонки, но в неё кладутся input + button + reset: `src/Pismolet.Web/Endpoints/SimplifiedRecipientStepEndpoints.cs`.
- `.full-pay-button { width: 100%; }` делает кнопку оплаты огромной даже там, где рядом уже есть большой блок суммы: `src/Pismolet.Web/wwwroot/payment.css`.

### Что сделать

- Развести варианты кнопок:
  - primary CTA: только одно главное действие на экран;
  - secondary: спокойная кнопка без тени;
  - tertiary/text: сброс, назад, открыть документ;
  - compact: поиск, добавить адрес, обновить.
- Убрать глобальный селектор `button` из общего правила. Стилизовать через классы.
- Для inline-форм сделать сетку `input | compact button | text link`, а не равные колонки.
- На мобильном primary CTA может быть `width: 100%`, на desktop - не должен растягиваться без причины.

## P1. Разная ширина и «пляшущая» сетка ЛК

### Симптомы

- Регистрация узкая, wizard шире, сообщение и оплата работают в разных пропорциях, история и отправка снова выглядят иначе.
- На соседних шагах меняется ощущение рабочего пространства.
- Админские full-width хаки живут в payment CSS, что смешивает разные области UI.

### Источники

- `.page` имеет `max-width: 1120px`, `.form-card` - `max-width: 560px`, message/payment/send задают свои grids.
- `payment.css` содержит правила для `.admin-shell`, хотя это не платежный слой.

### Что сделать

- Ввести общий контейнер ЛК: `.cabinet-page` или `.workspace`.
- Для всех экранов ЛК использовать один базовый max-width, например `960px`.
- Для сложных экранов вроде редактора письма можно расширять до `1040px`, но это должно быть явно и редко.
- Админскую ширину вынести из `payment.css` в admin-specific CSS и не смешивать с ЛК.

## P1. Шаг адресов: слепленные и невыровненные элементы

### Симптомы

- Поиск, сброс и добавление адреса выглядят как разрозненные элементы.
- Кнопки «Найти» и «Добавить» визуально тяжелее самих полей.
- Подтверждение базы помещено рядом с управлением адресами и выглядит вторично, хотя юридически это важный шаг.

### Источники

- В `ManagementPage(...)` используются inline styles и `.row` для разных задач: `src/Pismolet.Web/Endpoints/SimplifiedRecipientStepEndpoints.cs`.
- `CompactRecipientDeclarationUiMiddlewareExtensions` переписывает HTML через regex и добавляет inline styles.

### Что сделать

- Убрать regex-middleware для декларации базы и рендерить финальную разметку напрямую.
- Разделить экран на блоки:
  - краткая сводка импорта;
  - статус подтверждения базы;
  - управление списком адресов;
  - CTA «Перейти к письму».
- Для поиска использовать компактную строку: поле, кнопка `Найти`, ссылка `Сбросить`.
- Для добавления адреса использовать такую же компактную строку, не огромную синюю кнопку на пол-экрана.

## P1. Оплата: слишком тяжёлый экран

### Симптомы

- Слишком много карточек и подтверждений перед оплатой.
- Большая сумма визуально спорит с подтверждениями.
- Текст подтверждений слишком юридический и слишком жирный.
- Провайдер оплаты виден пользователю как `Robokassa`, хотя для новичка это техническая деталь.

### Источники

- `PaymentPage(...)` строит 5 stat cards, блок подтверждений и большую карточку суммы: `src/Pismolet.Web/Endpoints/PaymentEndpoints.cs`.
- `.pay-card .sum` использует очень крупный размер и отрицательный letter-spacing: `src/Pismolet.Web/wwwroot/payment.css`.

### Что сделать

- Кнопка для пользователя: «Оплатить 13 ₽», без имени провайдера.
- Провайдера показывать мелко в пояснении или на следующем переходе.
- Свести подтверждения к одному финальному checkbox.
- Сумму сделать крупной, но не hero-scale. Достаточно 40-48px на desktop.
- Сводку импорта оставить, но сократить до 3 ключевых чисел: принято, исключено, к оплате.

## P1. Экран запуска перегружен технической информацией

### Симптомы

- ЛК показывает delivery statuses, provider states, suppression, opens, clicks, replies, подробные таблицы по email.
- Для «сервиса рассылок для чайников» это похоже на админскую диагностику.
- `details` с доставкой и кликами открыты по умолчанию.

### Источники

- `SendPage(...)` выводит много технических таблиц и подробных метрик прямо в основной пользовательский экран: `src/Pismolet.Web/Endpoints/SendEndpoints.cs`.

### Что сделать

- Основной экран для клиента:
  - статус: «Готово к запуску», «Отправляется», «Отправлено», «Приостановлено»;
  - всего писем, отправлено, не удалось, ответов;
  - одна главная кнопка.
- Детали доставки, кликов и технический отчёт спрятать в collapsed-блок «Подробный отчёт».
- `Dev-сводка событий` не должна отображаться обычному пользователю; оставить только в dev/admin режиме.
- Таблицы по получателям показывать после отправки или по явному запросу пользователя.

## P1. Превью письма показывает лишнюю технику

### Симптомы

- В превью видны `/unsubscribe/example-token` и служебный идентификатор.
- Технический служебный блок занимает внимание до того, как пользователь написал письмо.
- Это повышает когнитивную нагрузку на шаге, где человек должен просто написать текст.

### Источники

- `SimplifiedMessageStepEndpoints.MessageForm(...)` всегда показывает служебный блок, отписку и идентификатор в preview.

### Что сделать

- В основном preview показывать только то, что пользователь редактирует: отправитель, тема, текст.
- Служебный блок заменить короткой строкой: «Письмолёт автоматически добавит причину получения, отписку и служебный номер».
- Полный служебный блок вынести в collapsed preview или ссылку «Показать служебный блок».

## P2. Жирность и визуальный шум

### Симптомы

- Слишком много элементов с `font-weight: 800/900`: labels, buttons, dt/dd, stats, confirmations.
- Пользователь не понимает, что действительно важно.

### Источники

- `app.css`, `payment.css`, `cabinet.css` активно используют 800/900.

### Что сделать

- Базовый текст: 400.
- Labels: 500/600.
- Кнопки: 600/700.
- Метрики: число 700, подпись 400/500.
- Юридический текст и подсказки не делать жирными целиком.

## P2. Мобильная адаптация ЛК

### Симптомы

- Сетки местами переходят в одну колонку, но кнопки остаются огромными.
- Таблицы без единообразных wrappers могут ломать ширину.
- Header на мобильном становится колонкой, но рабочие формы не получают отдельной мобильной иерархии.

### Что сделать

- Для ЛК определить mobile rules на уровне workspace:
  - `padding: 16px`;
  - карточки без лишней тени;
  - primary CTA full-width только внизу блока;
  - inline controls stacked, но с компактными кнопками;
  - таблицы всегда в `.table-wrap { overflow-x:auto; }`.
- Проверять минимум 390px и 768px viewport.

## Рекомендуемый порядок работ

1. Починить checkbox-компонент и регистрацию. Это самый заметный визуальный дефект.
2. Развести типы кнопок и убрать глобальное влияние `button`.
3. Ввести общий layout для ЛК и стабилизировать ширину wizard-экранов.
4. Переписать шаг адресов без regex-middleware и inline styles.
5. Упростить оплату: меньше подтверждений, меньше жирного текста, меньше provider-техники.
6. Упростить экран отправки для клиента, технические таблицы спрятать.
7. Пройти mobile QA по регистрации, адресам, письму, оплате и запуску.

## План устранения по спринтам

### UI Sprint 1. Формы, чекбоксы и согласия

Статус: принят после проверки и ревью.

Цель: убрать самый заметный визуальный дефект, из-за которого согласия выглядят оторванными от чекбоксов.

Объём:

- ввести отдельный checkbox-компонент для регистрации, декларации базы и оплаты;
- убрать влияние глобального `label` и `input { width: 100%; }` на `input[type="checkbox"]`;
- привести регистрацию к двум компактным строкам согласий;
- снизить жирность текста согласий;
- добавить/обновить renderer- и endpoint-тесты на наличие нужных ссылок и обязательных checkbox-полей.

Готово, когда:

- чекбокс и текст визуально связаны на desktop и mobile;
- на регистрации нет висящих маркеров и случайных пустых строк;
- юридические ссылки на документы сохраняют корректный `returnUrl`.

### UI Sprint 2. Кнопки и действия

Статус: ожидает проверки и ревью.

Цель: сделать иерархию действий понятной: одно главное действие на экран, вторичные действия не конкурируют с ним.

Объём:

- развести классы кнопок: primary, secondary, tertiary/text, compact;
- убрать общий стиль с селектора `button`, оставить оформление через явные классы;
- привести `Найти`, `Сбросить`, `Добавить`, `Назад` к компактным вариантам;
- оставить большие CTA только для перехода к следующему шагу wizard-а или оплаты;
- проверить профильное меню и админку на регрессии после снятия глобального `button`.

Готово, когда:

- вторичные действия не выглядят как главная синяя кнопка;
- inline-формы не растягивают кнопки на половину экрана;
- существующие кнопки меню и админки не потеряли кликабельность и читаемость.

### UI Sprint 3. Единая рабочая ширина ЛК и мобильная база

Статус: не начат.

Цель: стабилизировать рабочее пространство ЛК без ограничения админки.

Объём:

- ввести общий контейнер ЛК: `.cabinet-page` или `.workspace`;
- задать единый max-width для основных экранов ЛК;
- вынести админские full-width правила из `payment.css` в admin-specific стили;
- унифицировать отступы панелей, grid-gap и mobile padding;
- добавить table wrappers там, где таблицы могут ломать ширину на телефоне.

Готово, когда:

- wizard-экраны не прыгают по ширине между шагами;
- админка остаётся широкой;
- на 390px нет горизонтального скролла, кроме таблиц внутри явного scroll-wrapper.

### UI Sprint 4. Шаг адресов

Статус: не начат.

Цель: сделать загрузку и управление адресами понятным шагом wizard-а, без regex-middleware и inline styles.

Объём:

- убрать `CompactRecipientDeclarationUiMiddlewareExtensions` из пользовательского сценария;
- рендерить финальную разметку шага адресов напрямую;
- разделить экран на сводку импорта, статус базы, управление адресами и CTA;
- привести поиск, сброс и ручное добавление адреса к компактным aligned controls;
- проверить сценарии: файл, ручная вставка, поиск, сброс, добавление, удаление.

Готово, когда:

- элементы управления адресами выровнены и не слеплены;
- декларация базы не выглядит случайной вставкой;
- тесты покрывают основной happy path и ошибки ручного ввода.

### UI Sprint 5. Оплата и подтверждения

Статус: не начат.

Цель: убрать дублирующиеся подтверждения и сделать оплату похожей на простой платёжный шаг, а не на юридическую стену.

Объём:

- оставить юридические подтверждения базы на шаге адресов;
- на оплате заменить повторные checkbox-подтверждения read-only summary;
- оставить один финальный checkbox про проверку рассылки и условия оплаты/запуска;
- заменить текст кнопки на пользовательский вариант вроде `Оплатить 13 ₽`;
- уменьшить визуальный вес суммы и блока подтверждений;
- проверить соответствие legal-страницам и сценарию фейковой Робокассы.

Готово, когда:

- на оплате нет повторных подтверждений базы;
- рекламная рассылка без рекламного согласия не проходит дальше;
- пользователь видит сумму, состав расчёта и одно понятное финальное действие.

### UI Sprint 6. Письмо и превью

Статус: не начат.

Цель: снизить технический шум на шаге написания письма.

Объём:

- в основном preview показывать только отправителя, тему и текст;
- заменить техническую отписку и служебный идентификатор коротким пояснением;
- полный служебный блок вынести в collapsed preview;
- проверить, что реальная отправка всё ещё получает обязательный unsubscribe/footer;
- обновить тесты preview под новый пользовательский контракт.

Готово, когда:

- пользователь пишет письмо без лишней технической информации перед глазами;
- обязательные служебные элементы не исчезают из реального письма;
- тесты различают preview и production rendering.

### UI Sprint 7. Запуск, история и отчётность для новичка

Статус: не начат.

Цель: оставить в ЛК простую картину отправки, а техническую диагностику спрятать.

Объём:

- сделать основной экран запуска вокруг статуса, прогресса и одного CTA;
- сократить верхние метрики до пользовательских: всего, отправлено, не удалось, ответов;
- спрятать delivery/open/click/reply таблицы в collapsed-блок `Подробный отчёт`;
- показывать `Dev-сводку событий` только в dev/admin режиме;
- упростить историю рассылок до понятных статусов и следующих действий.

Готово, когда:

- обычный пользователь не видит provider/raw/dev-детали в основном сценарии;
- технический отчёт доступен, но не занимает первый экран;
- история и экран запуска говорят одним языком статусов.

### UI Sprint 8. Финальный visual QA и регрессия

Статус: не начат.

Цель: пройти весь пользовательский сценарий и закрыть оставшиеся визуальные шероховатости.

Объём:

- проверить регистрацию, создание рассылки, адреса, письмо, оплату, запуск, историю;
- проверить desktop, tablet и mobile viewport;
- убрать остаточные `font-weight: 800/900`, где они не несут смысловой нагрузки;
- выровнять пустые состояния, notices, hints и таблицы;
- зафиксировать screenshots или краткий QA-отчёт в docs.

Готово, когда:

- все acceptance criteria ниже выполнены;
- build/test gate зелёный;
- в docs отмечено, какие UI-долги закрыты и что осталось за рамками.

## Acceptance criteria

- Чекбокс и его текст всегда находятся на одной строке начала, без оторванных маркеров.
- На каждом экране есть только один визуально главный CTA.
- Вторичные действия не выглядят как главная кнопка.
- Рабочая область ЛК не прыгает по ширине между шагами wizard-а.
- Юридические подтверждения не повторяются без причины.
- Обычный пользователь не видит dev/provider/raw-техническую информацию в основном сценарии.
- На мобильном 390px нет горизонтального скролла, кроме таблиц внутри явных wrappers.
