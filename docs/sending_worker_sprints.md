# План спринтов: автоматическая отправка рассылок через отдельный worker

Дата: 2026-06-26  
Направление: Техника  
Статус: план реализации

## 1. Контекст и принятое решение

Текущая проблема: при отправке рассылки на заметное количество адресатов пользователь вынужден вручную нажимать кнопку «Продолжить», чтобы проталкивать следующие порции отправки. Такой сценарий неприемлем для продукта: пользователь должен один раз запустить рассылку и иметь возможность закрыть страницу.

Принятое техническое решение: не использовать in-process background worker внутри web-приложения как основной механизм. Для MVP выбран отдельный worker-процесс под systemd:

- `pismolet.service` — web-приложение, личный кабинет, создание рассылки, оплата, модерация, запуск;
- `pismolet-worker.service` — отдельный процесс, который автоматически обрабатывает рассылки в статусе `Sending` и отправляет письма батчами.

Очередь для MVP: PostgreSQL / текущая база данных. Отдельные Redis/RabbitMQ/Hangfire не вводятся на этом этапе.

## 2. Целевое поведение

Пользовательский сценарий должен быть таким:

1. Пользователь нажимает «Запустить отправку» один раз.
2. Web-приложение переводит рассылку в статус `Sending`.
3. Пользователь видит экран прогресса: всего писем, отправлено, ошибок, осталось, последнее событие.
4. Пользователь может закрыть страницу.
5. `pismolet-worker` продолжает отправку самостоятельно.
6. После обработки всех получателей рассылка получает финальный статус: `Sent`, `Failed` или иной согласованный статус для частичной отправки.
7. Кнопка «Продолжить» скрыта от обычного пользователя и остаётся только как аварийное/админское действие либо удаляется из пользовательского сценария.

## 3. Архитектурные принципы

- Отправка не должна зависеть от длительного HTTP-запроса.
- Рестарт web-приложения не должен останавливать уже запущенную рассылку.
- Worker должен быть идемпотентным: повторный запуск не должен отправлять одно и то же письмо одному получателю дважды.
- Источником правды о прогрессе остаются записи в БД: получатели, события отправки, статусы рассылки.
- Все ошибки отправки фиксируются в отчёте и не должны валить весь процесс, если можно продолжить отправку по остальным получателям.
- Лимиты отправки должны применяться на стороне worker, а не через ручные действия пользователя.

## 4. Спринт 1 — Подготовка модели автоматической отправки

Цель: описать и стабилизировать состояние рассылки и получателей так, чтобы worker мог безопасно продолжать отправку после рестарта.

### Задачи

- Проанализировать текущие статусы `MailingStatus` и определить, достаточно ли `Sending`, `Sent`, `Failed`, `Paused` для worker-сценария.
- Определить, как worker понимает, какие получатели уже обработаны.
- Проверить уникальность события отправки на пару `mailingId + recipientEmail` или иной ключ получателя.
- Добавить недостающие индексы/ограничения БД, если они нужны для идемпотентности.
- Зафиксировать правило: повторная обработка батча не должна создавать повторную фактическую отправку письма.
- Описать минимальные настройки worker:
  - размер батча;
  - пауза между циклами;
  - дневной лимит пользователя;
  - глобальный лимит отправки;
  - возможность выключить worker через конфиг.

### Автотесты

- Повторный вызов обработки одного и того же батча не дублирует отправку уже обработанным получателям.
- Рассылка без доступных получателей корректно завершает отправку.
- Ошибка по одному получателю не блокирует обработку остальных.
- Статус рассылки корректно меняется при завершении всех получателей.

### Ручные проверки

- Создать рассылку на 5 адресов и убедиться, что прогресс считается по событиям отправки.
- Прервать обработку после части получателей и убедиться, что повторный запуск продолжает с оставшихся.

## 5. Спринт 2 — Создание отдельного worker-приложения

Цель: добавить отдельный исполняемый проект `Pismolet.Worker`, который можно запускать как самостоятельный systemd-сервис.

### Задачи

- Добавить проект `src/Pismolet.Worker`.
- Подключить общие application/infrastructure сервисы и конфигурацию.
- Реализовать основной цикл worker:
  - загрузить настройки;
  - найти рассылки в статусе `Sending`;
  - выбрать следующий батч получателей;
  - вызвать существующий сервис отправки;
  - записать прогресс;
  - уснуть на заданный интервал;
  - повторить.
- Добавить graceful shutdown по `CancellationToken`.
- Разнести логи web и worker по понятным категориям.
- Подготовить systemd unit `pismolet-worker.service` в документации или deploy-шаблонах.

### Автотесты

- Worker обрабатывает рассылку в статусе `Sending` без HTTP-запроса.
- Worker не берёт рассылки в статусах `Draft`, `PaymentPending`, `ReviewRequired`, `Rejected`, `Sent`, `Failed`, `Paused`.
- Worker корректно останавливается по cancellation token.

### Ручные проверки

- Запустить worker локально/на сервере командой `dotnet run` или опубликованным бинарником.
- Убедиться, что после запуска рассылки web-страницу можно закрыть, а отправка продолжается.
- Перезапустить worker во время отправки и убедиться, что отправка продолжается без дублей.

## 6. Спринт 3 — Изменение пользовательского сценария запуска

Цель: убрать зависимость пользователя от кнопки «Продолжить».

### Задачи

- Изменить обработчик «Запустить отправку»: он должен только переводить рассылку в `Sending` и показывать экран прогресса.
- Скрыть кнопку «Продолжить» для обычного пользователя.
- Оставить аварийное действие ручной обработки батча только для администратора или временно удалить из UI.
- Добавить текст на экран отправки:
  - «Рассылка запущена»;
  - «Можно закрыть страницу — отправка продолжится автоматически»;
  - «Отправка идёт постепенно, чтобы снизить риск блокировок и ошибок доставки».
- Добавить автообновление страницы прогресса или отдельный lightweight endpoint для прогресса.
- Уточнить статусы на dashboard: `Отправляется`, `Отправлено`, `Есть ошибки`, `На паузе`.

### Автотесты

- После нажатия «Запустить отправку» пользователь получает экран прогресса без необходимости нажимать «Продолжить».
- В HTML обычного пользователя нет кнопки ручного продолжения.
- Администратор видит аварийное действие, если оно оставлено.
- Dashboard показывает статус запущенной рассылки.

### Ручные проверки

- Запустить рассылку на 100 адресов.
- Закрыть страницу сразу после запуска.
- Через несколько минут открыть рассылку и увидеть увеличившийся прогресс.
- Проверить, что пользователю не нужно выполнять дополнительные действия.

## 7. Спринт 4 — Лимиты, паузы и защита от перегрузки

Цель: сделать отправку управляемой и безопасной для домена/SMTP.

### Задачи

- Ввести настройки worker:
  - `Worker:BatchSize`;
  - `Worker:PollIntervalSeconds`;
  - `Worker:DelayBetweenMessagesMs`;
  - `Worker:MaxMessagesPerUserPerDay`;
  - `Worker:MaxMessagesGlobalPerHour`.
- Начальные значения для MVP:
  - батч: 5–10 писем;
  - пауза между циклами: 5–15 секунд;
  - пауза между отдельными письмами: 0.5–3 секунды, если требуется SMTP-провайдером.
- Подготовить основу для будущих лимитов по доменам получателей: `gmail.com`, `yandex.ru`, `mail.ru` и т. п.
- Добавить режим `Paused`, если лимит исчерпан или отправка временно остановлена администратором.
- Сделать ошибки SMTP нефатальными для всей рассылки, если это ошибка конкретного адресата.

### Автотесты

- Worker соблюдает размер батча.
- Worker не отправляет сверх дневного лимита пользователя.
- Worker корректно пропускает/фиксирует ошибочного получателя и продолжает остальных.
- Рассылка переводится в `Paused` или остаётся `Sending` с понятным состоянием при достижении лимита.

### Ручные проверки

- Запустить рассылку на 100 адресов и проверить темп отправки.
- Временно снизить лимит и убедиться, что worker не превышает его.
- Смоделировать ошибку SMTP и проверить отчёт.

## 8. Спринт 5 — Деплой и эксплуатация worker

Цель: сделать worker частью production-деплоя.

### Задачи

- Обновить deploy-скрипт, чтобы он публиковал web и worker.
- Добавить `pismolet-worker.service`.
- Добавить команды:
  - `systemctl status pismolet-worker`;
  - `journalctl -u pismolet-worker`;
  - `systemctl restart pismolet-worker`.
- Описать порядок деплоя:
  - остановить web/worker при необходимости;
  - выполнить publish;
  - применить миграции/проверку схемы;
  - запустить web;
  - запустить worker.
- Настроить автозапуск worker после перезагрузки сервера.
- Разнести диагностику: web 500/502 отдельно, отправка писем отдельно.

### Автотесты

- Минимальная проверка конфигурации worker стартует в тестовом окружении.
- Ошибка отсутствующей обязательной настройки даёт понятное сообщение.

### Ручные проверки

- После reboot сервера оба сервиса поднялись.
- `pismolet.service` отвечает `/health`.
- `pismolet-worker.service` работает и пишет heartbeat/цикл обработки.
- Рассылка продолжает отправку после `systemctl restart pismolet-worker`.

## 9. Спринт 6 — Отчётность и наблюдаемость

Цель: чтобы пользователь и администратор понимали, что происходит с отправкой.

### Задачи

- Добавить явный прогресс:
  - всего получателей;
  - отправлено;
  - ошибок;
  - ожидает отправки;
  - последнее событие;
  - время старта;
  - время завершения.
- Добавить worker-метрики в логах:
  - сколько рассылок найдено;
  - сколько писем отправлено за цикл;
  - сколько ошибок;
  - сколько времени занял цикл.
- Добавить админский экран/блок активных рассылок, если потребуется.
- Проверить Excel-отчёт после автоматической отправки.

### Автотесты

- Прогресс корректно считается после частичной отправки.
- Финальный отчёт содержит доставленные и ошибочные адреса.
- Dashboard не показывает внутренние технические поля пользователю.

### Ручные проверки

- Запустить рассылку на 100 адресов и наблюдать прогресс.
- Скачать отчёт после завершения.
- Проверить логи worker по одной рассылке.

## 10. Спринт 7 — Улучшения после MVP

Этот спринт не обязателен для первого production-MVP, но нужен для развития.

### Возможные задачи

## 11. Критерии готовности всего блока

Блок считается готовым, когда выполнены условия:

- пользователь запускает рассылку одним действием;
- для рассылки на 100 адресов не требуется ручная кнопка «Продолжить»;
- страницу можно закрыть без остановки отправки;
- worker продолжает отправку после перезапуска;
- повторный запуск worker не создаёт дублей;
- пользователь видит понятный прогресс;
- администратор видит ошибки и может диагностировать отправку;
- деплой включает оба сервиса: `pismolet` и `pismolet-worker`.

## 12. Открытые вопросы

- Какими должны быть стартовые лимиты для production: батч, пауза, лимит в час, лимит в день.
- Нужно ли сразу делать доменные лимиты для популярных почтовиков или отложить до первых реальных метрик.
- Нужно ли показывать пользователю ETA или достаточно текущего прогресса.
