# Production smoke reply alias

Дата: 2026-06-28

Статус: подготовка к спринту 3.8g. Спринт 3.8f после проверки на сервере зелёный.

## Цель

Проверить полный пользовательский сценарий новой модели ответов:

1. исходящее письмо уходит с коротким reply alias;
2. обычный ответ получателя попадает в Postfix и spool;
3. приложение сопоставляет ответ по `In-Reply-To` / `References` и outbound message mapping;
4. ответ пересылается владельцу рассылки;
5. fallback для известного alias не теряет письмо;
6. неизвестный alias не пересылается клиентам.

## Предварительные условия

- DNS для `reply.pismolet.ru` указывает на production-сервер.
- Postfix принимает почту на 25 порту и маршрутизирует домен `reply.pismolet.ru` в spool Письмолёта.
- `InboundReplies__Enabled=true`.
- Приложение задеплоено с миграцией reply alias и outbound message mapping.
- В production БД есть таблицы для `client_reply_aliases` и `outbound_reply_message_mappings`.

## Проверка 1. Исходящее письмо

1. Создать тестовую рассылку от реального клиента на внешний адрес.
2. Отправить рассылку.
3. В полученном письме проверить заголовки:
   - `Reply-To` должен быть вида `<alias>@reply.pismolet.ru`;
   - в `Reply-To` не должно быть длинного token;
   - `Message-Id` должен быть стабильным служебным id Письмолёта.
4. В БД проверить, что для письма создан outbound message mapping.

## Проверка 2. Обычный reply

1. Нажать обычную кнопку Reply в почтовом клиенте получателя.
2. Отправить короткий текст ответа.
3. Проверить логи приложения:
   - spool reader обработал `.eml`;
   - статус не `unmatched`;
   - reply event поставлен в очередь пересылки;
   - forwarding завершился успешно.
4. Проверить `/admin/replies`:
   - виден alias;
   - flow — `matched`;
   - указана рассылка;
   - указан получатель;
   - raw MIME, тело ответа и reply token не отображаются.
5. Проверить почту владельца рассылки: ответ должен прийти как пересланное письмо.

## Проверка 3. Known alias fallback

1. Отправить письмо напрямую на известный alias клиента: `<alias>@reply.pismolet.ru`.
2. Не добавлять валидный `In-Reply-To` / `References`.
3. Ожидаемое поведение:
   - reply event создаётся;
   - flow в admin diagnostics — `known-alias fallback`;
   - письмо пересылается владельцу alias;
   - в теме или теле пересылки есть пометка, что ответ не сопоставлен с конкретной рассылкой.

## Проверка 4. Unknown alias

1. Отправить письмо на несуществующий alias: `unknown-smoke-<date>@reply.pismolet.ru`.
2. Ожидаемое поведение:
   - reply event создаётся как `unmatched`;
   - в admin diagnostics видна причина `reply_alias_unknown`;
   - письмо не пересылается ни одному клиенту.

## Проверка 5. Auto-reply

1. Подложить или отправить письмо с признаками auto-reply.
2. Ожидаемое поведение:
   - статус `ignored_auto_reply`;
   - пересылка клиенту не выполняется.

## Что зафиксировать после smoke

- commit SHA задеплоенной версии;
- время отправки тестовой рассылки;
- внешний адрес получателя;
- видимый `Reply-To` без раскрытия приватных данных;
- результат matched reply;
- результат known-alias fallback;
- результат unknown alias;
- ссылки на admin diagnostics без raw MIME и тела ответа.
