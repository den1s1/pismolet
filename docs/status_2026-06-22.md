## Проверено в production

### Open tracking

Проверено на реальных письмах:

- в письма вставляется tracking pixel;
- endpoint `/t/open/{token}.gif` отдаёт 1x1 GIF;
- `FirstOpenedAt`, `LastOpenedAt`, `OpenCount` обновляются;
- в UI используются формулировки `Открыто`, `Открытия`, не `Прочитано`.

### Click tracking

Проверено на реальном переходе:

- обычные `http/https` ссылки в письме переписываются через `/t/click/{token}`;
- видимый текст ссылки остаётся исходным URL;
- unsubscribe, `/t/open/`, `/t/click/`, `mailto:`, `tel:`, якоря и относительные ссылки не трогаются;
- переходы сохраняются в `tracked_links` и `click_events`;
- UI показывает клики и кликнувших получателей.

### Postfix queue id

Проверено на реальных письмах:

- `ProviderMessageId` теперь содержит Postfix queue id, например `BCC9083ED5`, `DD405840E9`, `CD71284177`;
- эти же id находятся в `/var/log/mail.log`;
- строки `postfix/smtp ... status=sent` сопоставляются с `send_events`.

### Manual Postfix log reader

Проверено вручную из админки:

- первый запуск инициализирует cursor в конец файла;
- последующие запуски читают только новые строки;
- служебные строки `smtpd`, `cleanup`, `qmgr`, `removed` игнорируются;
- строки `postfix/smtp` распознаются;
- событие сохраняется в `postfix_delivery_events`;
- соответствующий `send_event` обновляется.

Пример подтверждённого результата:

```text
CD71284177 | den1s@mail.ru | Sent | Delivered | dsn=2.0.0
```

И в `send_events`:

```text
den1s@mail.ru | CD71284177 | Delivered | sent; dsn=2.0.0; 250 OK ...
```

### Automatic Postfix log reader

Проверено в production:

- hosted service читает `/var/log/mail.log` автоматически;
- интервал задаётся через админку;
- установлен интервал 60 секунд;
- настройки хранятся в `/var/lib/pismolet/postfix-delivery-settings.json`;
- ручной запуск в админке сохранён.

Проверенные результаты:

```text
F2B33842DB -> Delivered
CD71284177 -> Delivered
6853E83ED5 -> SoftBounce
```

Для `pinya77@gmail.com` получен корректный `SoftBounce`:

```text
deferred; dsn=4.2.2; recipient's inbox is out of storage space
```

## HardBounce suppression - начато

Начат блок `5.7.1 HardBounce suppression`.

Сделано:

- если Postfix даёт `bounced` или `expired`, событие мапится в `HardBounce`;
- если событие найдено по `ProviderMessageId` в `send_events`, адрес добавляется в `client_suppressions`;
- suppression привязан к владельцу рассылки через `sendEvent.OwnerEmail`;
- `SoftBounce` не подавляется;
- `Delivered` не подавляется;
- в результат ingestion добавлен счётчик `ClientSuppressions`;
- добавлены тесты на `bounced -> HardBounce -> suppression` и на то, что `deferred -> SoftBounce` не создаёт suppression.

Последний фикс:

```text
b2d1489 Fix empty Postfix reader ingestion results
```

Причина фикса: после добавления `ClientSuppressions` в `PostfixDeliveryLogIngestionResult` в `PostfixDeliveryLogReaderService` оставались старые вызовы конструктора без шестого аргумента.

### 2. Проверить HardBounce suppression в production

Нужно отправить письмо на адрес, который гарантированно даст permanent bounce, например несуществующий ящик на домене, где SMTP отвечает `550 user unknown`.

После этого проверить:

```bash
cd /opt/pismolet

CONN="$(sudo grep '^ConnectionStrings__PismoletDb=' /etc/pismolet/pismolet.env | cut -d= -f2-)"
PGHOST="$(printf '%s' "$CONN" | tr ';' '\n' | awk -F= '$1=="Host"{print $2}')"
PGPORT="$(printf '%s' "$CONN" | tr ';' '\n' | awk -F= '$1=="Port"{print $2}')"
PGDATABASE="$(printf '%s' "$CONN" | tr ';' '\n' | awk -F= '$1=="Database"{print $2}')"
PGUSER="$(printf '%s' "$CONN" | tr ';' '\n' | awk -F= '$1=="Username"{print $2}')"
PGPASSWORD="$(printf '%s' "$CONN" | tr ';' '\n' | awk -F= '$1=="Password"{print $2}')"
export PGPASSWORD

psql -h "$PGHOST" -p "$PGPORT" -U "$PGUSER" -d "$PGDATABASE" -P pager=off -c "
SELECT
  \"RecipientEmail\",
  \"ProviderMessageId\",
  \"DeliveryStatus\",
  \"LastDeliverySummary\",
  \"CreatedAt\"
FROM send_events
ORDER BY \"CreatedAt\" DESC
LIMIT 15;
"

psql -h "$PGHOST" -p "$PGPORT" -U "$PGUSER" -d "$PGDATABASE" -P pager=off -c "
SELECT
  \"ClientId\",
  \"EmailNormalized\",
  \"Reason\",
  \"SourceMailingId\",
  \"SourceProviderMessageId\",
  \"CreatedAt\",
  \"LastSeenAt\"
FROM client_suppressions
ORDER BY \"LastSeenAt\" DESC
LIMIT 15;
"
```

Ожидаемый результат:

```text
DeliveryStatus = HardBounce
client_suppressions.EmailNormalized = адрес с hard bounce
Reason = HardBounce
```
