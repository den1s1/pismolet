# План спринтов: переход Письмолёта на собственную SMTP-отправку с VDS

## Цель

Перевести Письмолёт с отправки через `smtp.timeweb.ru` на собственную исходящую почтовую инфраструктуру, чтобы убрать зависимость от фильтра `smtp-spam.timeweb.ru`, получить контроль над PTR/rDNS, HELO/EHLO, SPF, DKIM, DMARC и начать управляемо наращивать доставляемость.

Базовая схема после уточнения архитектуры:

```text
Письмолёт на app.pismolet.ru -> SMTP-сервер mail.pismolet.ru -> почтовые серверы получателей
```

Роли доменов:

```text
app.pismolet.ru  - веб-приложение и личный кабинет
mail.pismolet.ru - отдельный исходящий SMTP-сервер, HELO/EHLO и PTR/rDNS
pismolet.ru      - основной домен отправителя, DKIM/DMARC identity и брендовый домен
```

## Контекст текущего решения

Сейчас отправка через `smtp.timeweb.ru` технически работает, SPF/DKIM/DMARC проходят, но исходящий фильтр Timeweb добавляет письмам заголовки вида:

```text
X-Spam-Status: Yes
X-Spam-Score: 6.4
DBL_SPAM(6.50)
```

После этого Яндекс кладёт письма в спам. При этом Gmail и Mail.ru могут принимать письма во входящие, но сама метка от Timeweb остаётся в заголовках.

Вариант А - продолжать использовать обычный SMTP Timeweb - не подходит как основа продукта, потому что лимит почтового кластера мал для сервиса рассылок и мы зависим от их антиспам-фильтра.

Вариант Б - собственная исходящая отправка с VDS - выбран как следующий технический путь для MVP.

После ответа поддержки Timeweb дополнительно зафиксировано:

1. Для текущей локации Новосибирск PTR/rDNS для IP сервера, по ответу поддержки, технически недоступен.
2. Для собственного SMTP без корректного PTR/rDNS стартовать нельзя.
3. Основной рабочий сценарий - отдельный mail-сервер в локации, где Timeweb сможет выставить PTR на `mail.pismolet.ru`.
4. Веб-приложение можно оставить на текущем сервере, если отдельный mail-сервер сможет принимать SMTP от приложения по внутреннему или защищённому каналу.
5. Перед созданием mail-сервера ждём подтверждение Timeweb по московской локации, PTR для IPv4/IPv6 и исходящему порту 25.

## Целевая почтовая схема

Нормальная схема для собственного SMTP:

```text
mail.pismolet.ru A    <IPv4 mail-сервера>
mail.pismolet.ru AAAA <IPv6 mail-сервера, если используем IPv6>
PTR <IPv4> -> mail.pismolet.ru
PTR <IPv6> -> mail.pismolet.ru
HELO/EHLO -> mail.pismolet.ru
SPF pismolet.ru -> разрешает IP mail-сервера
DKIM -> подписывает письма от имени pismolet.ru
DMARC -> действует для pismolet.ru
```

PTR не ставим на голый `pismolet.ru`, чтобы не смешивать сайт, домен бренда и почтовый узел. Почтовый узел должен называться `mail.pismolet.ru`.

## Спринт 0. Подготовка и архитектурное решение

### Цель

Зафиксировать схему отправки и подготовить DNS/серверные предпосылки.

### Задачи

1. Дождаться ответа Timeweb по вопросам:
   - можно ли в московской локации выставить PTR/rDNS на `mail.pismolet.ru`;
   - можно ли выставить PTR для IPv4 и IPv6;
   - будет ли открыт исходящий порт 25;
   - можно ли оставить приложение на текущем сервере, а в Москве создать отдельный VPS только под почту.
2. Выбрать схему размещения:
   - предпочтительно: отдельный VPS под почту в Москве;
   - запасной вариант: перенос всего сервера приложения в Москву.
3. Выбрать технический hostname для исходящей почты:
   - основной вариант: `mail.pismolet.ru`.
4. Подготовить DNS-записи после получения нового IP:
   - `A mail.pismolet.ru -> <IPv4 mail-сервера>`;
   - `AAAA mail.pismolet.ru -> <IPv6 mail-сервера>`, если IPv6 будет использоваться.
5. Написать в Timeweb запрос на PTR/rDNS:
   - `<IPv4 mail-сервера> -> mail.pismolet.ru`;
   - `<IPv6 mail-сервера> -> mail.pismolet.ru`, если IPv6 будет использоваться.
6. Зафиксировать, что `smtp.timeweb.ru` остаётся только запасным транспортом.
7. Основной транспорт - собственный SMTP на `mail.pismolet.ru`.
8. Зафиксировать стартовые лимиты прогрева:
   - не более 50 писем в день на первом этапе;
   - не более 1 письма в 30-60 секунд;
   - только проверенные адреса.

### Результат спринта

- Получено подтверждение Timeweb по PTR/rDNS и порту 25.
- Выбрана схема размещения mail-сервера.
- Есть выбранный hostname `mail.pismolet.ru`.
- Подготовлены DNS-записи для нового IP.
- Отправлена заявка на PTR.
- Зафиксирована архитектура отправки.

### Критерий готовности

```text
mail.pismolet.ru резолвится в IP mail-сервера
PTR-заявка отправлена в Timeweb
Timeweb подтвердил техническую возможность PTR/rDNS для выбранной локации
```

## Спринт 1. Подъём собственного SMTP на VDS

### Цель

Поставить и настроить сервер отправки писем на VDS с hostname `mail.pismolet.ru`.

### Задачи

1. Создать или подготовить VDS в локации, где доступен PTR/rDNS.
2. Настроить hostname ОС:
   - `mail.pismolet.ru`.
3. Установить Postfix.
4. Настроить Postfix только для исходящей отправки.
5. Ограничить приём писем извне, чтобы сервер не стал открытым relay.
6. Настроить HELO/EHLO как `mail.pismolet.ru`.
7. Настроить очереди Postfix.
8. Настроить базовые лимиты скорости.
9. Проверить, что сервер может отправлять письма напрямую на внешние адреса.
10. Проверить, что сервер не является open relay.
11. Если приложение остаётся на отдельном VPS, выбрать защищённый способ передачи писем от приложения к mail-серверу:
    - SMTP submission с авторизацией;
    - VPN/private network, если будет доступно;
    - временно ограниченный firewall по IP приложения.

### Результат спринта

- Postfix установлен.
- Сервер отправляет письма напрямую.
- Сервер не принимает чужую почту для пересылки.
- Есть базовая защита от open relay.
- HELO/EHLO совпадает с `mail.pismolet.ru`.

### Критерий готовности

Тестовая отправка проходит через наш сервер, а не через `smtp.timeweb.ru`.

```text
HELO/EHLO: mail.pismolet.ru
Outgoing IP: <IP mail-сервера>
```

## Спринт 2. DKIM, SPF, DMARC, PTR

### Цель

Сделать отправку технически корректной для Gmail, Yandex, Mail.ru.

### Задачи

1. Установить и настроить OpenDKIM.
2. Сгенерировать DKIM-ключ для `pismolet.ru`.
3. Добавить DKIM TXT-запись в DNS.
4. Обновить SPF:
   - добавить `ip4:<IPv4 mail-сервера>`;
   - добавить `ip6:<IPv6 mail-сервера>`, если IPv6 будет использоваться;
   - оставить `include:_spf.timeweb.ru` только если обычная почта Timeweb ещё используется.
5. Проверить DMARC для `pismolet.ru`.
6. Проверить PTR/rDNS после ответа Timeweb:
   - `<IP mail-сервера> -> mail.pismolet.ru`.
7. Проверить forward DNS:
   - `mail.pismolet.ru -> <IP mail-сервера>`.
8. Убедиться, что HELO/EHLO, PTR и A/AAAA согласованы.
9. Протестировать отправку на:
   - mail-tester;
   - Gmail;
   - Mail.ru;
   - Yandex.
10. Сравнить результат с текущей отправкой через `smtp.timeweb.ru`.

### Результат спринта

- SPF проходит.
- DKIM проходит.
- DMARC проходит.
- PTR совпадает с почтовым hostname.
- Mail-tester показывает понятную диагностическую картину.

### Критерий готовности

В заголовках письма:

```text
spf=pass
dkim=pass
dmarc=pass
```

Цепочка отправки больше не содержит:

```text
smtp-spam.timeweb.ru
smtpout14.timeweb.ru
```

И содержит наш почтовый узел:

```text
mail.pismolet.ru
```

## Спринт 3. Интеграция Письмолёта с собственным SMTP

### Цель

Переключить приложение на отправку через `mail.pismolet.ru`.

### Задачи

1. Добавить режим отправки:
   - `MailProvider=Smtp` с новым SMTP-host;
   - или отдельный режим `MailProvider=OwnSmtp`, если потребуется отличать транспорт.
2. Если mail-сервер на том же VPS, настроить окружение:

```env
MailProvider=Smtp
Smtp__Host=127.0.0.1
Smtp__Port=25
Smtp__SecureSocketOptions=None
Smtp__FromEmail=info@pismolet.ru
Smtp__FromName=Письмолёт
```

3. Если mail-сервер отдельный, настроить окружение для submission:

```env
MailProvider=Smtp
Smtp__Host=mail.pismolet.ru
Smtp__Port=587
Smtp__SecureSocketOptions=StartTls
Smtp__Username=<smtp-user>
Smtp__Password=<smtp-password>
Smtp__FromEmail=info@pismolet.ru
Smtp__FromName=Письмолёт
```

4. Проверить, нужен ли SMTP-логин для localhost или submission.
5. Если текущий код требует логин и пароль, доработать адаптер под локальный SMTP без авторизации.
6. Убедиться, что:
   - `From` остаётся `info@pismolet.ru`;
   - `Reply-To` остаётся `reply+...@reply.pismolet.ru`;
   - `List-Unsubscribe` абсолютный;
   - HTML-ссылка отписки абсолютная;
   - `Message-ID` имеет домен `pismolet.ru` или согласованный домен.
7. Добавить явное логирование транспорта:
   - SMTP через Timeweb;
   - собственный SMTP;
   - ошибка отправки;
   - результат отправки.

### Результат спринта

- Письмолёт отправляет через собственный SMTP.
- В письмах сохраняются корректные заголовки.
- В логах видно, каким транспортом ушло письмо.

### Критерий готовности

Тестовое письмо уходит через:

```text
mail.pismolet.ru / <IP mail-сервера>
```

а не через:

```text
smtp.timeweb.ru
```

## Спринт 4. Очередь отправки и лимиты прогрева

### Цель

Не дать Письмолёту отправлять слишком быстро и убивать репутацию IP.

### Задачи

1. Перевести отправку из `Inline` в очередь.
2. Ввести глобальный лимит отправки:
   - писем в минуту;
   - писем в час;
   - писем в сутки.
3. Ввести лимиты по доменам:
   - `gmail.com`;
   - `yandex.ru`;
   - `mail.ru`;
   - остальные.
4. Ввести лимиты по клиентам:
   - новый клиент;
   - проверенный клиент;
   - доверенный клиент.
5. Добавить паузы между письмами.
6. Добавить статус рассылки:
   - создана;
   - в очереди;
   - отправляется;
   - отправлена;
   - остановлена;
   - ошибка.
7. Добавить возможность вручную остановить рассылку.

### Результат спринта

- Письма уходят не залпом, а управляемо.
- Можно безопасно прогревать IP.
- Один клиент не может испортить репутацию всего сервиса.

### Стартовые лимиты

```text
Глобально: 50 писем в день
Скорость: 1 письмо в 30-60 секунд
Новый клиент: 25-50 писем в день
Проверенный клиент: 100-300 писем в день
```

### Критерий готовности

Рассылка на 50 адресов не отправляется мгновенно, а постепенно проходит через очередь.

## Спринт 5. Bounce-обработка

### Цель

Перестать отправлять на несуществующие и проблемные адреса.

### Задачи

1. Настроить адрес возвратов:
   - например `bounce@pismolet.ru` или VERP-адреса.
2. Настроить получение технических ответов.
3. Научиться связывать bounce с:
   - рассылкой;
   - клиентом;
   - конкретным адресом получателя.
4. Разделить ошибки:
   - hard bounce - адрес не существует;
   - soft bounce - временная ошибка;
   - mailbox full;
   - rejected;
   - domain not found.
5. Добавить статусы адресов:
   - активен;
   - отписан;
   - hard bounce;
   - временная ошибка;
   - заблокирован.
6. Исключать hard bounce из будущих рассылок.
7. Показывать клиенту отчёт по недоставленным письмам.

### Результат спринта

- Письмолёт перестаёт слать на плохие адреса.
- Репутация отправителя защищается от грязных баз.

### Критерий готовности

Если письмо вернулось как `user does not exist`, адрес автоматически исключается из будущих рассылок.

## Спринт 6. Жалобы, отписки и рейтинг клиента

### Цель

Защитить сервис от клиентов, которые портят репутацию.

### Задачи

1. Проверить работу глобальной отписки.
2. Логировать каждую отписку:
   - дата;
   - адрес;
   - рассылка;
   - клиент;
   - источник.
3. Добавить показатель качества клиента:
   - процент отписок;
   - процент hard bounce;
   - количество жалоб, если доступны;
   - ручные блокировки.
4. Ввести автоматические ограничения:
   - высокий bounce rate - остановка рассылки;
   - высокий процент отписок - ручная проверка;
   - подозрительный текст - ручная проверка.
5. Добавить предупреждение при загрузке базы:
   - нельзя использовать купленные базы;
   - нужно иметь основание для рассылки;
   - адреса из глобальной отписки исключаются.
6. Подготовить внутреннюю страницу администратора:
   - клиенты;
   - рассылки;
   - bounce;
   - отписки;
   - остановленные рассылки.

### Результат спринта

- Письмолёт начинает защищать собственную почтовую репутацию.
- Плохой клиент не ломает отправку всем остальным.

### Критерий готовности

Если рассылка даёт много ошибок или отписок, она останавливается или уходит на ручную проверку.

## Рекомендуемый порядок запуска

### Минимально необходимый набор для первого боевого теста

```text
Спринт 0 - подтверждение PTR/локации и DNS
Спринт 1 - Postfix на mail.pismolet.ru
Спринт 2 - SPF/DKIM/DMARC/PTR
Спринт 3 - интеграция Письмолёта
Спринт 4 - очередь и лимиты
```

После этого можно аккуратно использовать Письмолёт для хорошей реальной задачи, где база чистая и нет риска жалоб.

### До массовой коммерческой эксплуатации обязательно

```text
Спринт 5 - bounce
Спринт 6 - отписки, жалобы, рейтинг клиента
Спринт 7 - прогрев и постмастеры
```

## Практическая цель на ближайшую неделю

1. Получить от Timeweb подтверждение по московской локации, PTR и исходящему 25 порту.
2. Создать отдельный mail-сервер или принять решение о переносе текущего сервера.
3. Настроить `mail.pismolet.ru`, PTR/rDNS и Postfix.
4. Добиться SPF/DKIM/DMARC pass.
5. Убрать `smtp-spam.timeweb.ru` из цепочки доставки.
6. Отправить тесты на mail-tester, Gmail, Mail.ru, Yandex.
7. Запустить маленькую реальную рассылку на чистую базу.
8. Собирать результаты и не превышать стартовые лимиты.

## Риски

1. Новый IP может иметь слабую или неизвестную репутацию.
2. Без PTR/rDNS доставляемость будет плохой.
3. Если Timeweb не разрешит исходящий порт 25, прямая отправка не заработает.
4. Если использовать IPv6 без PTR, доставляемость может ухудшиться.
5. Без очереди и лимитов можно быстро испортить репутацию.
6. Без bounce-обработки сервис будет повторно слать на плохие адреса.
7. Один плохой клиент может повредить репутации всего сервиса.
8. Для коммерческого запуска понадобится постоянный мониторинг доставляемости.

## Временные решения

1. `smtp.timeweb.ru` можно оставить как запасной транспорт.
2. Для первых тестов использовать только чистую базу и небольшие объёмы.
3. Массовую отправку не запускать до завершения спринтов 0-4.
4. Коммерческую отправку не запускать до появления хотя бы базовой bounce-обработки и клиентских лимитов.
5. Текущий сервер приложения можно не переносить до подтверждения, что отдельный mail-сервер в Москве решает задачу PTR/rDNS и прямой отправки.
