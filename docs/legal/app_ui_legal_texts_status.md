## Правило фиксации статусов

По ходу выполнения каждый шаг из плана legal-пакета должен обновляться в этом файле отдельной отметкой статуса.

Допустимые статусы:

- `Не готово` — страница или интеграция отсутствует.
- `В работе` — начато внедрение, но ещё нет зелёной проверки.
- `Частично` — есть route или часть текста, но нет полного текста, ссылок, тестов или синхронизации с UI.
- `Готово, ждёт проверки` — код внесён, нужна серверная проверка build/test/deploy.
- `Готово` — код внесён, проверка зелёная, статус обновлён.
- `Отложено` — сознательно исключено из текущего прохода с причиной.

После каждого шага фиксировать:

1. изменённые файлы;
2. commit SHA;
3. результат проверки;
4. следующий шаг.

## Главный приоритет с 2026-06-24

Работу перестроить вокруг публикации полного legal-пакета на сайте и в приложении.

До завершения этого блока не переходим к импорту, оплате, расширению рассылок и другим продуктовым сценариям, кроме исправления критичных ошибок сборки/тестов.

Цель блока: все политики, соглашения, правила и юридически значимые тексты должны быть доступны как публичные страницы `/legal/...`, иметь понятные `document_key`/version, использоваться в UI-ссылках и быть связаны с legal evidence там, где есть акцепт.

## Решения после сверки

1. Тестовый почтовик больше не используется как допустимый сценарий разработки или тестирования.
2. Тестовый почтовик должен быть удалён из кода, конфигурации, тестов и пользовательского/админского интерфейса.
3. Новые тесты не должны зависеть от тестового почтовика.
4. Тест подтверждения email через тестовый почтовик удалён из текущего плана.
5. Для подтверждения email нужен отдельный production-like тестовый подход: реальный SMTP-адаптер, тестовый SMTP sink или прямой сервисный тест без dev UI.
