# ТЗ на разработку: юридический слой доказательств Письмолёта

Статус: задание на разработку.
Дата: 2026-06-24.
Источник: юридическая ветка проекта и анализ подхода `stranachitaliya` к версиям документов, журналу согласий и snapshot-логам.

## 1. Цель

Сделать в Письмолёте отдельный юридический слой доказательств: версии документов, подтверждения клиента, snapshot рассылки, журнал отписок, жалоб и запросов доказательств.

Главная задача слоя — чтобы по любой спорной рассылке можно было восстановить:

- кто запустил рассылку;
- какую базу загрузил клиент;
- какой источник базы указал клиент;
- какие декларации и подтверждения клиент принял;
- под какой версией текста он поставил галочку;
- какое письмо ушло адресату;
- какие отписки, жалобы, блокировки и модерационные решения были применены.

## 2. Важные ограничения MVP

1. Письмолёт не собирает базы для заказчика.
2. Письмолёт не реализует формы подписки адресатов для клиентов.
3. Double opt-in в MVP не нужен и не должен появляться в макете или моделях MVP.
4. Основной сценарий: клиент загружает свою уже имеющуюся базу и подтверждает её законность.
5. В MVP база адресатов — только email. ФИО адресатов не загружаем.
6. Клиент до первой отправки должен иметь в профиле email, ФИО и телефон.

## 3. Термины

- Клиент — пользователь Письмолёта, загружающий свою базу и запускающий рассылку.
- Адресат — получатель письма из базы клиента.
- Юридический документ — текст, под которым клиент ставит галочку или который применяется к сервису: соглашение, политика, декларация, подтверждение рекламного согласия, антиспам-правила.
- Подтверждение клиента — юридически значимое действие клиента: акцепт документа, декларация базы, подтверждение рекламы, согласие с тарифом, подтверждение запуска.
- Snapshot — неизменяемый снимок состояния на момент действия: текст письма, версия декларации, расчёт, список результатов импорта, решение модерации.

## 4. Модель данных

### 4.1. `LegalDocumentVersion`

Хранит версии юридически значимых текстов.

Поля:

- `id`;
- `document_type`;
- `version`;
- `title`;
- `body`;
- `body_hash`;
- `is_active`;
- `published_at`;
- `created_at`;
- `created_by_admin_id`;
- `comment`.

Типы документов:

- `terms_of_service`;
- `privacy_policy`;
- `client_personal_data_consent`;
- `recipient_data_processing_assignment`;
- `database_legality_declaration`;
- `advertising_consent_confirmation`;
- `anti_spam_policy`;
- `global_unsubscribe_rules`;
- `prohibited_content_policy`;
- `payment_terms`;
- `reply_retention_rules`;
- `service_email_footer`.

Требования:

- документ нельзя физически менять после публикации;
- новая редакция создаётся новой версией;
- у активного документа должен быть hash текста;
- все подтверждения должны ссылаться на конкретную версию документа.

### 4.2. `ClientLegalEvent`

Журнал юридически значимых действий клиента.

Поля:

- `id`;
- `client_id`;
- `user_id`;
- `event_type`;
- `document_version_id`;
- `document_type`;
- `document_version`;
- `document_hash`;
- `campaign_id`;
- `import_batch_id`;
- `message_id`;
- `value_json`;
- `accepted_at`;
- `revoked_at`;
- `status`;
- `method`;
- `page_url`;
- `ip_address`;
- `user_agent`;
- `created_at`.

Типы событий:

- `terms_accepted`;
- `privacy_policy_accepted`;
- `client_pd_consent_accepted`;
- `recipient_processing_assignment_accepted`;
- `database_source_selected`;
- `database_legality_confirmed`;
- `advertising_consent_confirmed`;
- `anti_spam_policy_accepted`;
- `payment_terms_accepted`;
- `campaign_launch_confirmed`;
- `proof_requested`;
- `proof_provided`;
- `proof_rejected`;
- `client_blocked`;
- `client_unblocked`.

`value_json` должен хранить прикладные данные события: выбранный источник базы, текст пояснения, количество адресов, цену, тип письма, id модератора, причину блокировки.

### 4.3. `ImportBatchEvidence`

Фиксирует импорт базы.

Поля:

- `id`;
- `client_id`;
- `campaign_id`;
- `uploaded_by_user_id`;
- `uploaded_at`;
- `file_name`;
- `file_hash`;
- `source_type`;
- `source_comment`;
- `total_rows`;
- `valid_emails_count`;
- `duplicate_count`;
- `invalid_count`;
- `global_unsubscribed_count`;
- `blocked_count`;
- `accepted_for_send_count`;
- `ip_address`;
- `user_agent`.

В MVP не хранить ФИО адресатов. Если клиент загружает лишние колонки, они игнорируются или отклоняются по отдельному правилу.

### 4.4. `CampaignSnapshot`

Неизменяемый снимок рассылки на момент запуска/оплаты/отправки.

Поля:

- `id`;
- `campaign_id`;
- `client_id`;
- `created_at`;
- `snapshot_type`;
- `subject`;
- `body_hash`;
- `body_snapshot`;
- `service_footer_snapshot`;
- `message_type`;
- `import_batch_id`;
- `pricing_snapshot_json`;
- `confirmation_events_json`;
- `moderation_snapshot_json`;
- `status`.

Типы snapshot:

- `before_payment`;
- `after_payment`;
- `before_send`;
- `after_moderation`;
- `complaint_freeze`.

### 4.5. `GlobalSuppressionEvent`

Фиксирует глобальную отписку и блокировку адреса.

Поля:

- `id`;
- `email_hash`;
- `email_encrypted` при необходимости;
- `event_type`;
- `source`;
- `campaign_id`;
- `client_id`;
- `message_id`;
- `created_at`;
- `ip_address`;
- `user_agent`;
- `reason`;
- `status`.

Типы событий:

- `unsubscribe`;
- `complaint`;
- `manual_block`;
- `hard_bounce_block`;
- `legal_request_block`.

Клиенту показывать только количество исключённых адресов, без раскрытия истории чужих отписок.

### 4.6. `RecipientComplaintCase`

Карточка жалобы адресата.

Поля:

- `id`;
- `email_hash`;
- `campaign_id`;
- `client_id`;
- `message_id`;
- `complaint_text`;
- `created_at`;
- `status`;
- `actions_json`;
- `client_notified_at`;
- `proof_requested_at`;
- `proof_due_at`;
- `proof_status`;
- `resolved_at`.

Статусы:

- `new`;
- `suppressed`;
- `client_notified`;
- `proof_requested`;
- `under_review`;
- `resolved_valid`;
- `resolved_violation`;
- `client_blocked`.

## 5. Сценарии интерфейса

### 5.1. Регистрация и профиль клиента

До первой отправки клиент должен заполнить:

- email;
- ФИО;
- телефон.

При регистрации или перед первой отправкой клиент принимает актуальные версии:

- пользовательского соглашения;
- политики обработки персональных данных;
- согласия клиента на обработку его персональных данных;
- поручения на обработку ПДн адресатов;
- антиспам-политики.

Каждый акцепт пишется в `ClientLegalEvent`.

### 5.2. Импорт базы

Перед запуском рассылки клиент выбирает источник базы:

- клиенты, ранее покупавшие товар или услугу;
- участники мероприятия;
- подписчики сайта клиента;
- контакты НКО, партнёров или волонтёров;
- адресаты, давшие согласие офлайн;
- другая законная база с пояснением.

После выбора источника клиент подтверждает декларацию законности базы. Галочка не должна быть проставлена заранее.

### 5.3. Рекламное письмо

Если клиент выбирает тип письма `рекламное` или `не уверен`, интерфейс должен требовать отдельное подтверждение наличия согласия на рекламу.

Если письмо рекламное, без подтверждения рекламного согласия запуск невозможен.

### 5.4. Оплата и запуск

Перед оплатой показать:

- количество загруженных строк;
- количество принятых email;
- количество исключённых email;
- цену;
- подтверждения клиента.

Перед запуском создать `CampaignSnapshot` типа `before_payment` или `before_send`.

### 5.5. Жалоба адресата

При жалобе адресата:

1. email добавляется в глобальную отписку или блокировку;
2. создаётся `RecipientComplaintCase`;
3. создаётся snapshot рассылки `complaint_freeze`;
4. клиент уведомляется о претензии;
5. при необходимости клиенту отправляется запрос доказательств;
6. все действия пишутся в журнал.

Адресату можно показать факт того, что клиент подтвердил законность базы, но нельзя раскрывать лишние данные других клиентов или внутренние данные модерации.

## 6. Админка

Нужны разделы:

1. Юридические документы и версии.
2. Журнал подтверждений клиентов.
3. Импорты и доказательства по базе.
4. Snapshot рассылок.
5. Глобальная отписка и блокировки.
6. Жалобы адресатов.
7. Запросы доказательств у клиентов.

В карточке рассылки администратор должен видеть:

- клиента;
- профиль клиента;
- импорт;
- источник базы;
- декларации клиента;
- рекламное подтверждение;
- snapshot письма;
- отписки и жалобы;
- модерационные решения.

## 7. Acceptance criteria

1. Для каждого юридически значимого текста хранится версия и hash.
2. Нельзя подтвердить декларацию без привязки к версии документа.
3. Нельзя запустить рассылку без декларации законности базы.
4. Нельзя запустить рекламную рассылку без отдельного подтверждения рекламного согласия.
5. По campaign_id можно восстановить все подтверждения клиента.
6. По жалобе адресата можно найти письмо, клиента, импорт и декларации.
7. Глобально отписавшийся адрес исключается из новых рассылок всех клиентов.
8. Клиент не видит историю чужих отписок, только количество исключений.
9. Double opt-in и формы подписки отсутствуют в MVP.
10. Удаление клиента должно предусматривать анонимизацию профиля с сохранением audit trail.

## 8. Что не делать в MVP

- Не делать формы подписки для клиентов.
- Не делать double opt-in.
- Не собирать базы за клиента.
- Не загружать ФИО адресатов.
- Не показывать клиенту персональные детали глобальной отписки.
- Не считать LLM или автоматическую проверку юридическим заключением.
