# SimpleMail / Письмолёт — ТЗ на платформу

Версия: 0.3  
Дата: 2026-06-08  
Статус: рабочий технический документ

## 1. Назначение документа

Документ описывает техническую реализацию MVP web-сервиса SimpleMail / «Письмолёт».

Цель MVP — дать нетехническому пользователю один простой сценарий:

> загрузить таблицу с адресами → написать письмо → увидеть расчёт → оплатить → запустить рассылку.

Платформа должна скрывать технические сущности от клиента, но внутри обеспечивать безопасность, логи, глобальную отписку, лимиты, risk-score, очередь отправки и обработку ответов.

## 2. Принятые технические решения MVP

1. MVP ориентирован на РФ.
2. Интерфейс — русский.
3. Валюта — рубль.
4. Клиент не настраивает DNS, SPF, DKIM, DMARC и SMTP.
5. Отправка идёт с домена сервиса.
6. Ответы принимает SimpleMail и пересылает клиенту.
7. Отписка глобальная для всего сервиса.
8. Базовый дневной лимит — до 1000 писем на клиента.
9. Лимиты изменяемые.
10. Денежный баланс клиента в MVP не хранится.
11. Основной платёжный сценарий MVP — оплата конкретной рассылки перед отправкой.
12. Письма-кредиты возможны как развитие, но в БД не должно быть рублёвого кошелька клиента.
13. LLM-проверка используется как risk-score, а не как юридическое заключение.
14. Премодерация гибридная: автоматические правила + ручная проверка рискованных случаев.
15. Купленные, спарсенные и чужие базы запрещены.
16. Собственный SMTP-сервер не входит в MVP.
17. Отправка выполняется через внешний email-провайдер по API.
18. Код должен использовать `EmailProviderAdapter`, чтобы можно было заменить провайдера.

## 4. Общая архитектура MVP

```text
Browser
  ↓
Django Web App
  ↓
PostgreSQL

Django Web App
  ↓
Redis
  ↓
Celery Workers
  ↓
EmailProviderAdapter
  ↓
External ESP API

ESP Webhooks
  ↓
Django Web App
  ↓
PostgreSQL
```

Основные компоненты:

- web-приложение Django;
- PostgreSQL как основная БД;
- Redis как broker/cache;
- Celery workers для импорта, отправки, webhooks, cleanup;
- внешний email-провайдер;
- inbound webhooks для ответов и bounce/complaints;
- Django Admin для внутренних операций;
- файловое хранилище для импортов и временных тел ответов.

## 5. Пользовательский MVP-flow

В интерфейсе пользователь не должен видеть сложные сущности вроде MailingList, ImportBatch, Campaign, SendEvent и CreditTransaction.

Снаружи есть одна понятная сущность:

> Рассылка.

Шаги:

1. Загрузить адреса.
2. Написать письмо.
3. Проверить список и стоимость.
4. Оплатить.
5. Дождаться отправки.
6. Посмотреть простой результат.

Сервис должен скрывать внутреннюю сложность и не выглядеть как профессиональная email-маркетинговая платформа.

## 6. Роли

### Клиент

Может:

- зарегистрироваться;
- войти;
- загрузить CSV/XLSX;
- подтвердить источник и законность базы;
- написать письмо;
- увидеть расчёт;
- оплатить рассылку;
- запустить отправку;
- посмотреть простой результат;
- получать ответы адресатов.

### Администратор

Может:

- просматривать клиентов;
- блокировать клиентов;
- менять лимиты;
- просматривать рассылки;
- отправлять рассылки на ручную проверку;
- одобрять или отклонять рассылки;
- просматривать жалобы, отписки, ошибки доставки;
- управлять глобальным suppression list;
- просматривать журналы событий.

### Адресат

Может:

- получить письмо;
- ответить на письмо;
- глобально отписаться от писем через сервис;
- пожаловаться через почтовый клиент.

Ответственность:

- `campaigns` — письмо, тема, тело, статус кампании.
- `sending` — очередь отправки, SendEvent, лимиты.
- `inbound` — ответы, bounce, complaints, webhooks.
- `suppression` — глобальная отписка и блок-листы.
- `moderation` — risk-score и ручная проверка.
- `billing` — расчёт стоимости, оплата рассылки, списания.
- `audit` — журналирование значимых событий.
- `admin_tools` — кастомные admin pages.

## 9. Импорт адресов

Система должна поддерживать CSV и XLSX.

Минимальная колонка:

- `email`.

Опционально:

- `name`.

Возможные дополнительные поля:

- дата согласия;
- способ согласия;
- источник;
- комментарий;
- URL формы;
- признак double opt-in.

После загрузки файла система должна:

1. создать ImportBatch;
2. прочитать файл;
3. нормализовать email;
4. удалить дубли;
5. проверить синтаксическую валидность email;
6. проверить глобальную отписку;
7. проверить внутренние блок-листы;
8. посчитать статистику импорта;
9. показать результат клиенту;
10. запросить источник базы и декларацию;
11. создать получателей только для разрешённых адресов.

## 10. Подтверждение законности базы

Перед запуском рассылки клиент обязан выбрать источник базы и подтвердить декларацию.

Запрещены:

- купленные базы;
- скачанные из интернета базы;
- спарсенные адреса;
- чужие CRM;
- открытые каталоги без согласия;
- чаты, группы, форумы и соцсети без согласия адресатов.

Требования:

- галочка не проставляется заранее;
- без подтверждения отправка невозможна;
- текст декларации версионируется;
- факт подтверждения логируется: дата, IP, user-agent, пользователь, версия текста, import/campaign.

## 11. Создание письма

Пользователь задаёт:

- отображаемое имя отправителя;
- тему;
- текст письма;
- тип письма: информационное, рекламное, сервисное, неизвестное.

Система добавляет автоматически:

- ссылку глобальной отписки;
- служебный блок «почему вы получили это письмо»;
- идентификатор рассылки;
- технические заголовки;
- plain text версию.

## 12. Расчёт и оплата

MVP-сценарий:

1. Система считает адреса, принятые к отправке.
2. Исключаются дубли, невалидные, глобально отписанные, заблокированные.
3. Система показывает стоимость.
4. Клиент переходит на оплату.
5. После successful payment рассылка уходит на проверки и отправку.

Не хранить денежный баланс клиента.

Возможное развитие:

- пакеты писем-кредитов;
- бесплатный стартовый тестовый лимит;
- снижение цены письма при объёме.

Если кредиты будут введены:

- хранится количество кредитов, а не рублёвый остаток;
- списание идемпотентно;
- повторная Celery-задача не должна списать кредиты дважды.

## 13. Очереди Celery

Очереди:

```text
import_queue
risk_queue
send_queue
webhook_queue
reply_queue
billing_queue
cleanup_queue
```

Правило:

> В очередь передаются ID сущностей, а не большие тела писем, файлы или списки адресов.

Задачи должны быть идемпотентными.

## 14. State machine кампании

```text
draft
  → uploaded
  → content_ready
  → priced
  → payment_pending
  → paid
  → pending_checks
  → review_required
  → approved
  → sending
  → sent
```

Возможные боковые статусы:

- cancelled;
- blocked;
- failed;
- paused.

Кредиты или оплата фиксируются до постановки в очередь отправки.

## 15. State machine импорта

```text
uploaded
  → processing
  → processed
  → waiting_declaration
  → confirmed
  → failed
  → blocked
```

## 16. EmailProviderAdapter

Код не должен быть жёстко привязан к одному ESP.

Минимальный контракт:

```python
class EmailProviderAdapter:
    def send_email(self, message): ...
    def send_test_email(self, message): ...
    def parse_delivery_webhook(self, payload): ...
    def parse_bounce_webhook(self, payload): ...
    def parse_complaint_webhook(self, payload): ...
    def parse_inbound_reply(self, payload): ...
```

Provider events:

- accepted;
- delivered;
- soft_bounce;
- hard_bounce;
- complaint;
- rejected;
- inbound_reply.

## 17. Обработка ответов

Варианты идентификации:

- уникальный Reply-To на кампанию;
- уникальный Reply-To на получателя;
- служебный идентификатор в заголовках;
- комбинация reply-to и заголовков.

Процесс:

1. Адресат отвечает на письмо.
2. Провайдер принимает входящее письмо.
3. Провайдер вызывает inbound webhook.
4. SimpleMail определяет клиента, кампанию, адресата.
5. Создаёт ReplyEvent.
6. Пересылает письмо клиенту.
7. Тело ответа хранится ограниченный срок.
8. Cleanup-задача удаляет тело и вложения, оставляя технический лог.

## 18. Bounce, complaints и deliverability

Hard bounce:

- фиксируется в SendEvent;
- может переводить Recipient в bounced;
- блокирует будущие отправки на этот адрес в рамках клиента.

Soft bounce:

- фиксируется;
- повторные soft bounce могут привести к блокировке адреса.

Complaint:

- добавляет email в GlobalSuppression;
- увеличивает risk клиента;
- может привести к ручной модерации или блокировке.

## 19. Глобальная отписка

Отписка действует на весь сервис.

Требования:

- добавлять email в GlobalSuppression;
- исключать email при импорте;
- исключать email перед отправкой;
- не раскрывать клиенту лишние сведения о том, от кого и когда адрес отписался;
- логировать попытки добавления отписанных адресов;
- защищать ссылку отписки от перебора.

## 20. LLM/risk-проверка

LLM-проверка возвращает:

- итоговый статус;
- risk-score;
- категории риска;
- краткое объяснение;
- рекомендацию: пропустить, предупредить, отправить на ручную проверку, заблокировать.

Категории риска:

- экстремизм;
- терроризм;
- разжигание ненависти;
- политически чувствительный контент;
- мошенничество;
- фишинг;
- финансовые обещания;
- медицина;
- БАДы;
- алкоголь/табак/запрещённые товары;
- взрослый контент;
- спам;
- агрессивная реклама;
- подозрительные ссылки;
- отсутствие понятного отправителя;
- отсутствие понятной причины получения письма.

## 23. Требования к разработке LLM-агентами

Разработка может выполняться Codex/DeepSeek-агентами, но:

- агент не принимает архитектурные решения без ADR;
- каждое изменение сопровождается тестами;
- критичные сценарии требуют human review;
- запрещено менять бизнес-логику оплаты, списаний, отписки, отправки и модерации без отдельного review;
- внешние интеграции покрываются mock/fake provider в тестах;
- должен быть набор seed/demo данных для локальной проверки.

## 24. Acceptance criteria MVP

MVP считается готовым к первичному тестированию, если:

1. Клиент может зарегистрироваться и подтвердить email.
2. Клиент может загрузить CSV.
3. Система показывает результат проверки адресов.
4. Система исключает дубли, невалидные адреса и глобально отписанных.
5. Клиент подтверждает источник базы и декларацию.
6. Клиент может написать простое письмо.
7. Система добавляет ссылку отписки.
8. Система считает количество писем и стоимость.
9. Клиент может оплатить конкретную рассылку.
10. После оплаты система запускает risk-проверку.
11. Подозрительная рассылка попадает на ручную проверку.
12. Одобренная рассылка отправляется через очередь.
13. Получатель может отписаться.
14. Глобально отписанный адрес больше не получает письма.
15. Ответ получателя пересылается клиенту.
16. Клиент видит простой результат рассылки.
17. Администратор может заблокировать клиента или рассылку.
18. Все ключевые действия логируются.
19. Проект запускается локально через Docker Compose.
20. Есть fake email provider и базовый набор тестов.

## 25. Что не входит в технический MVP

- собственный SMTP;
- Kubernetes;
- микросервисы;
- отдельный React/Next.js SPA;
- подключение домена клиента;
- Kafka/event streaming;
- сложный визуальный редактор писем;
- CRM;
- сегментация;
- автоворонки;
- открытия и клики, если не будет отдельного решения;
- денежный кошелёк клиента.

7. Какой стартовый лимит установить новым клиентам?
8. Какой домен выбрать для сервиса и отправки?

10. Какой срок хранения полных текстов отправленных писем?
