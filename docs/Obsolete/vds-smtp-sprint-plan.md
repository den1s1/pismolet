# План спринтов: переход Письмолёта на собственную SMTP-отправку с VDS

## Цель

Перевести Письмолёт с отправки через `smtp.timeweb.ru` на собственную исходящую почтовую инфраструктуру, чтобы убрать зависимость от фильтра `smtp-spam.timeweb.ru`, получить контроль над PTR/rDNS, HELO/EHLO, SPF, DKIM, DMARC и начать управляемо наращивать доставляемость.

Актуальная базовая схема после обсуждения стоимости инфраструктуры:

```text
Письмолёт на app.pismolet.ru -> локальный SMTP на том же московском VPS -> почтовые серверы получателей
```

Промежуточная схема размещения:

```text
app.pismolet.ru  -> московский VPS
mail.pismolet.ru -> тот же московский VPS
PTR <IP московского VPS> -> mail.pismolet.ru
```

Целевая схема на будущее, если потребуется разнести нагрузку или репутацию:

```text
app.pismolet.ru  -> отдельный app-VPS
mail.pismolet.ru -> отдельный mail-VPS
PTR <IP mail-VPS> -> mail.pismolet.ru
```

Роли доменов:

```text
app.pismolet.ru  - веб-приложение и личный кабинет
mail.pismolet.ru - почтовый hostname, HELO/EHLO и PTR/rDNS
pismolet.ru      - основной домен отправителя, DKIM/DMARC identity и брендовый домен
```

## Статус решения на 21 июня 2026 г.

Поддержка Timeweb подтвердила, что московская локация подходит для выбранной схемы:

```text
PTR/rDNS на mail.pismolet.ru можно установить
hostname mail.pismolet.ru можно установить
исходящий порт 25 может быть закрыт при создании сервера, но его откроют по запросу
текущий сервер приложения можно оставить как есть
отдельный VPS в Москве можно использовать только под почтовый сервер
особых требований к почтовым серверам нет, действуют общие правила платформы
```

После внутреннего решения по бюджету инфраструктуры выбран промежуточный вариант: **не платить за два VPS на этапе разработки и раннего MVP**. Вместо этого переносим текущий сервер приложения в московскую локацию, на этом же сервере поднимаем почтовую инфраструктуру и выставляем PTR на `mail.pismolet.ru`.

Актуальный план теперь такой:

```text
1. Создать образ/снапшот текущего VPS.
2. Поднять новый VPS в Москве из образа или перенести сервер средствами Timeweb.
3. Старый VPS не выключать до полной проверки нового.
4. Проверить приложение, PostgreSQL, nginx, systemd, SSL и деплой на новом сервере.
5. Настроить hostname нового сервера: mail.pismolet.ru.
6. Добавить DNS:
   app.pismolet.ru  A <новый IPv4>
   mail.pismolet.ru A <новый IPv4>
7. Настроить PTR/rDNS нового IP: <новый IPv4> -> mail.pismolet.ru.
8. Запросить открытие исходящего порта 25.
9. Настроить Postfix, DKIM, SPF, DMARC.
10. Переключить Письмолёт на локальный SMTP.
11. Старый VPS удалить только после проверки app и mail на новом сервере.
```

## Контекст текущего решения

Сейчас отправка через `smtp.timeweb.ru` технически работает, SPF/DKIM/DMARC проходят, но исходящий фильтр Timeweb добавляет письмам заголовки вида:

```text
X-Spam-Status: Yes
X-Spam-Score: 6.4
DBL_SPAM(6.50)
```

После этого Яндекс кладёт письма в спам. При этом Gmail и Mail.ru могут принимать письма во входящие, но сама метка от Timeweb остаётся в заголовках.

Вариант А - продолжать использовать обычный SMTP Timeweb - не подходит как основа продукта, потому что лимит почтового кластера мал для сервиса рассылок и мы зависим от их антиспам-фильтра.

Вариант Б - собственная исходящая отправка с VDS - выбран как следующий технический путь для MVP.

После ответа поддержки Timeweb и выбора промежуточной инфраструктурной схемы зафиксировано:

1. Для текущей локации Новосибирск PTR/rDNS для IP сервера, по прошлому ответу поддержки, технически недоступен.
2. Для собственного SMTP без корректного PTR/rDNS стартовать нельзя.
3. Поддержка подтвердила, что для VPS в Москве можно выставить PTR и hostname `mail.pismolet.ru`.
4. Поддержка подтвердила, что порт 25 откроют по запросу.
5. Отдельный mail-VPS технически возможен, но на этапе разработки и раннего MVP экономически избыточен.
6. Основной рабочий сценарий на ближайший этап - один московский VPS, на котором временно живут и приложение, и почтовый сервер.
7. Разделение на `app-VPS` и `mail-VPS` остаётся будущим масштабируемым вариантом.

## Целевая почтовая схема

Нормальная схема для собственного SMTP на одном московском VPS:

```text
app.pismolet.ru A  <IPv4 московского VPS>
mail.pismolet.ru A <IPv4 московского VPS>
PTR <IPv4> -> mail.pismolet.ru
hostname ОС -> mail.pismolet.ru
HELO/EHLO -> mail.pismolet.ru
SPF pismolet.ru -> разрешает IPv4 московского VPS
DKIM -> подписывает письма от имени pismolet.ru
DMARC -> действует для pismolet.ru
```

PTR не ставим на голый `pismolet.ru`, чтобы не смешивать сайт, домен бренда и почтовый узел. Почтовый узел должен называться `mail.pismolet.ru`.

Для первого запуска предпочтительно использовать только IPv4. IPv6 можно подключить позже, когда будет понятно, что для него также корректно выставлен PTR/rDNS и он не ухудшает доставляемость.

Важно: `app.pismolet.ru` и `mail.pismolet.ru` могут временно указывать на один IP. Для веб-приложения это нормально: nginx обслуживает `app.pismolet.ru`, а Postfix представляется внешним почтовым серверам как `mail.pismolet.ru`.

## Спринт 0. Подготовка и архитектурное решение

### Цель

Закрыть архитектурную неопределённость и подготовить перенос текущего VPS в московскую локацию.

### Задачи

1. Зафиксировать ответ поддержки Timeweb:
   - PTR/rDNS на `mail.pismolet.ru` в Москве можно;
   - hostname `mail.pismolet.ru` можно;
   - исходящий порт 25 откроют по запросу;
   - отдельный VPS только под почту можно;
   - текущий сервер приложения можно не переносить, но перенос в Москву тоже допустим.
2. Выбрать схему размещения на ближайший этап:
   - основной вариант: один московский VPS для app + mail;
   - будущий вариант: отдельные VPS для app и mail.
3. Выбрать технический hostname для исходящей почты:
   - `mail.pismolet.ru`.
4. Подготовить перенос текущего сервера:
   - сделать образ/снапшот текущего VPS;
   - поднять новый VPS в Москве из образа или согласовать перенос с Timeweb;
   - старый VPS оставить включённым до проверки нового.
5. После получения нового IP подготовить DNS-записи:
   - `A app.pismolet.ru -> <IPv4 московского VPS>`;
   - `A mail.pismolet.ru -> <IPv4 московского VPS>`.
6. Настроить PTR/rDNS:
   - `<IPv4 московского VPS> -> mail.pismolet.ru`.
7. Запросить открытие исходящего порта 25 у Timeweb после запуска московского VPS.
8. Зафиксировать, что `smtp.timeweb.ru` остаётся запасным транспортом.
9. Основной транспорт после настройки - локальный SMTP на `mail.pismolet.ru`.
10. Зафиксировать стартовые лимиты прогрева:
   - не более 50 писем в день на первом этапе;
   - не более 1 письма в 30-60 секунд;
   - только проверенные адреса.

### Результат спринта

- Ответ поддержки Timeweb зафиксирован.
- Выбрана схема: один московский VPS для app + mail.
- Есть выбранный hostname `mail.pismolet.ru`.
- Понятен порядок переноса, DNS, PTR и открытия порта 25.
- Зафиксирована архитектура отправки.

### Критерий готовности

```text
Решение по переносу текущего VPS в Москву принято
Timeweb подтвердил PTR/rDNS и hostname mail.pismolet.ru
Timeweb подтвердил возможность открытия исходящего порта 25 по запросу
Старый VPS не выключается до проверки нового
```

## Спринт 1. Перенос VPS в Москву и подъём собственного SMTP

### Цель

Перенести текущий сервер приложения в московскую локацию и поставить на нём сервер отправки писем с hostname `mail.pismolet.ru`.

### Задачи

1. Создать образ/снапшот текущего VPS.
2. Поднять новый VPS в московской локации Timeweb из образа или выполнить перенос через механизм Timeweb.
3. Проверить на новом VPS:
   - код в `/opt/pismolet`;
   - опубликованное приложение в `/var/www/pismolet`;
   - PostgreSQL и базу `pismolet`;
   - `/etc/pismolet/pismolet.env`;
   - systemd service `pismolet`;
   - nginx config;
   - сертификаты или возможность перевыпустить сертификаты;
   - `curl http://127.0.0.1:5000/health`.
4. Настроить hostname ОС:
   - `mail.pismolet.ru`.
5. Добавить DNS:
   - `A app.pismolet.ru -> <IPv4 московского VPS>`;
   - `A mail.pismolet.ru -> <IPv4 московского VPS>`.
6. Перевыпустить или проверить SSL для `app.pismolet.ru` после переключения DNS.
7. Настроить PTR/rDNS нового IP на `mail.pismolet.ru` через панель Timeweb или запросом в поддержку.
8. Запросить открытие исходящего порта 25 для московского VPS.
9. Установить Postfix.
10. Настроить Postfix для исходящей отправки и локального submission от приложения.
11. Ограничить приём писем извне, чтобы сервер не стал open relay.
12. Настроить HELO/EHLO как `mail.pismolet.ru`.
13. Настроить очереди Postfix.
14. Настроить базовые лимиты скорости.
15. Проверить прямую отправку на внешний адрес.
16. Проверить, что сервер не является open relay.
17. Старый VPS выключать или удалять только после полной проверки нового сервера.

### Результат спринта

- Приложение работает на московском VPS.
- `app.pismolet.ru` резолвится в IP московского VPS.
- `mail.pismolet.ru` резолвится в тот же IP.
- PTR/rDNS указывает на `mail.pismolet.ru`.
- Порт 25 открыт.
- Postfix установлен.
- Сервер отправляет письма напрямую.
- Сервер не принимает чужую почту для пересылки.
- HELO/EHLO совпадает с `mail.pismolet.ru`.

### Критерий готовности

```text
https://app.pismolet.ru/health -> ok
HELO/EHLO: mail.pismolet.ru
Outgoing IP: <IP московского VPS>
PTR: <IP московского VPS> -> mail.pismolet.ru
Тестовая отправка идёт напрямую, а не через smtp.timeweb.ru
```

## Спринт 2. DKIM, SPF, DMARC, PTR

### Цель

Сделать отправку технически корректной для Gmail, Yandex, Mail.ru.

### Задачи

1. Установить и настроить OpenDKIM.
2. Сгенерировать DKIM-ключ для `pismolet.ru`.
3. Добавить DKIM TXT-запись в DNS.
4. Обновить SPF:
   - добавить `ip4:<IPv4 московского VPS>`;
   - добавить `ip6:<IPv6 московского VPS>`, только если IPv6 будет использоваться;
   - оставить `include:_spf.timeweb.ru` только если обычная почта Timeweb ещё используется.
5. Проверить DMARC для `pismolet.ru`.
6. Проверить PTR/rDNS:
   - `<IP московского VPS> -> mail.pismolet.ru`.
7. Проверить forward DNS:
   - `mail.pismolet.ru -> <IP московского VPS>`.
8. Убедиться, что hostname, HELO/EHLO, PTR и A/AAAA согласованы.
9. Протестировать отправку на:
   - mail-tester;
   - Gmail;
   - Mail.ru;
   - Yandex.
10. Сравнить результат с текущей отправкой через `smtp.timeweb.ru`.

### Результат спринта

- SPF проходит.
- DKIM проходит.
- DMARC проходит.
- PTR совпадает с почтовым hostname.
- Mail-tester показывает понятную диагностическую картину.

### Критерий готовности

В заголовках письма:

```text
spf=pass
dkim=pass
dmarc=pass
```

Цепочка отправки больше не содержит:

```text
smtp-spam.timeweb.ru
smtpout14.timeweb.ru
```

И содержит наш почтовый узел:

```text
mail.pismolet.ru
```

## Спринт 3. Интеграция Письмолёта с локальным SMTP

### Цель

Переключить приложение на отправку через локальный Postfix на том же московском VPS.

### Задачи

1. Использовать текущий режим `MailProvider=Smtp` с локальным SMTP-host.
2. Настроить окружение приложения для локальной отправки:

```env
MailProvider=Smtp
Smtp__Host=127.0.0.1
Smtp__Port=25
Smtp__SecureSocketOptions=None
Smtp__FromEmail=info@pismolet.ru
Smtp__FromName=Письмолёт
```

3. Проверить, требует ли текущий SMTP-адаптер логин/пароль.
4. Если требует, доработать адаптер под локальный SMTP без авторизации или настроить локальный submission с авторизацией.
5. Убедиться, что:
   - `From` остаётся `info@pismolet.ru`;
   - `Reply-To` остаётся `reply+...@reply.pismolet.ru`;
   - `List-Unsubscribe` абсолютный;
   - HTML-ссылка отписки абсолютная;
   - `Message-ID` имеет домен `pismolet.ru` или согласованный домен.
6. Добавить явное логирование транспорта:
   - SMTP через Timeweb;
   - локальный SMTP;
   - ошибка отправки;
   - результат отправки.
7. Сохранить `smtp.timeweb.ru` как запасной транспорт, но не использовать его по умолчанию.

### Результат спринта

- Письмолёт отправляет через локальный Postfix.
- В письмах сохраняются корректные заголовки.
- В логах видно, каким транспортом ушло письмо.

### Критерий готовности

Тестовое письмо уходит через:

```text
mail.pismolet.ru / <IP московского VPS>
```

а не через:

```text
smtp.timeweb.ru
```

## Спринт 4. Очередь отправки и лимиты прогрева

   - `gmail.com`;
   - `yandex.ru`;

4. Ввести лимиты по клиентам:
   - новый клиент;
   - проверенный клиент;
   - доверенный клиент.
5. Добавить паузы между письмами.
6. Добавить статус рассылки:
   - создана;
   - в очереди;
   - отправляется;
   - отправлена;
   - остановлена;

7. Добавить возможность вручную остановить рассылку.

## Спринт 5. Bounce-обработка

### Цель

Перестать отправлять на несуществующие и проблемные адреса.

### Задачи

1. Настроить адрес возвратов:
   - например `bounce@pismolet.ru` или VERP-адреса.
2. Настроить получение технических ответов.
3. Научиться связывать bounce с:
   - рассылкой;
   - клиентом;
   - конкретным адресом получателя.
4. Разделить ошибки:
   - hard bounce - адрес не существует;
   - soft bounce - временная ошибка;
   - mailbox full;
   - rejected;
   - domain not found.
5. Добавить статусы адресов:
   - активен;
   - отписан;
   - hard bounce;
   - временная ошибка;
   - заблокирован.
6. Исключать hard bounce из будущих рассылок.
7. Показывать клиенту отчёт по недоставленным письмам.

### Результат спринта

- Письмолёт перестаёт слать на плохие адреса.
- Репутация отправителя защищается от грязных баз.

### Критерий готовности

Если письмо вернулось как `user does not exist`, адрес автоматически исключается из будущих рассылок.

   - подозрительный текст - ручная проверка.
5. Добавить предупреждение при загрузке базы:
   - нельзя использовать купленные базы;

6. Подготовить внутреннюю страницу администратора:
   - клиенты;
   - рассылки;
   - bounce;
   - отписки;
   - остановленные рассылки.

### Результат спринта

- Письмолёт начинает защищать собственную почтовую репутацию.
- Плохой клиент не ломает отправку всем остальным.

### Критерий готовности

Если рассылка даёт много ошибок или отписок, она останавливается или уходит на ручную проверку.

   - день 6-10: 300-500 писем;
   - дальше по результатам.
4. Фиксировать результаты:
   - входящие;
   - промо/рассылки;
   - спам;
   - отклонено.
5. Корректировать лимиты по доменам.
6. Подготовить правила для первой реальной рассылки.

### Результат спринта

- Есть первая карта доставляемости.
- Понятно, какие домены принимают нормально, а где нужны ограничения.

### Критерий готовности

Есть таблица тестов по основным почтовым провайдерам и понятный стартовый лимит для реальной рассылки.

## Рекомендуемый порядок запуска

### Минимально необходимый набор для первого боевого теста

```text
Спринт 0 - архитектура подтверждена, переносим VPS в Москву
Спринт 1 - app + Postfix на московском VPS, PTR на mail.pismolet.ru
Спринт 2 - SPF/DKIM/DMARC/PTR
Спринт 3 - интеграция Письмолёта с локальным SMTP
Спринт 4 - очередь и лимиты
```

После этого можно аккуратно использовать Письмолёт для хорошей реальной задачи, где база чистая и нет риска жалоб.

### До массовой коммерческой эксплуатации обязательно

```text
Спринт 5 - bounce
Спринт 6 - отписки, жалобы, рейтинг клиента
Спринт 7 - прогрев и постмастеры
```

## Практическая цель на ближайшую неделю

1. Сделать свежий образ/снапшот текущего VPS.
2. Поднять сервер в московской локации из образа или выполнить перенос через Timeweb.
3. Не выключать старый сервер до полной проверки нового.
4. Проверить приложение на новом VPS: PostgreSQL, systemd, nginx, env, `/health`, деплой.
5. Назначить hostname сервера `mail.pismolet.ru`.
6. Добавить DNS:
   - `app.pismolet.ru A <IPv4 московского VPS>`;
   - `mail.pismolet.ru A <IPv4 московского VPS>`.
7. Настроить PTR/rDNS `<IPv4 московского VPS> -> mail.pismolet.ru` через панель Timeweb или поддержку.
8. Запросить открытие исходящего порта 25.
9. Установить и настроить Postfix.
10. Добиться согласования hostname, HELO/EHLO, A-записи и PTR.
11. Настроить DKIM, SPF, DMARC.
12. Убрать `smtp-spam.timeweb.ru` из цепочки доставки.
13. Отправить тесты на mail-tester, Gmail, Mail.ru, Yandex.
14. Запустить маленькую реальную рассылку на чистую базу только после прохождения технических проверок.
15. Собирать результаты и не превышать стартовые лимиты.

## Риски

1. Новый IP может иметь слабую или неизвестную репутацию.
2. Без PTR/rDNS доставляемость будет плохой.
3. Порт 25 может быть закрыт при создании сервера, его нужно открыть запросом в Timeweb.
4. Если использовать IPv6 без PTR, доставляемость может ухудшиться.
5. В промежуточной схеме app и mail делят один IP.
6. Если почтовая репутация IP пострадает, это будет тот же IP, где работает приложение.
7. Перенос сервера может дать простой, если переключить DNS до проверки нового VPS.
8. Без очереди и лимитов можно быстро испортить репутацию.
9. Без bounce-обработки сервис будет повторно слать на плохие адреса.
10. Один плохой клиент может повредить репутации всего сервиса.
11. Для коммерческого запуска понадобится постоянный мониторинг доставляемости.

## Временные решения

1. `smtp.timeweb.ru` можно оставить как запасной транспорт.
2. Для первых тестов использовать только чистую базу и небольшие объёмы.
3. Массовую отправку не запускать до завершения спринтов 0-4.
4. Коммерческую отправку не запускать до появления хотя бы базовой bounce-обработки и клиентских лимитов.
5. Старый VPS держать до полной проверки нового московского сервера.
6. Отдельный mail-VPS пока не создаём, чтобы не платить за два VPS на этапе разработки.
7. Когда нагрузка, репутационные риски или архитектура потребуют, mail можно вынести на отдельный VPS без изменения доменной идеи: `mail.pismolet.ru` останется почтовым узлом.
