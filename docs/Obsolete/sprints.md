# SimpleMail / Письмолёт — план спринтов MVP

Версия: 0.1  
Дата: 2026-06-15  
Статус: рабочий план реализации

## 1. Назначение документа

Документ описывает план разработки MVP сервиса SimpleMail / «Письмолёт» по спринтам.

План составлен с приоритетом на архитектуру и вертикальные пользовательские сценарии: каждый спринт должен давать видимый результат в интерфейсе, а не только внутренние изменения.

Главный пользовательский сценарий MVP:

> загрузить таблицу с адресами → написать письмо → увидеть расчёт → оплатить → запустить рассылку.

## 2. Принятые решения для этого плана

1. Приоритетная архитектура реализации — .NET.
2. Web UI — ASP.NET Core Razor Pages.
3. Для переиспользуемых частей интерфейса используются частичные представления / partial components.
4. HTML-шаблоны из папки `ui` являются ориентиром для внешнего вида и пользовательского flow.
5. Fake-оплата сохраняется до конца MVP и используется во всех спринтах вместо реальной платёжной интеграции.
6. Реальный платёжный агент подключается после стабилизации MVP-flow отдельным этапом.
7. Внутренние сущности могут быть технически сложными, но пользователь должен видеть простую сущность «Рассылка».
8. Каждый спринт должен включать unit-тесты, интеграционные тесты и ручные тесты.

## 3. Базовые архитектурные допущения

Целевая структура solution:

```text
src/
  Pismolet.Web/
  Pismolet.Application/
  Pismolet.Domain/
  Pismolet.Infrastructure/

tests/
  Pismolet.UnitTests/
  Pismolet.IntegrationTests/
```

Рекомендуемый стек:

- ASP.NET Core;
- Razor Pages;
- partial views / view components для повторяемых UI-блоков;
- PostgreSQL;
- Entity Framework Core;
- Hangfire для фоновых задач и очередей MVP;
- внешний email-провайдер через `IEmailProviderAdapter`;
- fake email provider для разработки и тестов;
- fake payment provider до конца MVP;
- xUnit;
- FluentAssertions;
- Testcontainers для интеграционных тестов с PostgreSQL;
- Docker Compose для локального запуска.

## 4. Сквозные правила разработки

1. Каждый спринт должен завершаться работающим вертикальным сценарием.
2. Внешний UI не должен показывать клиенту технические сущности вроде `Campaign`, `ImportBatch`, `SendEvent`, `PaymentAttempt`.
3. Пользовательский язык интерфейса — русский.
4. Разработка LLM-агентами допускается, но архитектурные изменения фиксируются через ADR.
5. Изменения в оплате, отписке, отправке, модерации и лимитах требуют отдельного review.
6. Внешние интеграции должны иметь fake/mock-реализации.
7. Задачи фоновой обработки должны быть идемпотентными.
8. В очередь передаются ID сущностей, а не большие тела писем, файлы или списки адресов.
