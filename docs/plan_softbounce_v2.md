# План SoftBounce v2

Дата фиксации: 2026-06-23
Статус: план, к реализации пока не приступаем.

## 1. Контекст

В текущей реализации Письмолёта доставка уже отслеживается по событиям отправки, а постоянные ошибки доставки приводят к исключению адресов из будущих рассылок.

Текущее поведение:

- `HardBounce` — постоянная ошибка доставки; адрес попадает в suppression и больше не используется для будущих отправок клиента.
- `Complaint` — жалоба получателя; адрес должен рассматриваться как высокий репутационный риск.
- `Unsubscribe` — получатель отказался от рассылок; адрес нельзя использовать без нового законного основания.
- `SoftBounce` — временная ошибка доставки; адрес не блокируется автоматически, но может показываться как предупреждение.

Пример SoftBounce:

```text
recipient's inbox is out of storage
```

Такой ответ означает, что ящик получателя переполнен. Это не обязательно мёртвый адрес: он может снова стать доступным позже.

## 2. Проблема

Если адрес много раз подряд даёт SoftBounce, продолжать отправлять на него каждую новую рассылку рискованно:

- ухудшается репутация домена и IP;
- растёт количество неуспешных доставок;
- клиент может платить за письма, которые с высокой вероятностью не будут доставлены;
- службе поддержки сложно объяснять повторяющиеся временные ошибки без единой политики.

При этом нельзя обращаться с SoftBounce так же жёстко, как с HardBounce: временная ошибка может исчезнуть.

## 3. Цель SoftBounce v2

Добавить промежуточный режим между предупреждением и постоянным suppression:

```text
несколько повторяющихся SoftBounce
→ временное исключение адреса из отправки
→ через заданный срок адрес снова можно попробовать
```

Цель — снизить нагрузку на доставку и репутационные риски, не превращая временные ошибки в пожизненную блокировку.

## 4. Предлагаемые MVP-правила

### 4.1. Условие создания временного suppression

Временное исключение создаётся, если одновременно выполняются условия:

- для пары `ClientId + EmailNormalized` было не менее 3 SoftBounce;
- эти SoftBounce произошли за последние 14 дней;
- по адресу нет постоянного suppression (`HardBounce`, `Complaint`, `Unsubscribe`, `Manual`);
- адрес не находится в активном временном suppression.

### 4.2. Срок временного suppression

Базовый срок:

```text
7 дней
```

На этот период адрес пропускается при импорте, расчёте получателей, оплате и отправке.

### 4.3. Поведение после истечения срока

После окончания срока временного suppression:

- адрес снова может участвовать в рассылках;
- история SoftBounce сохраняется;
- если SoftBounce продолжаются, временное suppression может быть создано повторно.

### 4.4. Что не делаем в MVP

На первом этапе не делаем:

- сложные разные пороги по доменам;
- ручную настройку порогов в UI;
- автоматическую эскалацию SoftBounce в HardBounce;
- отдельные правила для каждого email-провайдера;
- автоматическую очистку истории SoftBounce.

## 5. Новые сущности и поля

Есть два возможных технических варианта.

### Вариант A. Расширить существующий suppression

Добавить к существующей модели suppression:

- новый reason: `TemporarySoftBounce` или `SoftBounceThreshold`;
- поле `ExpiresAt` / `SuppressedUntil`;
- признак активности временного suppression через сравнение даты.

Плюсы:

- единый список исключённых адресов;
- `/admin/suppressions` может показывать постоянные и временные исключения вместе;
- проще проверять адрес перед отправкой.

Минусы:

- существующая модель suppression станет сложнее;
- нужно аккуратно не смешать постоянные и временные исключения.

### Вариант B. Отдельная таблица temporary suppressions

Создать отдельную таблицу для временных исключений.

Плюсы:

- постоянные suppression остаются простыми;
- проще разделить правила жизни временных исключений.

Минусы:

- при импорте и отправке нужно проверять два источника;
- сложнее объяснять статус адреса в одном месте.

### Предпочтение для MVP

Для MVP предпочтителен вариант A: расширить существующий механизм suppression и явно различать постоянные и временные причины.

## 6. Изменения в бизнес-логике

### 6.1. Импорт адресов

При импорте списка адресов нужно учитывать:

- постоянные suppression — адрес исключается;
- активные временные suppression — адрес временно исключается;
- SoftBounce без достижения порога — адрес остаётся, но может показываться предупреждение.

В UI импорта нужно разделять:

- окончательно исключённые адреса;
- временно исключённые адреса;
- предупреждения без исключения.

### 6.2. Расчёт стоимости

Активные временные suppression не должны попадать в оплачиваемое количество писем.

Клиент не должен платить за адреса, которые система сама решила временно не отправлять.

### 6.3. Отправка

Перед постановкой письма в отправку нужно проверять:

- постоянный suppression;
- активный временный suppression;
- глобальную отписку;
- ограничения клиента.

### 6.4. Создание временного suppression

Временное suppression можно создавать после обработки нового SoftBounce-события.

Алгоритм:

```text
получили SoftBounce
→ считаем SoftBounce по ClientId + EmailNormalized за последние 14 дней
→ если счётчик >= 3
→ создаём TemporarySoftBounce на 7 дней
```

## 7. Изменения в админке

### 7.1. /admin/suppressions

Добавить отображение:

- тип: постоянный или временный;
- дата начала;
- дата окончания для временных suppression;
- активен ли временный suppression сейчас;
- причина создания: повторяющиеся SoftBounce.

Фильтры:

- все;
- только постоянные;
- только временные;
- только активные временные;
- только истёкшие временные.

### 7.2. /admin/suppressions/detail

Для адреса показывать:

- историю SoftBounce;
- сколько SoftBounce было за окно 14 дней;
- почему был создан временный suppression;
- когда он истекает;
- что будет после истечения.

### 7.3. /admin/delivery/mailing/{mailingId}

В блоке «Что делать» добавить рекомендации:

- если SoftBounce повторяются часто — адрес может быть временно исключён;
- если уже есть TemporarySoftBounce — показать срок действия.

## 8. Изменения в клиентском интерфейсе

В клиентском импорте и перед отправкой желательно показывать понятное объяснение:

```text
Адрес временно пропущен из-за повторяющихся временных ошибок доставки. Мы попробуем снова после окончания периода временного исключения.
```

Для клиента это должно выглядеть не как наказание, а как защита качества отправки и бюджета.

## 9. Тесты

Минимальный набор тестов:

1. Один SoftBounce не создаёт suppression.
2. Два SoftBounce за 14 дней не создают suppression.
3. Три SoftBounce за 14 дней создают временный suppression.
4. SoftBounce старше 14 дней не учитывается в пороге.
5. Постоянный HardBounce suppression имеет приоритет над TemporarySoftBounce.
6. Активный TemporarySoftBounce исключает адрес из импорта/отправки.
7. Истёкший TemporarySoftBounce не исключает адрес из отправки.
8. Стоимость рассылки не включает временно исключённые адреса.
9. `/admin/suppressions` показывает временные suppression.
10. `/admin/suppressions/detail` показывает историю и срок действия временного suppression.

## 10. Риски и открытые вопросы

### 10.1. Риск ложного исключения

SoftBounce может быть случайным. Поэтому порог не должен быть слишком низким.

MVP-порог `3 за 14 дней` выглядит достаточно осторожным.

### 10.2. Поведение после повторного истечения

Нужно решить, увеличивать ли срок временного suppression при повторениях:

- всегда 7 дней;
- 7 дней, затем 14, затем 30;
- оставить для будущей версии.

Для MVP: всегда 7 дней.

### 10.3. Нужна ли ручная разблокировка

В MVP можно не делать отдельную кнопку ручной разблокировки временного suppression.

Достаточно автоматического истечения срока.

### 10.4. Нужно ли показывать клиенту точный bounce-текст

Полный технический текст bounce лучше оставлять в админке.

Клиенту показывать краткое объяснение без перегруза техническими деталями.

## 11. Рекомендуемый порядок реализации

1. Зафиксировать модель причины `TemporarySoftBounce` и поле срока действия.
2. Добавить миграцию БД.
3. Добавить сервис создания временного suppression по SoftBounce-порогам.
4. Подключить сервис к обработке delivery events.
5. Обновить импорт и расчёт стоимости.
6. Обновить отправку.
7. Обновить `/admin/suppressions`.
8. Обновить `/admin/suppressions/detail`.
9. Обновить клиентские сообщения импорта/отправки.
10. Покрыть сценарии тестами.

## 12. Решение на текущий момент

SoftBounce v2 считается следующим логичным этапом после завершения delivery drill-down и suppression admin UI.

К реализации сейчас не приступаем. Документ фиксирует план, MVP-правила и открытые вопросы для будущего технического блока.
