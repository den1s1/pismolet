# Чек-лист подготовки сайта к модерации Robokassa

Дата проверки: 2026-06-26

Направление: SEO, публичный сайт, платёжная готовность

Текущий статус: технический контур для тестовой проверки Robokassa готов; остаются финальная юридическая вычитка, ручная сквозная тестовая оплата из личного кабинета и сверка публичных сведений продавца с данными в кабинете Robokassa.

## 1. Что проверяется перед отправкой на модерацию

1. Сайт открывается публично и не выглядит как тестовая заглушка.
2. На главной понятно, что продаётся: простая email-рассылка по базе клиента.
3. На странице тарифов описан принцип расчёта стоимости и момент оплаты.
4. На сайте есть контакты для связи с продавцом/поддержкой.
5. С сайта доступны юридические документы: оферта, политика персональных данных, правила рассылок, условия возврата.
6. В публичных текстах нет внутренних слов и признаков разработки: `MVP`, `test`, `deploy`, `placeholder`, `soon`, «заглушка», «рабочая редакция», «готовится к запуску».
7. Sitemap содержит все публичные страницы, а страницы имеют canonical.
8. Информация на сайте совпадает с данными, которые будут указаны в Robokassa.

## 3.2. Сверка с официальной технической документацией Robokassa

Сверено с официальной документацией Robokassa:

- Quick start: `https://docs.robokassa.ru/ru/quick-start`;
- уведомления и переадресация: `https://docs.robokassa.ru/ru/notifications-and-redirects`;
- интерфейс оплаты: `https://docs.robokassa.ru/ru/pay-interface`.

Подтверждённые требования и соответствие:

- [x] для старта платежа используются `MerchantLogin`, `OutSum`, `InvId`, `SignatureValue`;
- [x] `InvId` формируется уникальным для платежной попытки и сохраняется как provider operation id;
- [x] пользовательский параметр `Shp_mailingId` включается в подпись и возвращается в callback;
- [x] `ResultURL` проверяет подпись через `Password2`;
- [x] `ResultURL` дополнительно сверяет сумму платежа с заказом;
- [x] `ResultURL` отвечает `OK{InvId}`;
- [x] `SuccessURL` не меняет статус оплаты сам по себе;
- [x] `FailURL` не меняет статус оплаты;
- [x] `IsTest=1` передаётся в тестовом режиме;
- [x] учтена разница формата `OutSum`: тестовая среда может присылать 2 знака после запятой, боевая среда — 6 знаков. В коде сумма теперь сравнивается численно, а не строкой.

## 6. Решение по статусу

Сайт и приложение технически готовы к тестовой проверке Robokassa: публичные страницы опубликованы, тарифная сетка зафиксирована, контакты и сведения о продавце вынесены на сайт, порядок возврата описан, платёжные URL на `app.pismolet.ru` доступны, тестовый режим Robokassa настроен, URL внесены в кабинет Robokassa.

Перед фактической отправкой на модерацию остаётся не технический, а владельческий/legal gate: финально вычитать юридические документы, сверить сведения продавца с кабинетом Robokassa и пройти одну сквозную тестовую оплату из личного кабинета. Перед боевым запуском после модерации нужно переключить `Robokassa__IsTest=false` и пройти основной платёж.
