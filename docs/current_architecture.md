# Письмолёт — краткое описание текущей архитектуры

Дата: 2026-06-24
Статус: краткая актуальная справка вместо устаревающего ТЗ

## 1. Назначение документа

Документ коротко фиксирует текущую архитектурную картину проекта Письмолёт без детализации таблиц, полей, методов и внутренних реализационных нюансов.

Он нужен как ориентир для будущих технических, юридических, продуктовых и маркетинговых обсуждений, если старое ТЗ будет удалено или перестанет использоваться как источник актуального состояния.

## 2. Общая модель продукта

Письмолёт состоит из двух логических частей:

1. Публичная часть — сайт `pismolet.ru`.
2. Приложение — рабочий сервис `app.pismolet.ru`.

Публичная часть отвечает за презентацию продукта, юридические страницы, маркетинговые тексты и справочную информацию. В ней не находится основной сценарий загрузки баз, оплаты и отправки рассылок.

Приложение отвечает за регистрацию и вход пользователей, профиль клиента, импорт email-базы, подготовку письма, расчёт стоимости, оплату, запуск рассылки, отчёты, отписки, delivery tracking, suppression, админские экраны и техническую диагностику.

## 3. Публичная часть

Публичный сайт может размещаться на обычном хостинге как отдельная статическая или почти статическая часть проекта.

В публичной части предполагаются:

- лендинг;
- описание сервиса;
- тарифы и позиционирование;
- справочные страницы;
- юридические документы;
- SEO-страницы и маркетинговый контент.

Публичная часть не должна принимать клиентские базы адресатов и не должна быть источником юридически значимых подтверждений по конкретным рассылкам. Такие подтверждения должны фиксироваться внутри приложения.

## 4. Приложение

Приложение размещается отдельно от публичного сайта и работает как серверное веб-приложение.

Текущая backend-архитектура построена на .NET / ASP.NET Core. Основной проект приложения находится в `src/Pismolet.Web`. Он подключает пользовательские и админские endpoint-ы, cookie-аутентификацию, авторизацию админов, статические файлы, фоновые сервисы и интеграции.

Приложение отвечает за:

- аккаунты и вход;
- профиль клиента;
- импорт email-адресов;
- проверку и подготовку базы;
- создание и запуск рассылки;
- расчёт стоимости;
- оплату;
- отправку писем;
- отчёты по отправке;
- tracking открытий и кликов;
- глобальную отписку;
- обработку ошибок доставки;
- suppression / исключённые адреса;
- админские панели диагностики.

## 5. Внутренняя структура решения

Решение разделено на несколько проектов:

- `Pismolet.Domain` — доменные сущности и правила предметной области;
- `Pismolet.Application` — прикладные сценарии и контракты;
- `Pismolet.Infrastructure` — работа с базой, почтой, внешними и системными интеграциями;
- `Pismolet.Web` — веб-приложение, endpoint-ы, middleware, UI и админские страницы;
- тестовые проекты — web/integration tests.

Такое разделение позволяет держать бизнес-логику отдельно от инфраструктуры и веб-слоя, а также постепенно расширять систему без превращения приложения в один монолитный файл.

## 6. Данные и хранение

Основное рабочее состояние приложения хранится в базе данных. Для production используется PostgreSQL.

В базе хранится состояние пользователей, клиентов, импортов, рассылок, платежных и отправочных сценариев, delivery-событий, отписок, suppression и другой рабочей информации сервиса.

Для тестов используется отдельная тестовая конфигурация, которая может подменять постоянное хранилище и почтового провайдера на тестовые реализации.

## 7. Отправка и доставка писем

Отправка писем выполняется приложением через инфраструктурный слой.

Система уже разделяет:

- подготовку рассылки;
- постановку писем к отправке;
- факт технической передачи письма;
- delivery-события;
- ошибки доставки;
- HardBounce / SoftBounce;
- suppression адресов;
- отчёты для клиента и админа.

Для production-сценария используется связка приложения, почтовой инфраструктуры и чтения delivery-логов. Ошибки доставки превращаются в понятные статусы и используются в отчётах, админке и suppression-логике.

## 8. Админская часть

Админская часть встроена в приложение и доступна только авторизованным администраторам.

В текущей архитектуре админка используется для:

- просмотра пользователей и клиентов;
- диагностики рассылок;
- payment/admin сценариев;
- delivery overview;
- drill-down по клиенту и рассылке;
- просмотра suppression;
- просмотра detail по конкретному исключённому адресу;
- технических настроек и диагностики Postfix/delivery.

Админские экраны нужны не только для поддержки, но и для объяснимости: почему письмо было отправлено, не отправлено, отбито, временно проблемное или исключено из будущих отправок.

## 9. Юридически значимые подтверждения

Юридические подтверждения должны фиксироваться внутри приложения, а не на публичном сайте.

Планируемый слой Legal Evidence должен хранить версии юридических текстов и события подтверждений клиента: принятие оферты, согласие на обработку данных, источник базы, декларацию законности базы, поручение на обработку email-адресов, рекламное согласие, подтверждение запуска рассылки и другие события.

Этот слой должен быть связан с клиентом, пользователем, импортом, рассылкой и версией текста, который видел клиент.

## 10. Развёртывание

Текущая рабочая модель предполагает раздельное развёртывание:

- публичная часть `pismolet.ru` — обычный хостинг для сайта и документов;
- приложение `app.pismolet.ru` — серверное приложение с базой данных, почтовой инфраструктурой и фоновыми задачами.

На сервере приложение обслуживается как отдельный сервис. Обновление приложения выполняется через получение актуальной версии из ветки разработки, сборку, тесты и перезапуск сервиса.

## 11. Что не фиксирует этот документ

Документ намеренно не описывает:

- конкретные таблицы и поля;
- внутренние методы и классы;
- точные SQL-индексы;
- подробные маршруты всех endpoint-ов;
- детальные бизнес-правила каждого сценария;
- устаревшие решения из старого ТЗ.

За деталями нужно обращаться к актуальным feature-планам, юридическим документам, спецификации и коду проекта.
