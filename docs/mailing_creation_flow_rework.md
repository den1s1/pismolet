# Перестройка флоу создания рассылки

Дата: 2026-06-28

Статус: принято к реализации.

## Причина изменения

Текущий пользовательский путь создания рассылки неудобен для основного сценария: сначала пользователь загружает список, затем пишет письмо, затем проходит подтверждения, оплату/модерацию и вручную запускает отправку.

Целевой сценарий должен начинаться с письма: пользователь сначала формулирует сообщение, затем выбирает или загружает адресатов, просматривает список, подтверждает юридические и продуктовые условия, оплачивает рассылку и больше не делает ручной запуск после модерации.

## Было

```text
Список адресатов -> письмо -> финальное подтверждение -> оплата и отправка на модерацию -> ручной запуск
```

Проблемы текущего поведения:

- пользователь вынужден начинать с базы, хотя часто сначала понятно письмо/повод;
- выбор типа рассылки и юридические подтверждения появляются слишком рано, на этапе загрузки получателей;
- после успешной модерации остаётся лишний ручной шаг запуска;
- нет явного выбора уже существующего списка адресатов как полноценного источника.

## Стало

```text
Письмо -> загрузка/выбор адресатов -> просмотр списка -> финальное подтверждение -> оплата -> автоматический запуск после успешной модерации
```

## Целевой пользовательский флоу

### 1. Письмо

Первый экран новой рассылки — подготовка письма.

Пользователь задаёт:

- имя отправителя;
- тему письма;
- текст/HTML письма;
- вложения, если они поддержаны текущей реализацией;
- тип письма на этом экране не подтверждается юридически, если это переносится на финальное подтверждение.

После сохранения письма пользователь переходит к адресатам.

### 2. Адресаты

На экране адресатов пользователь выбирает источник:

1. загрузить файл;
2. ввести адреса вручную;
3. выбрать уже существующий список адресатов.

На этом экране не должно быть:

- декларации законности базы;
- подтверждения рекламного согласия;
- выбора типа рассылки как юридически значимого действия;
- финальных чекбоксов запуска.

Задача экрана — только сформировать рабочий набор адресатов для этой рассылки.

### 3. Просмотр списка

После загрузки/ручного ввода/выбора существующего списка пользователь видит результат:

- сколько строк было в источнике;
- сколько адресов принято;
- сколько дублей;
- сколько некорректных;
- сколько исключено отпиской через Письмолёт;
- сколько исключено клиентским suppression, если включено;
- список ошибок/отклонений в безопасном виде.

Пользователь должен понимать, какой набор адресатов реально пойдёт в расчёт стоимости и отправку.

### 4. Финальное подтверждение

Все юридически и продуктово значимые подтверждения переносятся сюда.

На экране финального подтверждения пользователь должен видеть:

- краткое резюме письма;
- резюме адресатов;
- рассчитанное количество принятых адресатов;
- ориентировочную стоимость/стоимость к оплате;
- тип рассылки, если он нужен для проверки рекламного согласия;
- декларацию законности базы;
- подтверждение наличия согласия на рекламу для рекламных рассылок;
- подтверждение правил сервиса и антиспам-ограничений;
- предупреждение, что после оплаты рассылка уйдёт на проверку/модерацию или будет отправлена автоматически в зависимости от настроек.

Именно этот экран должен фиксировать legal evidence по базе, типу рассылки и запуску.

### 5. Оплата

После финального подтверждения система рассчитывает стоимость по принятым адресатам и переводит пользователя к оплате/списанию.

После успешной оплаты рассылка переходит в состояние ожидания проверок/модерации.

### 6. Модерация и автоматический запуск

Ручного запуска клиентом после успешной модерации быть не должно.

Целевое правило:

```text
Оплата успешна -> рассылка ожидает проверки/модерации -> администратор approves -> рассылка автоматически ставится в очередь отправки
```

Если рассылка не требует ручной модерации, она должна автоматически переходить к отправке после успешных автоматических проверок и оплаты.

Администратор запускает рассылку фактом approve. Отдельная кнопка ручного запуска после approve для клиента не нужна.

## Выбор существующего списка адресатов

Нужно добавить источник адресатов: существующий список.

Минимальная MVP-модель:

- пользователь видит свои ранее созданные/импортированные списки;
- может выбрать один список для новой рассылки;
- система копирует или снапшотит адресатов в текущую рассылку;
- дальнейшие изменения исходного списка не должны неявно менять уже подтверждённую и оплаченную рассылку;
- выбранный список проходит те же проверки dedupe/suppression/invalid, что файл или ручной ввод;
- на экране просмотра показывается, что источник — существующий список.

Открытое техническое решение:

- использовать текущие import batches как источник списка;
- или ввести отдельную сущность `RecipientList` / `AddressBookList`.

Для MVP предпочтительно не ломать текущую модель рассылок и сначала сделать адаптер поверх уже сохранённых импортов, если это возможно без риска для данных.

## Изменения в статусах рассылки

Требуется пересмотреть переходы статусов.

Целевой смысловой путь:

```text
Draft
-> MessagePrepared
-> RecipientsPrepared
-> RecipientsReviewed
-> DeclarationConfirmed
-> Priced
-> PaymentPending
-> Paid
-> PendingChecks / ReviewRequired
-> Approved
-> Sending
-> Sent / Failed
```

В UI порядок шагов должен быть именно:

1. письмо;
2. адресаты;
3. просмотр;
4. подтверждение;
5. оплата.

Текущие доменные статусы можно не переименовывать сразу, если это слишком рискованно. Главное — перестроить пользовательский routing и автоматический переход после approve.

## Модерация

Нужно изменить поведение admin approve:

- approve больше не просто переводит рассылку в `Approved` и ждёт ручного запуска клиента;
- approve должен ставить рассылку в очередь отправки или переводить её в состояние, из которого background sender стартует автоматически;
- audit log должен фиксировать, что отправка запущена вследствие admin approve;
- при reject отправка не запускается, пользователь видит причину отклонения.

## Acceptance criteria

Флоу считается готовым, когда:

- новая рассылка начинается с экрана письма;
- после письма пользователь переходит к адресу/источнику адресатов;
- на экране адресатов доступны файл, ручной ввод и выбор существующего списка;
- на экране адресатов нет юридических галочек и финального выбора типа рассылки;
- после формирования адресатов есть экран просмотра списка и статистики;
- декларация базы, рекламное согласие и финальные подтверждения находятся на экране подтверждения;
- после успешной оплаты рассылка не требует от клиента отдельного ручного запуска;
- если нужна модерация, admin approve автоматически запускает отправку;
- если модерация не нужна, отправка стартует автоматически после успешных проверок и оплаты;
- build/test зелёные;
- ручной smoke подтверждает новый порядок экранов.

## План реализации

### Спринт F1. Инвентаризация текущего флоу

- найти endpoints/pages создания рассылки;
- зафиксировать текущие route names;
- найти, где вызываются import, message draft, declaration, pricing, payment, moderation, send;
- найти admin approve path;
- найти текущую модель списков/импортов, которую можно использовать для выбора существующего списка.

### Спринт F2. Перестроить routing и шаги UI

- сделать экран письма первым шагом;
- после сохранения письма вести на экран адресатов;
- убрать декларации/тип рассылки с экрана загрузки адресатов;
- добавить экран просмотра результата адресатов перед финальным подтверждением.

### Спринт F3. Источник адресатов из существующего списка

- показать доступные списки пользователя;
- добавить выбор существующего списка;
- при выборе создавать snapshot адресатов для текущей рассылки;
- покрыть dedupe/suppression/invalid теми же правилами, что импорт файла/ручной ввод.

### Спринт F4. Финальное подтверждение

- перенести декларацию базы и рекламное согласие на финальный экран;
- показать резюме письма, адресатов, стоимости и типа рассылки;
- фиксировать legal evidence на финальном подтверждении;
- после подтверждения переводить к оплате.

### Спринт F5. Автоматический запуск после модерации

- изменить admin approve: approve ставит рассылку в очередь отправки;
- убрать клиентский обязательный ручной запуск после approve;
- проверить auto-start для рассылок без ручной модерации;
- добавить audit/log запись автоматического запуска.

### Спринт F6. Тесты и smoke

- обновить unit/integration tests под новый порядок;
- добавить smoke checklist;
- проверить файл, ручной ввод и существующий список;
- проверить paid -> moderation -> approve -> send;
- проверить paid -> no moderation -> send.

## Документы для синхронизации

Решение влияет на несколько направлений.

Нужно синхронизировать:

- `docs/specification.md`, раздел A — основной пользовательский путь создания рассылки;
- `docs/specification.md`, раздел C — момент фиксации декларации базы, рекламного согласия и legal evidence;
- `docs/production_readiness_plan.md` — если этапы готовности завязаны на старый порядок запуска;
- пользовательские подсказки/лендинг при необходимости — только если публичные тексты описывают пошаговый запуск.
