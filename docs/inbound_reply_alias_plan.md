# План перехода входящих ответов на клиентские alias

Дата: 2026-06-28

Статус: реализация начата; спринты 3.8b и 3.8c завершены, build/test зелёные.

## Причина изменения

Проверка реального ответа на тестовую рассылку выявила продуктовую и техническую проблему текущей модели: видимый адрес для ответа содержит длинный технический token. Такой адрес выглядит подозрительно для получателя и может не приниматься почтовым клиентом как корректный адрес для ответа.

Текущий вариант с token в видимом адресе не считать production-готовым поведением.

## Новое решение

Для каждого клиента выделяется стабильный человекопонятный alias на reply-домене Письмолёта.

Принцип:

- базовый alias берётся из части email клиента до символа `@`;
- alias приводится к безопасному виду;
- alias уникален в рамках reply-домена;
- при конфликте добавляется числовой суффикс;
- служебные имена зарезервированы и не выдаются клиентам;
- в MVP клиент не выбирает alias вручную.

Основной домен `pismolet.ru` не использовать для клиентских alias в MVP. Для ответов использовать отдельный reply-домен, чтобы не смешивать клиентские входящие ответы со служебной почтой проекта.

## Новое сопоставление ответа

Видимый адрес больше не кодирует рассылку и получателя. Сопоставление нужно выполнять по техническим заголовкам исходного письма и ответа.

При отправке каждого письма Письмолёт должен сохранять связь между исходящим message id, рассылкой, получателем, клиентом и send event.

При получении ответа Письмолёт должен:

1. определить клиентский alias по адресу входящего письма;
2. определить исходное письмо по заголовкам ответа;
3. найти рассылку и получателя через сохранённый mapping;
4. создать reply event;
5. переслать ответ владельцу рассылки;
6. если alias известен, но исходное письмо не найдено, переслать владельцу alias как несопоставленный ответ;
7. если alias неизвестен, оставить событие только в admin diagnostics.

## Правила alias

Минимальные правила для MVP:

- lower-case;
- разрешены латинские буквы, цифры, точка, дефис и подчёркивание;
- остальные символы заменяются или удаляются;
- начальные и конечные разделители убираются;
- минимальная длина после нормализации — 3 символа;
- максимальная длина — 48 символов;
- при конфликте использовать суффиксы вида `-2`, `-3`;
- если alias нельзя получить из email, использовать fallback с коротким hash;
- alias должен быть уникальным в БД;
- зарезервировать служебные имена: admin, support, help, info, legal, abuse, postmaster, root, security, billing, mail, reply, noreply.

## Acceptance criteria

Новая модель считается готовой, когда:

- видимый reply address не содержит token;
- alias короткий и человекопонятный;
- alias уникален и стабильно закреплён за клиентом;
- исходящие письма сохраняют message mapping;
- обычный ответ получателя сопоставляется с рассылкой и получателем;
- ответ пересылается владельцу рассылки;
- fallback известного alias не теряет письмо;
- неизвестный alias не пересылается случайному клиенту;
- auto-reply и bounce не пересылаются как обычные ответы;
- build/test зелёные;
- production matched-forward smoke пройден.
