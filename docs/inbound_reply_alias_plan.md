# План перехода входящих ответов на клиентские alias

Дата: 2026-06-28

Статус: завершено и проверено на production. Спринты 3.8a–3.8g закрыты, build/test зелёные, production smoke пройден.

## Причина изменения

Проверка реального ответа на тестовую рассылку выявила продуктовую и техническую проблему прежней модели: видимый адрес для ответа содержал длинный технический token. Такой адрес выглядел подозрительно для получателя и мог не приниматься почтовым клиентом как корректный адрес для ответа.

Видимый token-адрес не считается production-готовым поведением.

## Реализованное решение

Для каждого клиента выделяется стабильный человекопонятный alias на reply-домене Письмолёта.

Принцип:

- базовый alias берётся из части email клиента до символа `@`;
- alias приводится к безопасному виду;
- alias уникален в рамках reply-домена;
- при конфликте добавляется числовой суффикс;
- служебные имена зарезервированы и не выдаются клиентам;
- в MVP клиент не выбирает alias вручную;
- основной домен `pismolet.ru` не используется для клиентских alias;
- для ответов используется отдельный reply-домен.

## Сопоставление ответа

Видимый адрес больше не кодирует рассылку и получателя. Сопоставление выполняется по техническим заголовкам исходного письма и ответа.

При отправке каждого письма Письмолёт сохраняет связь между исходящим message id, рассылкой, получателем, клиентом и send event.

При получении ответа Письмолёт:

1. определяет клиентский alias по адресу входящего письма;
2. определяет исходное письмо по заголовкам `In-Reply-To` / `References`;
3. ищет mapping по исходящему message id;
4. создаёт reply event;
5. пересылает matched reply владельцу рассылки;
6. если alias известен, но исходное письмо не найдено, пересылает владельцу alias как known-alias fallback;
7. если alias неизвестен, оставляет событие только в admin diagnostics;
8. auto-reply / mail loop не пересылает.

## Правила alias

Минимальные правила для MVP:

- lower-case;
- разрешены латинские буквы, цифры, точка, дефис и подчёркивание;
- остальные символы заменяются или удаляются;
- начальные и конечные разделители убираются;
- минимальная длина после нормализации — 3 символа;
- максимальная длина — 48 символов;
- при конфликте используются суффиксы вида `-2`, `-3`;
- если alias нельзя получить из email, используется fallback с коротким hash;
- alias уникален в БД;
- зарезервированы служебные имена: `admin`, `administrator`, `abuse`, `postmaster`, `root`, `support`, `help`, `hello`, `info`, `legal`, `privacy`, `security`, `billing`, `mail`, `reply`, `noreply`, `no-reply`, `mailer-daemon`.

### Спринт 3.8g. Production smoke alias-модели

Статус: завершён.

Проверено на production:

- `Reply-To` стал нормальным alias: `den1s@reply.pismolet.ru`;
- длинного `reply+v1...` token в `Reply-To` нет;
- outbound message mapping создаётся;
- обычный Reply сопоставляется с рассылкой и получателем;
- matched reply пересылается владельцу рассылки;
- known-alias fallback без `In-Reply-To` / `References` пересылается владельцу alias;
- fallback сохраняет `ErrorCode = reply_message_reference_missing` и сообщение диагностики после пересылки;
- unknown alias получает `ProcessingStatus = Unmatched`, `ErrorCode = reply_alias_unknown` и не пересылается;
- auto-reply получает `ProcessingStatus = IgnoredAutoReply`, `ErrorCode = auto_reply` и не пересылается;
- `/admin/replies` показывает полные email-адреса и корректную диагностику;
- временный production DB trigger для сохранения fallback-диагностики удалён, поведение подтверждено кодом.

## Acceptance criteria

Новая модель считается готовой. Выполнено:

- видимый reply address не содержит token;
- alias короткий и человекопонятный;
- alias уникален и стабильно закреплён за клиентом;
- исходящие письма сохраняют message mapping;
- обычный ответ получателя сопоставляется с рассылкой и получателем;
- ответ пересылается владельцу рассылки;
- fallback известного alias не теряет письмо;
- неизвестный alias не пересылается случайному клиенту;
- auto-reply и bounce не пересылаются как обычные ответы;
- build/test зелёные;
- production matched-forward smoke пройден;
- production fallback/unknown/auto-reply smoke пройден.
