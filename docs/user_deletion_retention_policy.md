# Политика удаления пользователей из админки

Статус: действующая политика для текущей реализации.

Цель удаления из админки - заблокировать дальнейший вход пользователя и убрать его рабочие рассылки из кабинета и пользовательских сценариев. Удаление не является полным юридическим стиранием всех следов, потому часть технических записей нужна для антиспама, расследования жалоб, возвратов, бухгалтерии и защиты сервиса.

## Кто не может быть удалён

- Администратор не может удалить собственный аккаунт.
- Нельзя удалить администратора, заданного в конфигурации сервера.
- Нельзя удалить пользователя с назначенными через админку правами администратора, пока права не сняты.

Эти проверки есть и на endpoint-уровне, и внутри `AdminUserRemovalService`, чтобы сервисный слой не зависел только от UI.

## Что удаляется сейчас

При успешном удалении пользователя:

- удаляется запись пользователя из `users`, включая логин, парольный hash, статус подтверждения email, имя и телефон;
- удаляются все рассылки владельца из `mailings`;
- вместе с рассылками удаляются рабочие данные, которыми владеет рассылка:
  - импортированные получатели;
  - import batches и issues;
  - декларации законности базы;
  - черновик письма;
  - сохранённые вложения черновика.

После этого пользователь не может войти в кабинет, а его рассылки не должны отображаться в пользовательской истории и обычной админской сводке кампаний.

## Что сохраняется намеренно

Эти данные не удаляются текущим действием администратора:

- audit records, включая событие `admin_user_removed`;
- глобальные и клиентские suppression-записи;
- send events и delivery/reply events, если они уже были созданы;
- legal evidence events и версии юридических документов;
- служебные записи безопасности и антиспам-диагностики;
- платёжные и бухгалтерские следы, когда они станут durable-хранилищем.

Причина: эти записи нужны для доказательства согласий, обработки отписок, жалоб, hard bounce, разборов доставки, возвратов и споров. Для них нужен отдельный процесс анонимизации/удаления по запросу субъекта данных, с проверкой законных оснований и сроков хранения.

## Важная оговорка по оплатам

В текущей версии `IPaymentRepository` зарегистрирован как in-memory repository. История оплат в админской сводке строится из рассылок, поэтому после удаления рассылок пользователя она исчезает из обычного UI. При подключении durable-хранилища оплат нужно добавить отдельное правило: финансовые записи не удаляются физически вместе с пользователем, а либо сохраняются до истечения обязательного срока, либо анонимизируются после него.

## Проверки

Покрыто тестами:

- удаление пользователя удаляет логин и owned mailings;
- чужие пользователи и чужие рассылки не затрагиваются;
- пишется audit event `admin_user_removed`;
- нельзя удалить самого себя;
- нельзя удалить config-admin;
- нельзя удалить managed-admin до снятия прав;
- endpoint `/admin/users/{email}/remove` удаляет обычного пользователя и его рассылку через реальный web flow.

Остаточный долг:

- добавить EF-level integration-тест удаления пользователя после появления durable payment repository;
- добавить отдельный GDPR/152-ФЗ сценарий анонимизации retained-событий, если в продукте появится пользовательский запрос на полное стирание.
